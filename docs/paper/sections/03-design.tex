\section{设计}

\namei 的核心接口是一个单一 BPF decision function。kernel 在 lookup 或 readdir 事件上
构造上下文，policy 返回 \code{PASS} 或 \code{REDIRECT}。lookup 和 directory enumeration
使用同一个 policy 入口，通过事件类型区分行为。这个约束避免为每个操作暴露不同 ABI，也让
lookup/readdir 一致性可以成为 checker 的显式 oracle。

\subsection{边界}

\namei 不创建 dentry，不分配 inode，不实现 file operations，也不绕过 lower filesystem
权限。Phase 1 的 \code{REDIRECT} 只把当前 component 替换为同一父目录下的 backing
component。完整 parent path、cross-directory target registry、write-through alias、
recursive path rewrite 和用户态 policy interpreter 都不在当前已实现 ABI 中。论文中的
任何强 claim 都必须受这个边界约束。

\subsection{为什么不是静态 redirect table}

静态 table 可以表达 \code{(event, component) -> target}，因此它是必要 baseline。
最终 programmability 主张需要证明多类 extension 在同一 ABI 上需要不同的
verifier-safe policy logic。build graph 需要 precedence 和 poison 分支；fixture
workload 需要 path-class campaign 和 no-real-secret oracle；checkpoint/restore 需要
restore epoch consistency；cache locality 需要 content-state transition。当前版本只把
未满足该条件的 cases 作为反事实边界。若 table-only 在同等 table/update budget 下
通过全部主 oracle，论文必须降级为 kernel redirect-table artifact，而不能声称需要 eBPF
programmable path resolution。
