## Research Question

- RQ exactly as written in the paper:
  RQ1 asks whether a narrow VFS name-resolution extension can express real
  state-dependent path-view policies without taking over filesystem semantics.
- Specific uncertainty tested here:
  Whether an AgentFS-derived workspace lifecycle can be expressed as bounded
  lookup/readdir selection and visibility over existing lower objects, while
  ordinary file behavior remains with the lower filesystem.
- Why the answer matters:
  Agent/workspace filesystems are the strongest source family for the paper's
  thesis. If this matrix passes, the paper has a headline systems result rather
  than a toy hook demonstration. If it fails for a real boundary reason, the
  paper must report the workload/mechanism boundary before proceeding.

This experiment also includes RQ2 and RQ3 cells required by the frozen
evaluation promise:

- RQ2: feature-equivalent FUSE policy under the same correctness oracle.
- RQ3: invalid-policy containment and same-oracle boundary accounting versus
  custom or stackable filesystem ownership.

## Paper-Value Admission

- Planned role: headline.
- Largest credible paper story this experiment could unlock:
  `namei_ext` can satisfy a real agent-workspace path-view oracle through the
  real `cgroup/namei_ext` KVM attach path, while the kernel and lower
  filesystem retain filesystem ownership. The same matrix can then support the
  strongest RQ2/FUSE and RQ3/boundary claims for the paper's most important
  workload family.
- Strongest reviewer reject argument or load-bearing uncertainty addressed:
  A reviewer can argue that agent filesystems use FUSE or custom filesystem
  logic because the needed behavior is broader than name resolution. This
  matrix tests whether the oracle-relevant slice is actually path selection and
  visibility over existing lower objects.
- Independent evidence added beyond existing runs and published results:
  Prior source reproductions show that agent workspace systems and oracles
  exist. Prior KVM preflights show that `HIDE` and `SELECT_TARGET` run. This
  experiment adds the frozen same-oracle matrix with `namei_ext`, FUSE,
  controls, containment, boundary accounting, raw traces, and result review.
- Why the result is not tautological, already settled, or dominated:
  Published AgentFS/YoloFS/BranchFS/Sandlock/Mirage evidence motivates the
  workload but does not test this VFS boundary. Preflight and raw prototype
  runs do not answer FUSE parity, lower-FS preservation, or same-oracle
  boundary accounting. Environment/cache is the next decisive family, but
  Agent workspace has higher immediate decision value because its source
  oracles and implementation path are already closest to complete.
- Paper decision if positive:
  Treat Agent workspace as the headline RQ1 result, interpret RQ2 cost only
  after FUSE correctness parity, and use RQ3 boundary accounting for the same
  policy. Proceed next to Experiment B for cross-family evidence.
- Paper decision if contradictory, mixed, or inconclusive:
  If the real oracle requires synthetic contents, custom metadata persistence,
  data-path mediation, write-conflict resolution, or agent orchestration, record
  the boundary and move to the next admitted experiment rather than shrinking
  the thesis around a toy subset. If only the implementation or FUSE parity is
  incomplete, repair this experiment inside BUILD_AND_EVALUATE.
- Best alternative experiment and why this one has higher decision value:
  Experiment B environment/cache is the next best alternative, but it has a
  larger implementation admission risk around final-object/cache state
  selection. Experiment A is the highest-value first run because it directly
  attacks the closest source-system objection and has an existing KVM/FUSE
  control plane to repair into a final matrix.

Admission verdict: admitted as the Step 0002 headline experiment. Standalone
preflights, table diagnostics, materialized-view shootouts, and extra source
reproductions are rejected as independent paper experiments for this step.

## Expected And Alternative Outcomes

- Current expected answer:
  The AgentFS-derived path-view slice can be expressed with bounded
  lookup/readdir policy: base/upper/checkpoint selection, whiteout visibility,
  symlink lower-object behavior, logical writes landing in the selected lower
  object, and an epoch update that changes path selection without remounting or
  materializing a copy tree.
- Strongest competing explanation:
  Agent workspace semantics require filesystem ownership because the source
  oracle depends on synthetic contents, custom metadata persistence, post-open
  data mediation, conflict resolution, or runtime orchestration.
- Result that would contradict the expectation:
  A correctness oracle that cannot be satisfied while preserving lower-FS
  ownership, or a feature-equivalent FUSE implementation that matches or beats
  `namei_ext` on the policy path without imposing a meaningful RQ3 boundary
  tradeoff.

## Comparison

- Proposed system or method:
  `namei_ext` policy `agent_workspace_view.bpf.c` loaded and attached through
  the real `cgroup/namei_ext` KVM path.
- Main baselines and the competing position each represents:
  Feature-equivalent FUSE policy filesystem. It represents the closest
  programmable filesystem answer to the same path-view policy.
- Why each main baseline needs a matched run instead of citation alone:
  RQ2 asks the cost of the same policy on the VFS name-resolution path versus
  FUSE. Published FUSE overhead results do not measure this oracle, state
  machine, operation mix, or project implementation.
- Controls or ablations, labeled separately:
  No-hook/lower-FS control for overhead attribution; invalid-policy containment
  control for RQ3; source/native behavior only as oracle provenance, not as a
  performance baseline.
- Conclusion if each main baseline matches or wins:
  If feature-equivalent FUSE matches or wins on cost and does not impose a
  meaningful boundary burden for this oracle, RQ2/RQ3 are inconclusive or
  contradicted for Agent workspace even if RQ1 passes.
- Information, tuning, and compute fairness:
  `namei_ext` and FUSE receive the same workspace state, logical paths, policy
  epochs, and correctness oracle. Any state-machine or oracle change reruns
  both rows.
- Split or leakage rule when relevant:
  The workload and oracle are fixed before the full run. Result-dependent
  changes create a plan deviation and require rerunning affected rows.

## Workloads And Metrics

- Real workloads or tasks:
  A fixed AgentFS-derived workspace lifecycle over a base tree, upper tree,
  whiteout/delete state, symlink, generated file, negative-entry transition,
  and epoch update. The minimum accepted lifecycle must include logical path
  operations over the same stable workspace name before and after the state
  transition, lower-FS/no-hook control rows, and final manifest evidence.
- Primary metrics:
  Source oracle pass/fail, lookup/readdir coherence, lower-FS preservation,
  operation-weighted lookup/readdir/action counts, lookup/open/stat/access/exec
  where applicable, readdir latency, macro command runtime, FUSE daemon request
  counts, and invalid-policy containment.
- Correctness check or ground truth:
  Final logical tree manifest/hash, expected content for selected base/upper
  objects, whiteout absence in lookup and readdir, symlink lower-object
  behavior, generated-file placement, unchanged base tree except declared
  state, and no kernel warning/oops in dmesg.
- Repetitions, seeds, and uncertainty:
  Correctness rows run once per full matrix by default and fail hard. The
  repaired matrix must report at least 100 stat repetitions and 50 readdir
  repetitions for both `namei_ext` and FUSE. Result review may require broader
  repetition before paper numbers are used, but the matrix is incomplete if
  these minimum rows are absent.
- Cost estimate when material:
  The current KVM matrix is expected to be minutes, not hours. Environment/cache
  remains out of scope for this step.

## Interpretation

- Positive result:
  `namei_ext` passes the same oracle, preserves lower-FS semantics, FUSE passes
  the same oracle but shows characterized request-path cost, and boundary
  accounting shows broader filesystem ownership for the same policy.
- Negative or contradictory result:
  A source-derived oracle requires broader filesystem ownership, or
  feature-equivalent FUSE matches/wins without a meaningful boundary tradeoff.
- Mixed or inconclusive result:
  Missing FUSE parity, incomplete operation traces, absent performance
  repetitions, missing invalid-policy containment, or boundary accounting not
  tied to the same oracle.
- Target paper figure or table:
  One Agent workspace table covering correctness/oracle, cost versus FUSE, and
  boundary ownership for the same workload; one plot or table for operation
  counts/latency if the run produces stable performance evidence.
