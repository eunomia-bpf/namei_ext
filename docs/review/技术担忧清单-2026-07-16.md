### A. 这条担忧现在是什么状态——旧攻击失去靶子，但更深层的问题没消失

早先六轮红蓝对抗反复指出"应用场景狭窄"，当时攻击的靶子有三样：文件视图供给只占 agent 端到端很小一块（Amdahl 定律层面站不住）、万级视图无具名生产部署、agent 方向过窄。

合作者新版论文**不是反驳了这些攻击，而是把这些主张整个删掉了**：密度、万级、DeltaBox、"任务变快"、外推数字在 origin/main 全文检索命中数为零。论文改成了能力与边界体裁（RQ1 表达力 / RQ2 对 FUSE 的成本 / RQ3 边界）。因此 Amdahl、密度这类旧攻击已经失去了靶子——对一篇不主张提速的论文再去打提速，就是在攻击一个论文没有提出的主张。**这部分我们主动退出，不再使用，也不应该使用。**

**但更深层的一条担忧没有消失，只是换了表现形式**："没有杀手级应用 / 能力本身就窄"。它现在的表现是——把六轮里出现过的、namei_ext 确实占一定优势的候选场景逐一展开来看，每一个都因为限制条件过多而当不成杀手级场景；而且挡住它们的始终是同样那几类原因反复出现。这种"反复出现"本身，就是"窄"的结构性证据。

---

### B. 根本原因：可编程 VFS 能表达的动作集合受到两端限制

可编程 VFS 能表达的动作被 eBPF verifier（内核在加载 eBPF 程序前执行的静态检查器，限制条件包括：指令数上限约 100 万条、循环必须可证明有界、不允许无限递归等）从两端限制住：

- **简单的一端**——允许或拒绝访问、把路径重定向到一个**已经存在**的对象——已经被 **overlayfs / composefs / bpf-lsm / CamFlow** 覆盖，不是新的；
- **丰富的一端**——在内核里就地改写文件内容、注入数据、做脱敏处理——**超出 verifier 的上限，只能退回用户态执行，而退回用户态就是退回 FUSE**（FUSE 正是本项目立项时要替代的对象）。

因此，"既能在内核内以接近原生的速度运行、又足够新颖有用"的动作集合，很可能是空的。

**新论文的四个动作 PASS / REDIRECT / HIDE / SELECT_TARGET，做的正是"选择或隐藏一个已经存在的下层对象"——恰好落在上面说的那个已被覆盖的简单一端。** 论文本该依靠的独特能力（丰富动作），是经过排查后近乎为空的那个部分。

**一处修正（如实附上，doc08:169-177）**：bpftime（你们组 eunomia-bpf 的用户态 eBPF 运行时，不受内核 verifier 指令上限约束）对"性能/功能类"用途确实打破了"退回 FUSE"这一端的限制；**但对"安全/治理类"用途仍然需要内核钩子，仍然只能执行简单动作，那一端的结论不变**。所以根本原因的整体判断仍然成立。

### C. 逐场景：确有优势，但限制条件过多

> **说明**：下列每个场景，namei_ext 都确实占有一定真实优势，不是我们编造的靶子来打。裁决词逐字保留。Amdahl/密度等已失去靶子的限制不写入下面的"限制"中。

---

#### 场景 1 — agent 分支扇出

**含义**：一台机器上成百上千 agent 并行，每个立即拿到同一仓库的一份可独立读写、可整份丢弃的工作副本。

**裁决**：BUILD → ADOPT_MODIFIED

**① namei_ext 确实有的优势**

建一个分支只需要往内核表里写几条记录，微秒级完成，不需要挂载文件系统、不需要复制文件、不需要启动虚拟机。读写全程以原生速度进行。几千个分支共享同一底座文件的 page cache（内核为文件内容维护的内存缓存，多个使用者读同一文件只需一份内存）。每个视图只需要 O(改动数) 条 map 条目，而 overlayfs 每个视图需要一整套 superblock 加 O(树节点数) 的影子元数据，内存开销更大。新建文件、O_TRUNC（打开文件时先清空内容）、rename、unlink、mkdir 这五类名字层写动作不需要在内核中做内容拷贝。万个视图可以原子地换底座。活进程可以原位切换视图。git 是很好的负载发生器（但 git 本身不是瓶颈）。

**② 使它当不成杀手级的限制条件**

- **[能力边界]** 分支的独特价值在于"独立可读写的副本"，而写侧需要新建文件——新建的文件不是现有的对象，不能用 REDIRECT 指向。当前只有 PASS 与 REDIRECT（指向另一已有文件）两种读方向决策，加上一条受限的写意图规划；完整的写语义此前列为推迟项。残余的写类操作（底座已有文件的就地 O_RDWR / O_APPEND / 共享写 mmap）在 v1 中直接返回 EROFS（只读文件系统错误）——**就地修改不在当前能力范围内**。（doc21:25-26 / doc22:50）

- **[与本组系统撞车]** 宿主系统 BranchFS 是同组系统（Cong Wang / 郑昱笙），双盲评审下拿自家系统当主打场景容易被审稿人读成"上一篇工作的增量"。BranchFS 刻意留在 FUSE，`branch()` 只经三个通用 ioctl、不接触路径解析，因此 namei_ext 在结构上无法接入 BranchFS。想要差异化的"省掉 copy-up 的数据拷贝"被 overlayfs 源码事实抽空——overlayfs 的 copy_up.c（v6.12 L1187-1188）对 O_TRUNC/新建/改名/删除本就是零数据字节拷贝，裁判亲手读源码确认。（doc17:45 / doc22:66 攻击2）

- **[一行配置的现成近邻]** 最强近邻方案是"把 overlay 挂进受害者已经在付费维护的每实例 mount namespace"，这个做法距离现状只差一个内核 release 的工作量，而且没有人试过给出不可行的理由；在已发表的运行点（256 并行）上，本机制没有优势。（doc22:66 攻击3）

- **[层错位——真正的负载在别的层]** agent 分支的真实产品实现在虚拟机层：E2B 约 150ms、Morph 小于 250ms，各自运行独立的 guest 内核、不共享缓存。高频区间的现任服务者 DeltaBox 走进程 fork/CRIU（CRIU 是 Checkpoint/Restore In Userspace，一种把运行中进程快照并恢复的工具），瓶颈在进程侧而不在名字解析侧。（doc21:16 / doc20:186 / doc18:159）

**③ 一句话**：能表达的读侧选择与隐藏，overlayfs 零拷贝已经做到；真正独特的写侧不在当前能力范围内；宿主系统又是自家的 FUSE 实现，结构上接不进去。

#### 场景 2 — FaaS/serverless 高密度

**含义**：一台机器同时给 N 个函数各一套文件视图，N 从 100 到 10,000。

**裁决**：BUILD（曾被定为旗舰主载体）

**① namei_ext 确实有的优势**

SOCK（ATC'18）论文自己写明"因为伸缩性放弃了 mount namespace"——这是**唯一有一手论文证据说客户层在结构上缺少这个能力**的领域。每个视图只需要几条 map 条目，内存开销接近零。10³ 级挂载遇到瓶颈的证据全部有出处：Red Hat 报告 19 倍退化、systemd issue #31137、MOVE_MOUNT_BENEATH。高密度加上高换手（短时间内大量创建和销毁函数实例）时，挂载方案会遇到挂载表膨胀与 namespace_sem 全局锁，而 map 条目方案不会遇到这些问题。

**② 使它当不成杀手级的限制条件**

- **[层错位——真正的负载在别的层]** 生产环境中的高密度层在 microVM（每个函数运行在轻量虚拟机中，有自己的 guest 内核来解析路径）或语言级隔离中：AWS Lambda 用 Firecracker、Cloudflare Workers 用 V8 isolate、E2B/Daytona/Modal 跨租户用 microVM 或 gVisor。**这些方案都不共享宿主内核**，因此 namei_ext 在宿主内核路径解析层做的事情对它们没有作用。（doc17:600 / doc18:77 / doc20:35-38）

- **[能力边界 / 粒度]** 该场景的策略只需要两张纯查找表，不涉及任何计算决策。一张固定的 per-cgroup redirect 表（不需要 BPF 程序）就能实现同样的功能——不能拿这种纯表查找来当"可编程性"的证据。（doc17:574/598）

- **[一行配置的现成近邻]** 一个混合基线——"1 次 bind 挂载 + RO remount + chroot 内相对 symlink + handler 拷贝"——就能把每个沙箱的挂载表操作从 8 次降到 3-4 次、把上限提高到约 1500-2000/s，读一遍 sock.go 源码就能构造出来。（doc17:571/595）

- **[能力边界——安全]** 想要取代 microVM 又不损失安全隔离，做不到：namei_ext 共享同一宿主内核，不提供任何安全边界（fd 传递、/proc 信息泄露、namespace 逃逸都是攻击面），对互不信任的租户不成立。**只适用于同一信任域内的场景**（单租户批量扇出 / RL rollout / CI runner 池）。（doc20:84 / 42-43）

**③ 一句话**：真正需要高密度的生产环境在 microVM 或 V8 这一层（层错位），namei_ext 只服务同一信任域内的场景；而且策略只是查找表不是可编程性，混合基线已经把密度上限提高到约 2000/s。

#### 场景 3 — 证书/配置的 epoch 原子轮换

**含义**：运行中的服务不重启就完成换代，per-task（按进程/任务粒度）钉定版本，绝不读到"新证书配旧私钥"这种不一致状态。

**裁决**：BUILD → ADOPT_MODIFIED（曾被定为在线动态旗舰）

**① namei_ext 确实有的优势**

把版本钉在 task 上、跨 exec 到子进程仍保持一致，这是 mount namespace 与 openat 都给不出的能力（一次读取会话内跨多次读操作的一致性、按 cgroup 或 task 选择性分版本）。"无特权多线程进程里每线程拥有私有视图"这个能力缺口经 man page 核实确实存在。竞态的类别有名有姓——gRPC file_watcher 对两个文件做零协调轮询、C-core 自己承认无法校验证书与私钥是否配对，并在代码中留有 TODO 未修。

**② 使它当不成杀手级的限制条件**

- **[一行配置的现成近邻]** "给所有人原子换一次"这件事，nginx 的 `$realpath_root` / mod_realdoc（2013 年起的标准配方、Deployer 官方收录）一行配置就能做到；K8s kubelet AtomicWriter 用一次 `..data` 目录 rename 实现原子换代；Envoy 官方推荐 symlink 轮换——全部零内核改动。（doc17:874 / doc20:128）

- **[事实纠错 / 现有机制可替代]** "per-thread 视图用挂载做不到"是**事实错误**：单线程可以执行 `unshare(CLONE_FS)` + `unshare(CLONE_NEWNS)` 拿到每线程私有的 mount namespace，20 行 C 代码可复现；`openat` + `O_PATH` 目录 fd 钉定也能解决同一问题。（doc17:875/876）

- **[能力边界 / 无生产证人]** "1 万个进程共享同一 mount namespace、同一绝对路径按 cgroup 解析出不同内容"这条独占优势没有任何生产环境的使用者。"对黑盒应用（不修改代码的应用）能保证配对错配在结构上为零"这个主张被"超时式钉定"的算术推翻——按切换频率计算约有 5 次错配/秒——蓝队已放弃该论点。（doc17:877 / doc20:108/120）

- **[粒度]** 独占优势最后只剩下 per-cohort 选择性（按一组进程选择性地分配版本）加 per-task 配对钉定，且只覆盖"一个请求对应一个进程"这种结构（PHP-FPM / prefork / gunicorn sync worker），不覆盖 nginx worker 或 Node 事件循环这类一个进程处理多个请求的架构。真实的持续部署切换频率约为 0.145 次/秒。（doc20:124 / doc17:857/878）

**③ 一句话**：全局原子换代 nginx/K8s 一行配置就能做到；"挂载做不到 per-thread"是事实错误；剩下只有 per-cohort/per-task 的粒度差，且无生产证人，切换率极低。

#### 场景 4 — 环境/缓存 epoch（= 合作者定为 decisive 的实验 B）

**含义**：多个执行环境共享同一台机器，通过 epoch（世代号）翻转来原子切换缓存或环境版本，每个视图可以选择性地处于不同 epoch。

**裁决**：BUILD，后并入密度层

**① namei_ext 确实有的优势**

在多个优势维度的交汇中，这是最好的场景之一。per-view 选择性 epoch flip（按视图粒度选择性地翻转世代号）是 symlink swap 与 composefs 的静态清单结构上做不到的。ccache 的完整性检查（XXH3 哈希）只能防止数据损坏，防不了有意的投毒。

**② 使它当不成杀手级的限制条件**

- **[现有机制可替代 / 一行配置的现成近邻]** 全局 epoch 切换被"共享目录中的 per-entry symlink + 原子 rename"这一零内核改动方案覆盖，独占优势只剩 per-view 选择性。缓存投毒被"删除共享存储里的对应条目"这一零内核改动方案覆盖。（doc17:437 风险4 / 503 让步3）

- **[撞车]** "在内核内维护全局 generation（世代号）"这个机制的优先权让给了 YoloFS（arXiv 2604.13536，其核心机制描述为 "keeping only the latest override tree in the kernel"）。epoch 进一步收窄为 per-view 选择性。（doc17:503 让步5）

- **[现有机制可替代]** pthread 可以执行 `unshare(CLONE_NEWNS)` 获得独立的 mount namespace，所以"任何速度都做不到"这个说法只对 goroutine（Go 语言的轻量线程，不能调用 unshare）和非特权进程成立。（doc17:437 风险2 / 503 让步9）

- **[能力边界]** `__pycache__` 这类自身有第一方控制旋钮（环境变量 PYTHONDONTWRITEBYTECODE / PYTHONCACHEPREFIX）的缓存应该让给环境变量来控制；写隔离应该让给 `git apply` 来处理。（doc17:503 让步4）

**③ 一句话**：这正是被合作者定为 decisive（决定性的）的实验 B——它的独占优势只剩下 per-view 选择性 epoch，而全局换代和投毒防护零内核改动就能做到，YoloFS 已经在内核里实现了 generation 机制。**decisive 场景在结构上不需要本机制（对应下文 I-2 的具体讨论）。**

#### 场景 5 — Python 环境高密度

**含义**：同一台机器上运行上万个 Python 环境，同一个 `/app/.venv` 路径在不同 cgroup（内核用于分组管理进程资源的机制）中解析出不同的环境。

**裁决**：BUILD → ADOPT_MODIFIED

**① namei_ext 确实有的优势**

这是全项目**唯一一个关键常数已全部实测齐备**的密度负载：`.pth` 文件（Python 的路径配置文件，放在 site-packages 目录下，Python 启动时自动读取并把其中列出的路径加入 sys.path）处理时间 529µs、每视图 12-20KB、6591 个文件包集、七条基线的机械步骤全部现成。独占优势是"同一固定绝对路径按 cgroup 子集原子灰度（在部分 cgroup 先切换、确认无误再推广到全部），无需程序配合"。

**② 使它当不成杀手级的限制条件**

- **[一行配置的现成近邻]** 在纯 Python 框架下，一个 529µs 的 `.pth` 文件就覆盖了其余所有优势维度。SOCK 用不到 1µs 的 chroot 解决了同类问题。（doc17:352）

- **[现有机制可替代]** 只要基线一方获得同等程度的管理员配合（开机时挂载 K 个 squashfs 镜像），每个任务执行 `unshare` + `bind` 就能以同样方式通过它自己的验证标准（oracle），锁占用 1.5-6% 没有区分力。（doc18:95/118）

- **[能力边界 / 无生产证人]** "单机 ≥10³ 个互不相同、共享内核的 Python 环境"没有任何具名的部署者——工业界走 microVM 或用户态缓存方案。（doc18:90/92）

- **[撞车]** LUMI（欧洲超算中心）和 Apptainer（前 Singularity，高性能计算中广泛使用的容器运行时）用每任务 mount namespace + bind 挂载；Apptainer 1.1.0（2022-09）默认支持 rootless + squashfuse 近内核挂载。之前被引用的 Frontera loop 设备数量瓶颈是 2021 年旧栈上的问题，已修复。（doc18:96/119）

- **[粒度]** per-dist（按发行版粒度）与 per-image（按镜像粒度）只是同一机制的不同粒度选择，就该场景本身只是粒度上的差异。（doc16:224）

**③ 一句话**：一个 `.pth` 文件、一行配置就给出等价的环境视图；有管理员配合则 `unshare` + `bind` 通过其验证标准；没有 ≥10³ 的具名部署者。

#### 场景 6 — 内核名字层的工作区事务

**含义**：begin 操作 = 往 map 里写几条记录；commit 操作 = 校验读写足迹无重叠 + 版本号一次原子翻转。

**裁决**：BACKUP

**① namei_ext 确实有的优势**

提交准入的依据是**内核在名字解析处观测到的全进程读写足迹**——子进程、bash 直接写入的操作按构造都可见，不需要像某些方案那样每条命令都做整棵目录树的拷贝。提交操作只是版本号一次翻转，同机全部进程看不到中间状态。

**② 使它当不成杀手级的限制条件**

- **[能力边界]** 足迹捕获其实只需要系统调用边界上的普通 BPF（在 open/read/write 等系统调用入口挂钩子记录文件访问），**根本不需要 namei_ext 的名字解析钩子**——这一层的必要性从"独立的核心功能"降为"依附于其他机制的派生功能"。（doc22:171）

- **[一行配置的现成近邻]** 原子发布这个维度被 symlink 根翻转（与 K8s AtomicWriter 同型）打平，蓝队已承认（CONCEDED）。足迹观测方面，fanotify（内核提供的文件访问通知机制）加 worktree 用户态栈这条路没有被排除，只能当作最强基线。（doc22:177 攻击2）

- **[撞车]** 判定谓词（用什么规则判定一个事务是否可以提交）等同于 STORM 已形式化的 local state consistency；观测足迹冲突等同于 Rattle（2020）的方法；生命周期管理在 BranchFS 路线图上——三块分别已被先占。（doc22:177 攻击3）

**③ 一句话**：足迹观测普通 BPF 就够（不需要名字解析钩子），原子发布 symlink 就够，谓词/冲突/生命周期三块分别被 STORM/Rattle/BranchFS 先占。

#### 场景 7 — 丰富动作类候选 K1-K4

这四个方向（治理 / 可逆审计 / 上下文成本 / 零物化语义视图）是"丰富动作"的新颖性本该落地的地方。裁判对这场专门的杀手级动作探索给出的总结论逐字是：**`the_killer = none`**。四个都退化成了用现有的固定机制就能实现的方案：

**K1 — 治理 / 信息流执行层**

- **优势**：在名字解析钩子上附加标签，按标签做文件通道的信息流控制（即限定"哪个进程能访问哪些文件"的规则）。
- **限制**：唯一新的东西"agent 自报标签"在威胁模型下不可信（agent 自己可以伪造标签），退回可信的静态标签就等于 **P4Control（CCS'24）** 的文件通道版（P4Control 是一个利用 eBPF 在内核中实施信息流策略的系统）；而且主要的信息外泄通道是网络，VFS 只管文件这一条；丰富动作完全没用上——一个带标签的 bpf-lsm（只做允许/拒绝决策）就够了。（doc08:111/113/115）（你们组内部亦有相关信息流实现，此处不展开。）

**K2 — 可逆 + 可审计的世界边界 / 投机执行**

- **优势**：在文件系统层做"按步骤记录完整操作历史、事后可整层保留或丢弃"。
- **限制**：剥掉包装就是 **CamFlow**（内核 LSM 溯源系统，即在内核安全模块层记录进程对数据对象的所有操作形成因果图）加上版本化文件系统。"按步骤整层快照保留或丢弃"随手扩展已有的文件系统就能做到；真正难的"操作级选择性撤销"（只撤消某一步操作而保留其他步骤的结果）是大家都没有低成本解法的 RETRO 问题（重执行开销 35-127%、可能不收敛），不是可编程 VFS 能解决的；mmap 写侧还在模型之外。（doc08:131/183/210-212/133/135）**这是四个候选里排名第一、唯一没被"信息流"和"网络旁路"两个问题直接否决的**——但它仍然是 CamFlow 的增量。（你们组内部亦有相关信息流实现，此处不展开。）

**K3 — 文件系统即上下文 / 成本管理层**

- **优势**：在 VFS 层按策略控制 agent 读到的文件内容（例如只返回代码骨架而非完整文件），从而减少 token 消耗。
- **限制**：RTK（`read -l aggressive` 返回代码骨架）、Headroom、LangChain Deep Agents 已经在框架/代理层做到了同样的事情（实测削减 60-90% token）。把 read 重定向到骨架版会破坏字节级契约——seek 偏移、mmap、行号、哈希、编译报错都会出错。而唯一的真空地带（子进程直接 cat/open）恰好和"需要精确内容的消费者"（编译器、测试、哈希校验）走同一条代码路径，会损坏它声称要保护的 build 和 test。VFS 钩子只能拿到 `{event, name, hash, cgroup}` 这些信息，结构上缺少"当前任务/剩余 token 预算"这类语义。（doc08:286/288）

**K4 — 零物化语义视图**

- **优势**：多个逻辑树共享内容寻址的只读底层、内核内按需物化、接近原生速度、无 FUSE、page cache 共享。
- **限制**：composefs + EROFS-over-fscache / Nydus RAFS 已经在内核里做到了上述全部能力——**静态场景已经被解决**。唯一真正新的动作（就地内容转换/脱敏）超出 verifier 上限，退回 FUSE。（doc08:294/296）

**总判**：动作空间两端受限，让 K4 与众不同的"接近原生速度的丰富动作"，其可行的动作集合很可能是空的。裁判对这场杀手级探索的结论逐字是 `the_killer = none`。

#### 其余候选（六轮中也评估过，同样确有一点优势，但同样被判否；一行一条）

- **viewfd 通用视图对象（DROP）**：把"视图"做成可传递的一等内核对象。非文件消费者（SysV key、消息队列名、抽象 socket 名）找不到一个今天在实际使用的具名用户；新颖性外壳被 lwC（OSDI'16，切换 2.01µs）、Plan 9（1992）、主线 namespace-as-fd、本组 branch context（arXiv 2602.08199）占掉；主线还在把 namespace 做成更低成本的 fd（setns/open_tree/Linux 7.1 FSMOUNT_NAMESPACE），持续缩小成本差距。（doc23:123/125）

- **内存态组合分支（DROP）**：文件视图加内存快照绑定到同一视图号。撞 TClone（arXiv 2605.17320，已实现内核级同时分支内存加文件系统）和本组自己的 Fork/Explore/Commit（arXiv 2602.08199，作者含郑昱笙，已公开占据"统一 branch() 覆盖文件加进程态"这一愿景）；内存没有名字层，namei_ext 对内存那一半完全施加不了作用。（doc23:113/115）

- **服务 DeltaBox（DROP）**：把 namei_ext 定位成给 DeltaBox 做底层服务。表达力不够——钩子只有 LOOKUP/READDIR（namei_ext.h:12-13），create 期返回 -EOPNOTSUPP（pass-redirect:51-53），写侧全部生命周期（copy-up/合并/commit/abort/快照）无法表达，而 DeltaFS 要的正是写侧加块级 CoW（Copy-on-Write）；DeltaBox 扇出的瓶颈在进程侧 fork（p50 从 0.57ms 升到 5.5ms），名字解析层碰不到这个瓶颈；DeltaBox 闭源（"artifacts…are not currently public"），无法真正集成也无法第三方验证。（doc24:24 / doc23:86/78）

- **反事实二分调试（DROP）**：微秒级合成"混合世界"做差异二分定位。被 bwrap 一条命令（裁判实测 32ms/次）覆盖，加上写隔离缺失（裸跑时撞共享 `__pycache__` 导致判定静默出错），加上 Zeller 1999 / Chronus（OSDI'04）占据谱系。（doc22:261/248）

- **保留世界重放（DROP）**：事后重建"第 k 步看到的整个世界"。被 btrfs 子卷快照 O(1) + Turso AgentFS 覆盖；消费频率约等于 1（一次 overlay+chroot 或 Morph 250ms 就够）。（doc22:214）

- **进程内 agent 会话群（DROP）**：每个会话一份内核文件视图。容器税买的是"运行不受信任的原生代码的安全边界"（Cloudflare 原文 "run untrusted code securely"）而不是文件系统；运行时线程模型对不上（Node/libuv 卸载到共享线程池、io_uring 在 io-wq 线程执行，钩子按 current 找到的不是提交者）；`unshare(CLONE_FS)` + chdir 约 50 行用户态代码覆盖旗舰示例。（doc22:107/103/144）

- **线程并行测试（DROP）**：每线程一份内核文件视图。nextest 的《Why process-per-test?》理由清单里根本没有"文件冲突"这一条。（doc22:138）

- **常驻 worker 每请求视图（DROP）**："5.15 内核里线程 `unshare(CLONE_FS)` + `CLONE_NEWNS` 就给同进程两线程两个不同视图"——裁判以 root 和 uid 65534 两次复现，证伪了独占支柱。而且 Bazel worker 协议里没有线程身份字段，外部无从按 tid 绑定。**这里提炼出一条筛选器：凡是"该绑定到哪个线程"的信息只存在于目标进程内部、外部拿不到的场景，外部绑定的独占优势自动失效。**（doc22:298）

- **发行版 epoch 翻转升级（DROP）**：被 merged-/usr + `renameat2(RENAME_EXCHANGE)`（单个系统调用原子对换整棵 /usr）覆盖——AerynOS 的 moss 已经在生产中使用、不需要重启、零内核改动。而真正让发行版升级难受的 half-configured 状态（升级到一半出错导致系统处于不一致状态），其痛点在维护者脚本的执行阶段，名字层没有对应的解决手段。（doc22:341）

- **万级 agent 供给（DROP）**：pkgenv 已全面占据测量角色，四个具名平台全指向多内核架构。（doc19:143/145）

- **tmpfs 分支（DROP）**：overlay-on-tmpfs 本就共享同一批内存页；Shepherd（arXiv 2605.10913）的主基准平台就是它。（doc23:105）

- **服务证书轮换（BACKUP）**：kubelet AtomicWriter / SPIRE / 单文件 PEM 捆绑已经覆盖。（doc18:220-224）

- **ML 数据子集（DROP）**：应用层的 is_valid_file / Subset 清单零内核改动就能做到。（doc17:624）

- **bazel 沙箱（BACKUP）**：Bazel 8.3.0 默认开启 reuse_sandbox_directories，官方数据中位沙箱开销不过 5%。（doc17:148/192）

- **大仓虚拟检出（ADOPT_MODIFIED，仅作基线臂）**：EdenFS 的存在理由是**懒取**（未物化的 inode 只存 20 字节对象 ID，按访问时再从远端拉取内容），而 namei_ext 只能重定向到**已物化**的后端、不生成也不拉取内容——拿 EdenFS 当动机是名实不符。而且三大厂的用法是"每开发者一份虚拟检出"（N 约等于 1），不是"同机万份互不相同的视图"的证人。（doc19:156/160）

- **casefold / 策略多样性（BUILD，作策略多样性主打行）**：同一钩子想证明能表达 ≥6 种策略族。但 overlayfs casefold 已并入 Linux 6.18、`tune2fs -O casefold` 可事后开启，部署差异化的两根支柱被拆掉。Wine 的用户态每目录缓存（约 2.3µs/项）与 namei_ext 的 2-3µs 同量级。而且这一行本质上是"固定动作 + 一张守护进程维护的表"（配置而不是程序），不能拿来主张动态可编程性。还有一个硬门槛：表里至少 6 行要真实跑通，只跑 2 行会以"覆盖面太窄"被拒。（doc17:761/762 / doc20:151 / doc16:357）

### D. 跨场景的共同规律——这才是"窄"的真正证据

把上面二十来个候选场景放到一起看，挡住它们的**始终是同样这几类原因反复出现**。不是某一个场景运气不好碰到了障碍，而是结构性的：

1. **一行配置的现成近邻**：nginx `$realpath_root` / mod_realdoc、K8s AtomicWriter（`..data` rename）、Envoy symlink 轮换、Python `.pth`（529µs）、SOCK 的 <1µs chroot、`unshare(CLONE_FS)` / `unshare(CLONE_NEWNS)`（20-50 行）、btrfs 子卷快照 O(1)、`renameat2(RENAME_EXCHANGE)`（moss）、bwrap（32ms）、Apptainer squashfuse、单文件 PEM 捆绑、Bazel 8.3.0 默认沙箱复用、is_valid_file / Subset。——命中场景 2、3、4、5、6 及大部分 DROP 项。

2. **层错位（真正的负载在别的层执行，namei_ext 所在的层碰不到）**：生产高密度在 microVM（Firecracker/RunD）或语言隔离（V8/WASM/isolate）中执行；内存没有名字层；发行版升级的痛点在维护者脚本；DeltaBox 的瓶颈在进程 fork/CRIU；容器税买的是安全边界不是文件系统；运行时线程池（libuv/io_uring）的调度方式和钩子按 current（当前执行线程）查找的方式对不上。——命中场景 1、2、内存分支、服务 DeltaBox、进程内会话群、发行版升级、万级供给。

3. **能力边界（只能选择或隐藏已有对象，写侧在模型之外）**：只有 LOOKUP/READDIR 两个钩子、create 期返回 -EOPNOTSUPP、写侧全部生命周期无法表达、mmap 写绕开钩子、"外部绑定需要只存在于目标进程内部的信息"使得外部绑定失效；单分量 REDIRECT 表达不了整棵输入树；策略退化成纯查找表算不上可编程。——命中场景 1、2、6、服务 DeltaBox、常驻 worker、K2、K3。

4. **决定性的验证标准可被零内核方案通过**：实验 B 被 symlink + rename 通过、pkgenv 被 conda 前缀 + unshare + bind 通过、casefold 被 Wine 用户态缓存通过、证书轮换被超时钉定通过、FaaS 被用户态 rename 通过。——命中场景 2、3、4、5、casefold。

5. **撞车（本组或他人已在相同位置上占据了优先权）**：CamFlow、P4Control（CCS'24）、composefs/EROFS/Nydus、lwC（OSDI'16）、Plan 9、TClone（arXiv 2605.17320）、本组 Fork/Explore/Commit（arXiv 2602.08199，作者含郑昱笙）、YoloFS（arXiv 2604.13536）、Zeller 1999 / Chronus（OSDI'04）、STORM / Rattle、Turso AgentFS、Shepherd（arXiv 2605.10913）、TxOS（SOSP'09）、EdenFS / VFS-for-Git / CitC。——几乎每个场景至少命中一条。

**这五类限制反复出现的意义**：一个场景要真正成为杀手级，需要**同时**躲过这五类限制。六轮评估下来，没有一个候选场景做到了。这就是为什么"没有杀手级应用"不是一句情绪判断，而是二十来次具体尝试、反复遇到同样这五类限制后得出的结论。

---

### E. 合作者转体裁（"能力论文"）的评估：对在哪、没解决什么（双向诚实）

**对的部分（要承认）**：能力/边界体裁是真实存在的论文体裁。Bento（FAST'21）、lwC（OSDI'16）、sched_ext 都没有杀手级的性能提升也中了顶会。所以"你没有杀手级性能场景"这件事本身不致命。这是我们退掉 Amdahl 那条攻击的原因。

**没解决的部分（核心风险）**：能力论文仍然要求那个能力**既新颖又被需要**。转体裁把 Amdahl/密度的靶子撤了，但没有回答上面 B/C/D 提出的问题——到底存不存在一个"新颖且被需要"的动作。而且连"位置本身新颖"这条退路也被削弱：sched_ext 和 cache_ext 各自在自己的位置上解决了一个真正的硬问题（sched_ext 做到了不重编译内核就能换调度策略；cache_ext 用 eBPF 定制页缓存淘汰），而 namei_ext 位置上的真正硬问题——RCU-walk（内核路径解析的无锁快路径，大多数 lookup 不拿任何锁、开销极低）里不退出无锁快路径就完成判断、跨视图 dcache（目录项缓存）一致性、readdir 的逆映射（从底层目录项反查出当前视图应该展示什么内容）——论文一个都没有触碰。这些术语在全部论文 .tex 文件中零命中。位置新颖要成立，需要先解决位置上的硬问题。

### F. 结论与可降级条件（什么出现了，这条险要提示才可以降低优先级）

**结论**：不是被合作者的 AI 说服了，而是"删除主张"把旧攻击的靶子撤掉了。真正的"窄 / 无杀手"担忧没有被驳倒，只是不再被主张、所以显得消失了。逐场景看，每个有优势的候选都被过多且**同类**的限制条件挡在杀手级之外——这是当前最可能让论文在 FAST/OSDI 被拒的地方。

**可降级条件（出现以下任一条，这条险要提示可以降低优先级）**：

1. 出现一个"新颖且被需要"的动作：在钩子能拿到的 `{event, name, hash, cgroup}` 上语义可执行、agent 确实急需、不被 composefs / overlayfs / bpf-lsm / CamFlow 覆盖、且能通过 eBPF verifier；
2. 实验 B 加上"多个 cgroup 在同一路径前缀下并行不同 epoch"的判别条件，并证明 symlink/overlay 在不给每容器单独挂载的前提下通过不了（= I-2 的修法）；
3. 论文正面处理并解决位置上的硬问题（RCU-walk 不退出快路径 / dcache 一致性 / readdir 逆映射），把"位置新颖"坐实为"机制难点新颖"。

在这些条件出现之前，这条险要提示维持在全篇最高优先级。

## 一、想法层面的问题

**1. 论文三处把物化机制拉进论证，却不给一行成本数据。**

把 overlayfs（一个在挂载时把视图规则固定下来的内核文件系统，不能承载逐次查找时现算的策略代码）排除出 RQ2 的逐次查找成本对照本身可辩护——问题不在这。问题是论文自己立了三个需要物化机制数据的承诺，一个都没兑现：

- H2 假设的证伪条件逐字写着 "A fair FUSE implementation cannot be made feature-equivalent for the admitted oracle, **or another mechanism is a stronger direct cost opponent**"（idea-story.md:101），但实验矩阵没有任何一格去检查这第二支。
- 实验 A 的 oracle 自述来源为 "SOURCE_BEHAVIOR=bash/git overlay actions over AgentFS COW sandbox"（agentfs_lifecycle_trace.txt:3），是 overlayfs 的教科书用例（base/upper + whiteout 隐藏），已触发 idea-story.md:128 自写的重开条件 "If a selected source oracle makes a materialized mechanism the natural direct opponent"。
- 机制阶梯 "bind/Overlay/materialization < eBPF LSM < namei_ext < FUSE/custom"（evaluation.md:12-13）把物化机制放在 namei_ext 正下方一级，intro 断言这些机制 "must materialize or update a view"（01-introduction.tex:24-25），但对它们的成本零测量，而阶梯上一档的 FUSE 有实测行。

修法：实验 A、B 各加一条同 oracle 的物化参照行（overlay 或 symlink 森林，测端到端宏观时间 + 视图构建/切换成本，标注非 RQ2 对照），接到 H2 证伪条件第二支的检查上。

---

**2. 实验 B（合作者定为 decisive）的五态 oracle 可被"用户态标签管理器 + 一行原子 symlink 翻转"整体通过。**

内容验证在钩子外、两边拿同一份标签——论文明写 "cache population, content validation, build/test execution, and writes stay outside the hook"（05-evaluation.tex:76-78）、"namei_ext and FUSE receive the same cache-validity labels…"（2026-07-13-environment-cache-complete-experiment-plan.md:173-175）。eBPF 策略读不了文件内容，所以"这个缓存对象是不是新鲜的"这个判定必然由用户态验证器算出来再下发。既然用户态在每个状态切换的那一刻已经知道标签，同一刻它完全可以改一个 symlink（符号链接——一个指向另一路径的特殊文件，改它的指向就改变了后续路径查找的结果）。

逐态检验（oracle 定义在 plan:139-150）：hit/miss 静态 symlink 即满足；stale/corrupt 只要求 "stale/corrupt object is not read; source oracle passes"（plan:143-144），验证器打标签那刻 unlink/rename 即满足；epoch 要求 "old epoch does not leak into new source oracle"（plan:145），目录级 symlink 的 rename() 是 POSIX 原子操作，一行 `mv -T` 满足。五个状态全部能被这个方案通过。

唯一能区分 namei_ext 和 symlink 方案的结构差异是：同一路径前缀下多个 cgroup（Linux 内核的进程分组机制）同时看到不同 epoch——symlink 是全局单值做不到这件事。但 plan 全文里 cgroup 只作为挂策略的路径出现（plan:38、:155、:215），没有多 cgroup 并行视图的实验格；现有实现全程单 target id、单 cgroup。

修法：在实验 B 预注册一个"两个 cgroup 在同一固定路径前缀下同时以不同 epoch 各跑 source oracle，并证明 symlink/bind/overlay 在不给每容器单独挂载的前提下过不了同一格"的实验格。没有这个格，实验 B 的 decisive 地位不成立。

---

**3. RQ3 的问法按定义就成立，且把"比自建文件系统更安全"立成三大支柱之一。**

RQ3 原文 "Does namei_ext provide a narrower verifier-bounded, fail-closed ownership boundary than building a custom or stackable filesystem…"（05-evaluation.tex:19-22）。这里 verifier-bounded 指 eBPF 验证器对策略程序施加的边界约束，fail-closed 指遇到机制不支持的操作时拒绝并返回错误。"一个只做名字解析决策的接口比一个需要实现完整文件系统操作集的方案边界更窄"按定义就成立，不需要实验来证明。

而把"更安全"当贡献，与扩展点论文的惯例冲突——安全性是扩展点论文的准入条件，不是贡献本身；被接受的扩展点论文通常同时交付三样东西：未挂策略时的开销接近零、热路径上打赢现有固定功能方案、真实的策略多样性证据。在 canonical 文档里这三样都缺：pass-only 不超过 1.1 倍与 cache_ext 不超过 1.7% 的阈值 grep 零命中；策略多样性没有任何实验设计；RQ3 的 custom/stackable 那行是用 printf 打印的 JSON（mk/kvm.mk:296），不是跑出来的测量。

修法：给 RQ3 补"策略多样性（R1-R6 多个策略族各跑通）"与"未挂时开销接近零"两项独立于安全性的证据，并把 custom/stackable 行从 printf 改成有代码支撑的测量。

## 二、实现层面的问题

**1.（紧急，最先做）新方向的内核实现不在任何版本库里。**

origin/main 的 kernel gitlink 与 kernel_nameiext 的 HEAD/master 都停在 9f6695a（2026-07-03，"namei_ext: expose parent identity in BPF ctx"）。该 commit 的 fs/namei_ext.c 共 186 行，switch(action) 只有 case BPF_NAMEI_EXT_PASS 和 REDIRECT（行 91-94、129-141）；HIDE、SELECT_TARGET、target_id 命中 0。而 ABI 头（bpf/include/namei_ext.h:15-18）定义了 HIDE 2 / SELECT_TARGET 3，全部实验都依赖它们。

07-13 起的内核代码是合作者写的，只在会被重启的共享机 6787p 上。BPF 策略与用户态测试已随 179e642 入库，缺的是内核侧机制实现与原始结果。这不是作者标注过的待办。

修法：内核仓 commit + push + bump gitlink，并立刻备份。修法本身几分钟，前提是 6787p 工作树还在。

**2. RCU 降级与旁观者开销：机制在论文里未被承认，且预注册阈值已移除无替代。**

机制说明：挂上任何一个 namei_ext 策略，一个全局 static key（内核里一种低开销的全局条件跳转：不开时分支编译成 nop 接近零成本；一开，所有执行到该点的 CPU 都走新路径）打开。打开后，所有进程的路径解析在钩子点放弃 RCU-walk（内核解析路径名的无锁快路径，靠 RCU 保证安全、不加锁不加引用计数），退到 ref-walk（加锁加引用计数的慢路径）。这个"放弃"发生在检查"本 cgroup 有没有挂策略"之前，所以不使用视图的旁观者进程也付降级代价。方向可从代码记录推定（全局 static key + -ECHILD 退回慢路径在 per-cgroup 检查之前 + 7 月设计文档），幅度未测。现有实验把策略挂 cgroup 根（namei_ext_agent_workspace.c:725；mk/kvm.mk:292），per-cgroup 隔离无演示物。

数字（须标适用范围，不泛化）：pass-only（挂了策略但钩子只返回放行）相对原生的 p99 比值——旧 batch=64 协议（附着被测 cgroup）1.71x；ctx-init-split 优化同协议 1.095x；密尾采样最大 2.624x。当前内核挂钩 stat 1065 vs 无钩 611 ns/op（均值比约 1.74x，但路径深度 2 vs 3 级、时序、run 间约 2 倍噪声，不可信）。两轮 RCU 修复 PoC（pass-only 1.48x/1.38x）因 redirect 恶化 2.49x/2.43x 撤回。当前内核干净 pass-only 残差是未知数。

两处未被论文承认：

- 旁观者也付代价这一机制含义，论文和设计文档零处提及。即使最终填上数字，论文不说明这一点，审稿人看到"挂了策略"与"没挂"的差距会追问其他进程有没有受影响。
- pass-only 不超过 1.10x 的阈值随旧计划 2026-07-12 归档（变量还在 mk/eval_osdi.mk:258），新矩阵无任何开销阈值、无替代披露要求。cache_ext（SOSP'25）无操作开销不超过 1.7% 的先例摆在那里，本项目对应区间约 +9% 到 +162%。

不夸大：RQ2 对 FUSE 那一比的方向安全（约 13-21x 便宜，历史最小 policy/FUSE p99 加速 2.258x），残差不翻转；负结果留痕、无诚信问题。

修法：补两条对照臂——(a) 同路径同状态 pass-only 挂/不挂（路径深度一致、至少 30 次重复、p50/p95/p99 + 95% bootstrap CI）；(b) 旁观者臂（策略挂非根非被测 cgroup，在未挂 cgroup 测同一微基准）。在论文正文承认旁观者开销这一机制含义。确切数字要等这两条臂跑出来——归文末第三节；但"机制未被承认 + 阈值已移除"是现在就成立的问题。

**3. 论文正文对写侧 fail-closed 缺口表述过宽且披露不全。**

SELECT_TARGET 当前只支持目录选择。对 create、非目录的最终 open、最终文件级 target selection、合成父目录别名这四项 fail-closed（design.md:46-48 逐字 "It still fails closed for create, non-directory final opens, final file target selection, and synthetic parent-directory aliases"；implementation.md:55-58 同款）。但 04-implementation.tex:40-42 写 "supports intermediate components and final selected lower objects"，自然读法含文件。四个缺口正文只披露两个（04-implementation.tex:44-46 只提 synthetic aliases 与 create），limitations 只披露一个（07-limitations.tex:22-23）。与此同时 02-motivation.tex:126-127 与 05-evaluation.tex:74-79 把第二工作负载押在文件级 SELECT_TARGET 上——恰好落在未披露的缺口里。

一个值得注意的取舍：同一 fail-closed 行为在 RQ3 被当安全优点展示，而为覆盖文件级选择去扩大动作范围又会削弱"窄边界"叙事。需要说明：扩大动作覆盖（让机制支持更多操作类型）和放宽对畸形返回值的验证（对格式错误的 eBPF 返回值不再拒绝）是两件不同性质的事，前者是功能扩展，后者才是安全退让。

修法：把 04-implementation.tex:40-42 改成与内部文档一致的准确措辞（目录选择，列全四项 fail-closed），limitations 补齐，02/05 处标注 env/cache 行依赖尚未实现的文件级选择。

**4. FUSE 基线是最悲观配置 + 指标不对称。**

libfuse2（FUSE_USE_VERSION 26）+ 单线程 -s + 缓存全关（attr/entry/negative_timeout=0）；21x/13x 建立其上。FUSE 侧差距最小的 exec 恰好没在 FUSE 侧计时。论文自己在 refs.bib:440-441 注释写了 "must not be framed as generic unoptimized FUSE"。配置诚实记录了（fuse.c:1127-1128），所以问题不是"没做"，而是这个配置选择 + 指标不对称会被审稿人当作挑选对自己有利的数字。

修法：FUSE 缓存三行矩阵（coherent/default/passthrough）+ libfuse3；补齐 open/access/exec/macro 在 FUSE 侧的对称计时。
