# 给合作者 · 新方向技术担忧清单（2026-07-16）

评估对象：origin/main 提交 179e642（2026-07-15）的 idea-story.md 与论文草稿。方向（sched_ext 式 VFS 名字解析扩展点、三个 RQ、不卖提速）已与第六轮红蓝判断收敛，下面只列仍需处理的问题。

---

## 一、想法层面的问题

**1. 论文三处把物化机制拉进论证，却不给一行成本数据。**

把 overlayfs（一个在挂载时把视图规则固定下来的内核文件系统，不能承载逐次查找时现算的策略代码）排除出 RQ2 的逐次查找成本对照本身可辩护——问题不在这。问题是论文自己立了三个需要物化机制数据的承诺，一个都没兑现：

- H2 假设的证伪条件逐字写着 "A fair FUSE implementation cannot be made feature-equivalent for the admitted oracle, **or another mechanism is a stronger direct cost opponent**"（idea-story.md:101），但实验矩阵没有任何一格去检查这第二支。
- 实验 A 的 oracle 自述来源为 "SOURCE_BEHAVIOR=bash/git overlay actions over AgentFS COW sandbox"（agentfs_lifecycle_trace.txt:3），是 overlayfs 的教科书用例（base/upper + whiteout 隐藏），已触发 idea-story.md:128 自写的重开条件 "If a selected source oracle makes a materialized mechanism the natural direct opponent"。
- 机制阶梯 "bind/Overlay/materialization < eBPF LSM < namei_ext < FUSE/custom"（evaluation.md:12-13）把物化机制放在 namei_ext 正下方一级，intro 断言这些机制 "must materialize or update a view"（01-introduction.tex:24-25），但对它们的成本零测量，而阶梯上一档的 FUSE 有实测行。

修法：实验 A、B 各加一条同 oracle 的物化参照行（overlay 或 symlink 森林，测端到端宏观时间 + 视图构建/切换成本，标注非 RQ2 对照），接到 H2 证伪条件第二支的检查上。

---

**2. 实验 B（合作者定为 decisive）的五态 oracle 可被"用户态标签管理器 + 一行原子 symlink 翻转"整体通过。**

内容验证在钩子外、两边拿同一份标签——论文明写 "cache population, content validation, build/test execution, and writes stay outside the hook"（05-evaluation.tex:76-78）、"namei_ext and FUSE receive the same cache-validity labels…"（2026-07-13-environment-cache-complete-experiment-plan.md:173-175）。eBPF 策略读不了文件内容，所以"这个缓存对象是不是新鲜的"这个判定必然由用户态验证器算出来再下发。既然用户态在每个状态切换的那一刻已经知道标签，同一刻它完全可以改一个 symlink（符号链接——一个指向另一路径的特殊文件，改它的指向就改变了后续路径查找的结果）。

逐态检验（oracle 定义在 plan:139-150）：hit/miss 静态 symlink 即满足；stale/corrupt 只要求 "stale/corrupt object is not read; source oracle passes"（plan:143-144），验证器打标签那刻 unlink/rename 即满足；epoch 要求 "old epoch does not leak into new source oracle"（plan:145），目录级 symlink 的 rename() 是 POSIX 原子操作，一行 `mv -T` 满足。五个状态全部能被这个方案通过。

唯一能区分 namei_ext 和 symlink 方案的结构差异是：同一路径前缀下多个 cgroup（Linux 内核的进程分组机制）同时看到不同 epoch——symlink 是全局单值做不到这件事。但 plan 全文里 cgroup 只作为挂策略的路径出现（plan:38、:155、:215），没有多 cgroup 并行视图的实验格；现有实现全程单 target id、单 cgroup。

修法：在实验 B 预注册一个"两个 cgroup 在同一固定路径前缀下同时以不同 epoch 各跑 source oracle，并证明 symlink/bind/overlay 在不给每容器单独挂载的前提下过不了同一格"的实验格。没有这个格，实验 B 的 decisive 地位不成立。

---

**3. RQ3 的问法按定义就成立，且把"比自建文件系统更安全"立成三大支柱之一。**

RQ3 原文 "Does namei_ext provide a narrower verifier-bounded, fail-closed ownership boundary than building a custom or stackable filesystem…"（05-evaluation.tex:19-22）。这里 verifier-bounded 指 eBPF 验证器对策略程序施加的边界约束，fail-closed 指遇到机制不支持的操作时拒绝并返回错误。"一个只做名字解析决策的接口比一个需要实现完整文件系统操作集的方案边界更窄"按定义就成立，不需要实验来证明。

而把"更安全"当贡献，与扩展点论文的惯例冲突——安全性是扩展点论文的准入条件，不是贡献本身；被接受的扩展点论文通常同时交付三样东西：未挂策略时的开销接近零、热路径上打赢现有固定功能方案、真实的策略多样性证据。在 canonical 文档里这三样都缺：pass-only 不超过 1.1 倍与 cache_ext 不超过 1.7% 的阈值 grep 零命中；策略多样性没有任何实验设计；RQ3 的 custom/stackable 那行是用 printf 打印的 JSON（mk/kvm.mk:296），不是跑出来的测量。

修法：给 RQ3 补"策略多样性（R1-R6 多个策略族各跑通）"与"未挂时开销接近零"两项独立于安全性的证据，并把 custom/stackable 行从 printf 改成有代码支撑的测量。

**4. 动机里的痛点没有具名代价。**

七个来源系统全是 GitHub 仓库引用（agentfs/branchfs/sandlock/yolofs/swe_factory/menvagent/swe_rebench_v2，全部 @misc 条目，Accessed 2026-07-10），痛全是 "can" 假设句（01-introduction.tex:12-15 "Binding a name to the wrong lower object **can** validate the wrong revision, **can** leak a write…"）。没有一条"来源系统自己说过被这问题困住"的具名代价。

修法零研究成本，纯搬运：doc 24 已备好具名工程代价——DeltaBox 定制 6.8 内核 + 565 行 ioctl、TClone 改内核 + CRIU、YoloFS 2.5kLoC 内核模块、Bazel 五代沙箱 + sandboxfs 在 7.0.0 被移除归档——搬进动机即可。

**5. 撞车名单未处理，且新 RQ2 让部分缺席从"缺引"升级为"基线少一条臂"。**

FUSE-BPF（在 FUSE 框架内部用 BPF 程序加速文件系统操作的 Google 补丁集，机制上是 namei_ext 最近的已有工作）只一句 "keeping policy in a FUSE request path"（06-related-work.tex:12-19）带过，停滞 27 个月（自 2024-03 RFC v4）未提。cache_ext（SOSP'25）、XRP、bpftime、composefs 全零引用。本组 BranchFS 论文 arXiv 2602.08199 只引仓库不引论文。

新 RQ2 以 FUSE 为中心做成本对照，FUSE-BPF / ExtFUSE / FUSE passthrough（Linux 5.18 起内核支持的让 FUSE 直接转发读写到底层文件的特性，减少一次用户态来回）的缺席，从"相关工作缺引"升级为"成本基线少一条更快的 FUSE 变体臂"。

修法：补引 + 把更快的 FUSE 变体纳入 RQ2 基线集。

**6.（minor）SELECT_TARGET 依赖的"目标注册"步骤在论文全文里没写。**

main.tex:39-41 "kernel-registered lower paths" 但"注册"动作（谁注册、什么接口、什么特权、可不可变）零描述；"registered" 只作定语出现（01-introduction.tex:65 等）。已跑实验的世代切换正是经这个没被描述的 debugfs 接口完成的（namei_ext_agent_workspace.c:642, 951, 968）。

修法：补一段注册契约——(cgroup_id, target_id) 到内核 struct path、以 O_PATH fd 输入、特权要求、path_get 生命周期，依 2026-07-13-registered-target-selection-design.md:70-88；注明 debugfs 是 Phase-1 临时接口。在 RQ3 责任表把这个特权注册面记进 namei_ext 自己一列。

## 二、实现层面的问题

**1.（紧急，最先做）新方向的内核实现不在任何版本库里。**

origin/main 的 kernel gitlink 与 kernel_nameiext 的 HEAD/master 都停在 9f6695a（2026-07-03，"namei_ext: expose parent identity in BPF ctx"）。该 commit 的 fs/namei_ext.c 共 186 行，switch(action) 只有 case BPF_NAMEI_EXT_PASS 和 REDIRECT（行 91-94、129-141）；HIDE、SELECT_TARGET、target_id 命中 0。而 ABI 头（bpf/include/namei_ext.h:15-18）定义了 HIDE 2 / SELECT_TARGET 3，全部实验都依赖它们。

07-13 起的内核代码是合作者写的，只在会被重启的共享机 6787p 上。BPF 策略与用户态测试已随 179e642 入库，缺的是内核侧机制实现与原始结果。这不是作者标注过的待办。

修法：内核仓 commit + push + bump gitlink，并立刻备份。修法本身几分钟，前提是 6787p 工作树还在。

---

**2. RCU 降级与旁观者开销：机制在论文里未被承认，且预注册阈值已移除无替代。**

机制说明：挂上任何一个 namei_ext 策略，一个全局 static key（内核里一种低开销的全局条件跳转：不开时分支编译成 nop 接近零成本；一开，所有执行到该点的 CPU 都走新路径）打开。打开后，所有进程的路径解析在钩子点放弃 RCU-walk（内核解析路径名的无锁快路径，靠 RCU 保证安全、不加锁不加引用计数），退到 ref-walk（加锁加引用计数的慢路径）。这个"放弃"发生在检查"本 cgroup 有没有挂策略"之前，所以不使用视图的旁观者进程也付降级代价。方向可从代码记录推定（全局 static key + -ECHILD 退回慢路径在 per-cgroup 检查之前 + 7 月设计文档），幅度未测。现有实验把策略挂 cgroup 根（namei_ext_agent_workspace.c:725；mk/kvm.mk:292），per-cgroup 隔离无演示物。

数字（须标适用范围，不泛化）：pass-only（挂了策略但钩子只返回放行）相对原生的 p99 比值——旧 batch=64 协议（附着被测 cgroup）1.71x；ctx-init-split 优化同协议 1.095x；密尾采样最大 2.624x。当前内核挂钩 stat 1065 vs 无钩 611 ns/op（均值比约 1.74x，但路径深度 2 vs 3 级、时序、run 间约 2 倍噪声，不可信）。两轮 RCU 修复 PoC（pass-only 1.48x/1.38x）因 redirect 恶化 2.49x/2.43x 撤回。当前内核干净 pass-only 残差是未知数。

两处未被论文承认：

- 旁观者也付代价这一机制含义，论文和设计文档零处提及。即使最终填上数字，论文不说明这一点，审稿人看到"挂了策略"与"没挂"的差距会追问其他进程有没有受影响。
- pass-only 不超过 1.10x 的阈值随旧计划 2026-07-12 归档（变量还在 mk/eval_osdi.mk:258），新矩阵无任何开销阈值、无替代披露要求。cache_ext（SOSP'25）无操作开销不超过 1.7% 的先例摆在那里，本项目对应区间约 +9% 到 +162%。

不夸大：RQ2 对 FUSE 那一比的方向安全（约 13-21x 便宜，历史最小 policy/FUSE p99 加速 2.258x），残差不翻转；负结果留痕、无诚信问题。

修法：补两条对照臂——(a) 同路径同状态 pass-only 挂/不挂（路径深度一致、至少 30 次重复、p50/p95/p99 + 95% bootstrap CI）；(b) 旁观者臂（策略挂非根非被测 cgroup，在未挂 cgroup 测同一微基准）。在论文正文承认旁观者开销这一机制含义。确切数字要等这两条臂跑出来——归文末第三节；但"机制未被承认 + 阈值已移除"是现在就成立的问题。

**3. 论文正文对写侧 fail-closed 缺口表述过宽且披露不全。**

SELECT_TARGET 当前只支持目录选择。对 create、非目录的最终 open、最终文件级 target selection、合成父目录别名这四项 fail-closed（design.md:46-48 逐字 "It still fails closed for create, non-directory final opens, final file target selection, and synthetic parent-directory aliases"；implementation.md:55-58 同款）。但 04-implementation.tex:40-42 写 "supports intermediate components and final selected lower objects"，自然读法含文件。四个缺口正文只披露两个（04-implementation.tex:44-46 只提 synthetic aliases 与 create），limitations 只披露一个（07-limitations.tex:22-23）。与此同时 02-motivation.tex:126-127 与 05-evaluation.tex:74-79 把第二工作负载押在文件级 SELECT_TARGET 上——恰好落在未披露的缺口里。

一个值得注意的取舍：同一 fail-closed 行为在 RQ3 被当安全优点展示，而为覆盖文件级选择去扩大动作范围又会削弱"窄边界"叙事。需要说明：扩大动作覆盖（让机制支持更多操作类型）和放宽对畸形返回值的验证（对格式错误的 eBPF 返回值不再拒绝）是两件不同性质的事，前者是功能扩展，后者才是安全退让。

修法：把 04-implementation.tex:40-42 改成与内部文档一致的准确措辞（目录选择，列全四项 fail-closed），limitations 补齐，02/05 处标注 env/cache 行依赖尚未实现的文件级选择。

---

**4. FUSE 基线是最悲观配置 + 指标不对称。**

libfuse2（FUSE_USE_VERSION 26）+ 单线程 -s + 缓存全关（attr/entry/negative_timeout=0）；21x/13x 建立其上。FUSE 侧差距最小的 exec 恰好没在 FUSE 侧计时。论文自己在 refs.bib:440-441 注释写了 "must not be framed as generic unoptimized FUSE"。配置诚实记录了（fuse.c:1127-1128），所以问题不是"没做"，而是这个配置选择 + 指标不对称会被审稿人当作挑选对自己有利的数字。

修法：FUSE 缓存三行矩阵（coherent/default/passthrough）+ libfuse3；补齐 open/access/exec/macro 在 FUSE 侧的对称计时。

## 三、要等实验数据才能定论的点

论文自标的 [Result] 占位和 supporting-only 状态本身不是担忧——那是作者已登记的待办，本文档不重复。下面只列"要等某个实验数据出来才能下最终结论"的观察。

- **RCU pass-only 的确切开销，以及是否超过 cache_ext 不超过 1.7% 的常态。** 要等第二节第 2 条的两条 pass-only 对照臂跑出来才能定量。在此之前第二节只维持"机制未被论文承认 + 阈值已移除"这一层。
- **物化参照行与多 cgroup 并行 epoch 格的结果。** 第一节第 1 条和第 2 条的修法都需要新增并运行实验，结果出来前无法判定 namei_ext 相对物化机制和 symlink 方案的实际区分度。
- **FUSE 缓存矩阵与对称指标。** 需要按第二节第 4 条的修法重跑才能给出可发表的 RQ2 数字；当前 21x/13x 仅供参考。
- **统计口径。** p50/p95/p99 + 95% bootstrap CI、微基准至少 30 次重复、宏基准至少 20 次重复、6.12 裸机、磁盘文件系统而非 tmpfs——这些是测量卫生要求，数字重跑后满足即可。
