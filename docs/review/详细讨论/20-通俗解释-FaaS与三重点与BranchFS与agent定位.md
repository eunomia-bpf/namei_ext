## 一、什么是 FaaS

FaaS 是 Function-as-a-Service（函数计算）的缩写，属于 serverless（无服务器计算）的一种。用户不租一台一直跑着的服务器，只上传一小段函数——比如一个 Python 处理函数 `handler.py`——平台在有请求到来时把它启动起来，跑完就回收。一台物理机上会同时跑许多不同函数的许多份实例。实际产品例子：AWS Lambda、Cloudflare Workers、Google Cloud Run；学术开源平台 OpenLambda（论文实验会用它当载体）。

FaaS 和这篇论文的关系在于：每个函数实例需要看到属于自己的一套文件——自己的代码加上自己的依赖库。今天平台给每个函数搭建这套文件视图的办法是叠一摞挂载。以研究系统 SOCK（ATC'18）为例，它的 `sock.go` 每建一个沙箱要做 8 次挂载表操作（bind mount 拼接目录再 chroot）。SOCK 论文自己明写：因为伸缩性问题，它放弃了 mount namespace（mount namespace 是 Linux 给进程各自一份独立挂载表的机制；原文："we uncover scalability bottlenecks in the network and mount namespaces and identify lighter-weight alternatives"）。这就是 namei_ext 的切入点：同一台机器上成百上千个函数视图，用挂载会在规模上碰到硬限制，用几条 eBPF map（eBPF 是 Linux 内核里的一套安全可编程框架，map 是其中用于存储键值数据的结构）条目则不会。

## 二、三个重点场景到底在做什么

### 重点 1：FaaS 高密度——论文的主要实验载体

**问题是什么：** 一台机器上要同时给 N 个不同的 serverless 函数各自一套文件视图，N 从 100 涨到 10,000。

1: 回答我为何要一台机器上要同时给 N 个不同的 serverless 函数各自一套文件视图? 除了有共享库之外, 出于什么样场景, 为何要共享一套文件视图, 在工业界真的会存在吗, 到底什么样scenario? 如果是共享库的话, 那应该也许有比我们更成熟的方案, 比如各种连接? 有比我们的论文中更好更成熟的解决方吗

### ⮑ 解答（1）：为何同机多视图、工业界是否存在、共享库有无更成熟方案

#### ① 一台机器为何要给 N 个不同的 serverless 函数各自一套文件视图

serverless（把函数交给平台按需执行、按调用计费的模式，又叫 FaaS，即 Function as a Service）的经济模型决定了：平台要把许多不同的函数——往往属于不同客户或不同任务——塞进同一台物理机运行，以提高利用率。一台机器只跑一个函数在成本上不可接受。

每个函数实例需要看到的"文件视图"由三部分组成：一份共享的只读底座（语言运行时、系统库，大家一样），它自己的代码（比如 handler.py，各实例各不同），以及它自己那组依赖的特定版本（各实例各不同）。因此 N 个函数就需要 N 个不同的文件视图。"共享库"只是这些视图中相同的那部分。密度的来源不是共享库，而是"把 N 个互不相同的函数打包到一台机器"这件事本身。

#### ② 工业界真存在这个场景吗？——存在，但必须诚实划清边界

**大众多租户生产级 serverless 不在"共享内核的文件视图层"做隔离。**这一点必须先承认，否则审稿人会当场驳回：

- **AWS Lambda**：每个函数实例跑在 Firecracker microVM（一种轻量虚拟机）里，隔离在虚拟机层，各自有独立的内核。文件视图问题被虚拟机边界消解了。
- **Cloudflare Workers**：用 V8 isolate（Chrome 的 JavaScript 引擎提供的隔离机制）做隔离，单进程可以跑"hundreds or thousands of Isolates"（原文措辞）。它在语言运行时和内存层隔离，根本不用文件系统做隔离，也没有挂载或独立文件视图的概念。

结论：namei_ext **不适用**大众多租户 serverless。它们要么在虚拟机层、要么在语言运行时层隔离，文件名解析层的视图供给不是它们的问题。

**真实存在、但很窄的场景**是"共享内核 + 高密度 + 信任边界内"这个交集：

- **刻意用共享内核换密度的轻量/研究型 serverless 平台**：SOCK（ATC'18）和 OpenLambda 整篇论文就是在推共享内核的进程沙箱密度（用 chroot + bind mount 而非虚拟机），明确说因为虚拟机太重才走这条路，并且因为扩展性问题放弃了 mount namespace（内核提供的让每个进程看到不同挂载树的机制）。这类平台有具名的学术证据，但不是 AWS/Azure 的主流做法。
- **单租户批量扇出**：一个客户在自己的机器上跑大量相似但各自不同的任务——强化学习 rollout 集群、批量评估、CI runner 池。这些沙箱之间互相信任（同一个租户），所以共享一个内核可以接受，追求的是密度而非跨租户安全隔离。

诚实地说：这个场景真实但窄。它不是大众 Lambda 式多租户 serverless，而是上述特定交集。这也是论文把 FaaS 密度的主运行点（有 SOCK、systemd issue #31137、Red Hat 报告的 19 倍回归等具名证据支撑的规模）钉在 10^3 量级，并把 10^4 量级标为"作者提出的新运行点"的原因——10^4 是论文对趋势的外推，没有具名部署者在生产中报告过那个密度。

#### ③ 共享库这块确有更成熟的方案，namei_ext 不在这上面赢

这个质疑完全正确。如果目标是"让很多视图共享同一批库而不重复占用内存和磁盘"，今天有成熟且好用的方案，namei_ext 不比它们强：

- **共享只读 base 层 + overlayfs**（一种联合文件系统，把只读底层和可写上层叠在一起呈现为一个目录）：底座层共享，page cache（内核把文件内容缓存在内存里，多个进程读同一文件时共用同一份内存缓存）也共享。这是容器生态的标准做法。
- **内容寻址存储**（按文件内容的哈希值存储和去重，内容相同的文件只存一份）：composefs、EROFS、Nix store、pnpm 的硬链接存储都属于这一类，天然共享 page cache。
- **符号链接/硬链接 + 动态链接器配置**：通过 symlink 或 hardlink 指向共享的 .so 文件，配合 LD_LIBRARY_PATH（告诉动态链接器去哪找共享库的环境变量）让每个程序加载同一份库。

对"共享库/共享内容"这个具体目标，这些方案成熟、够用。namei_ext 不主张在内容去重或 page cache 共享上赢过它们，论文也不拿"共享库"当卖点。

#### ④ namei_ext 到底加了什么，有没有比它更好的方案

namei_ext 的差异不在共享内容，而在**每个视图的供给成本随视图数量增长时的表现**。上面列举的每种成熟方案，要给每个视图付出的代价是：overlayfs / bind mount / composefs 镜像每个视图至少需要一次挂载操作，挂载记录进入内核的全局挂载表；symlink / hardlink 农场每个视图需要物化一批链接和 inode（文件系统用来记录文件元信息的数据结构）。

当视图数量到 10^4 量级、且视图频繁创建销毁（高换手）时：挂载方案撞上挂载表规模膨胀和 namespace_sem（内核里保护挂载表的全局锁）的瓶颈；物化方案要付 10^4 份元数据和 inode 的成本。namei_ext 把每个视图表达成几条 map 条目（路径重写规则），不做挂载、不物化文件，所以每视图的供给成本接近零、挂载表不增长。共享的库内容仍然靠底层文件系统的 page cache 共享，这一点 namei_ext 没有改变也不需要改变。

**有没有比 namei_ext 更好的方案？答案取决于密度：**

- **低密度（几十个视图）**：overlayfs + 共享 base 层更成熟、更好。namei_ext 在这个区间没有优势，论文应当诚实说明"这时不需要用 namei_ext"。
- **高密度 + 高换手 + 共享内核（10^4 量级的短命视图）**：挂载和物化方案都会撞墙，map 条目方案不撞——这是唯一的缺口。这些成熟方案正是论文的对比基线，密度实验（图 4）整件事就是画出它们在什么密度开始撞墙、namei_ext 从哪个密度之后才开始赢。

所以"有没有比 namei_ext 更成熟的方案"的诚实答案是：低密度有，而且更好（overlayfs、内容寻址存储、动态链接）；高密度且共享内核这个具体区间没有（那正是缺口所在）。但这个高密度区间本身有一部分是外推——10^4 量级没有具名的生产部署者，是论文提出的新运行点。

#### ⑤ 一句话诚实结论

FaaS 密度是真实需求但适用范围很窄——只在"共享内核 + 高密度 + 信任边界内"成立，大众多租户 serverless（Lambda 用 Firecracker microVM、Workers 用 V8 isolate）在虚拟机或运行时层隔离，用不上 namei_ext。共享库这块成熟方案（overlayfs、内容寻址存储、动态链接）更好，namei_ext 不碰也不该碰。namei_ext 唯一的差异是"每视图供给在挂载方案会撞墙的高密度下仍接近零成本"，而这个高密度区间有一部分是论文自己提出的新运行点、缺乏具名部署证据。论文不应把 FaaS 或共享库当作核心卖点；FaaS 只是密度测量的一个载体，其中 10^3 量级有 SOCK（ATC'18）、systemd issue #31137、Red Hat 报告的 19 倍回归等具名证据，10^4 量级是作者的外推。

### ⮑ 解答（3）：SOCK 是不是被我们做成增量、我们到底想做什么、能否取代 microVM

**SOCK（ATC'18）做了什么——三句话。** SOCK（威斯康辛大学 Oakes 等）是一个 serverless（把用户函数按请求启动、用完即弃的云运行模式）沙箱系统，目标是让函数冷启动快、单机密度高。它靠两件事做到这一点：第一，每个沙箱用轻量的 bind mount（把已有目录树挂到另一个位置，让进程看到它需要的文件）加 chroot（把进程的根目录限定在某个子树下）拼出文件视图，而不是给每个沙箱分一个完整的 mount namespace（内核为每个进程维护的独立挂载表——创建和销毁开销随挂载数线性增长），因为 mount namespace 在高密度下伸缩不了。第二，它用 Zygote（预热母进程）方案：预先导入常用的 Python 包，新函数从这个热进程 fork 出来，省掉重复导入。SOCK 自己的实测就证明了"基于挂载的视图供给在高密度下撞墙"——这正是它放弃 mount namespace 的原因。

**"看起来像 SOCK 增量"——这个担心完全对。** 如果论文以"FaaS 场景、比 SOCK 更快的沙箱"开场，任何审稿人都会读成 SOCK 的增量工作。这恰恰是三轮红蓝对抗把 FaaS 降级为"只当密度测量的载体、不当论文卖点"的原因。区别在于：SOCK 的贡献是一个 serverless 沙箱系统本身；namei_ext 的贡献是一个内核机制——文件名解析路径上的可编程扩展点。论文里用 SOCK/OpenLambda 只做两件事：（a）当密度测量的负载来源；（b）当动机证据（SOCK 自己的数据已经证明挂载供给伸缩不了，我们引用它说明问题的存在，不和它竞争谁的沙箱更快）。论文的身份是"内核机制 + 挂载做不到的结构性能力"，不是"打赢 SOCK"。所以不是增量——但前提是论文里绝不能把 FaaS 当主打场景。

**到底想做什么——一段话。** 想在 Linux 文件名解析路径上加一个可编程决策点，类似 sched_ext（Linux 6.12 引入的机制，让调度策略可以用 eBPF 程序替换，不用改内核源码重新编译）给调度器做的事：让"这个路径名，对这个 cgroup（内核用来分组管理进程的机制，例如把同一个容器里的所有进程归为一组）或 task，解析到哪个后端文件对象"变成一条可编程策略，而不是每个视图都要一次挂载或一个新文件系统。论文要证明这个机制能做到挂载在结构上做不到的三件事——同一进程的不同线程看到不同文件视图、按 cgroup 原子地切换一批路径指向的版本、在一台机器上维护上万个免挂载视图——并测量它引入的开销。这是一个内核可扩展性工作，不是一个 serverless 系统。

**能否扩展 namei_ext 取代 microVM 而不损害安全——不能，这一点必须说清楚。** microVM（以 Firecracker 为代表）存在的目的是在互不信任的租户之间做安全隔离：每个函数运行在一个独立的 guest 内核（虚拟机里自己的操作系统内核）之上，由硬件虚拟化提供边界，一个被攻破的函数碰不到别的租户或宿主机。namei_ext 是共享同一个宿主内核上的名字解析机制：它改变一个进程看到哪些文件，但不提供任何安全边界——所有函数共享一个内核，一个内核漏洞就波及所有人；fd 传递（一个进程通过 Unix socket 把已打开的文件描述符发给另一个进程，绕过路径名解析）、/proc 访问、namespace 逃逸都是攻击面。三轮红蓝对抗一致结论：安全隔离这条轴让给 Landlock、gVisor、microVM，namei_ext 明确不做。所以 namei_ext 不能取代 microVM 做互不信任租户的隔离——那是另一个问题（安全隔离 vs. 视图供给）。namei_ext 能替代挂载或 microVM 的地方只限于信任边界以内的场景：一个租户跑自己的函数、强化学习的 rollout 农场、CI runner 池——这些场景本来就不需要跨租户安全隔离，microVM 是不必要的重型方案，namei_ext 在这里提供视图供给而不撞挂载的伸缩瓶颈。但"取代 microVM 而不损害安全"——对互不信任租户做不到，而且如果论文里暗示能做到，安全方向的审稿人一句"共享内核 + fd 传递 + /proc"就可以毙掉。诚实的说法是：namei_ext 和 microVM 解决不同的问题；namei_ext 只适用于本来就不需要 microVM 那层安全隔离的场景。

**实验怎么跑：** 载体是 OpenLambda。把每个函数原来的"叠一摞挂载拼出目录"换成"注册几条 namei_ext map 条目"（把该函数看到的路径重定向到它自己的代码和依赖目录）。随 N 增长测四条曲线：(a) 每个视图占多少内核内存（目标约 12–30 KB）；(b) 挂载表条数（namei_ext 方案零增长，对比挂载方案 +N 行）；(c) 建立视图的吞吐量；(d) 保护全局挂载表的那把大锁 namespace_sem（这是 Linux 内核中管理挂载命名空间的一把信号量，所有挂载操作都要先拿到它，因此高密度场景下成为串行瓶颈）的争用程度随 N 如何恶化。

**和谁比：** BranchFS 原样 FUSE（FUSE 是用户态文件系统框架，文件操作需要从内核态弹回用户态处理）、FUSE 满配 passthrough（passthrough 指 FUSE 把实际读写直接转发给底层文件系统，跳过用户态来回）、overlayfs 每视图一挂载、mount namespace 变体、symlink 森林、cp/git worktree、btrfs 快照，共七种基线。

**优势在哪：** 挂载和 overlayfs 每建一个视图要一次挂载操作，几千个视图时挂载表的规模限制就会成为瓶颈——SOCK 的放弃是一个证据，systemd issue #31137 记录了数千挂载拖垮整机的案例，Red Hat 实测某容器 run + teardown 耗时 0.5844 s 而精简方案只要 0.0305 s（约 19 倍差距）。namei_ext 用 map 条目实现视图，零挂载、内存接近零。

**诚实边界：** 论文的主运行点定在今天真实的量级——约 10^3 个视图、每秒约 150 个的供给速率，对标 Firecracker（AWS 的轻量虚拟机管理器）每机 150 microVM/s。超过 150/s 以及 10^4 视图这两个数据点今天没有具名的部署方，论文里明写"这是作者提出的新运行点"。

### 重点 2：证书/配置的原子轮换——在线动态能力的主要实验

**问题是什么：** 一个正在运行的服务（比如 nginx、Kubernetes pod）要把它的 TLS 证书加私钥、或者配置文件换成新版本，不能重启进程，而且换的瞬间必须是原子的——服务绝不能读到"新证书配旧私钥"这种半新半旧的组合（否则 TLS 握手直接失败）。今天 Kubernetes 用一个叫 AtomicWriter 的组件实现这件事：kubelet（Kubernetes 在每个节点上管理容器生命周期的进程）在 pod 目录下维护一个叫 `..data` 的符号链接，轮换时原子交换这个链接让它指向新版本的目录。Envoy（一种高性能代理服务器）官方也推荐符号链接轮换。SPIRE（一个零信任身份管理系统）默认证书每 1 小时轮换一次。

**实验怎么跑：** 给每个 cgroup（cgroup 是 Linux 按进程组分配和控制资源的机制，比如一个容器的所有进程属于同一个 cgroup）一个版本号。翻到新版本 = 对这一组视图做一次 8 字节的原子 map 写。用真实软件做验证：PHP-FPM（一种按"一个请求一个进程"模式工作的 PHP 运行方式）跑一个需要读 manifest 加 N 个 include 文件的页面，发 ≥10^5 个请求，证明混到半新半旧的请求数恒为 0。对照符号链接方案在高频轮换下的行为，它会落进一个可能读到混合版本的概率窗口。再加上 per-task 钉定：一个正在处理的请求全程锁定在同一个版本，中途不切换。

**和谁比：** 原子换符号链接、MOVE_MOUNT_BENEATH（Linux 6.x 新增的一种在已有挂载下方插入新挂载的操作）、蓝绿重启（停旧服务启新服务）、overlayfs，以及给不修改代码的应用用 openat(dirfd)（一种通过目录文件描述符打开文件的系统调用，可以绕过路径解析）这两种方案。

**优势在哪，以及诚实收缩的边界：** 论文不主张"对不修改代码的应用做到配对错配在结构上为零"——这条结论在第二轮内审时被精确计算推翻了（用超时钉定的方案大约每秒会出现几次错配），已经放弃。保留下来的主张是：在真实软件（PHP-FPM 这种一请求一进程的架构）上零混合版本 + 按 cgroup 子集翻转版本 + per-task 钉定——这几件事挂载方案做不到，或者需要付全局锁的代价。\

### ⮑ 解答（2）：把证书轮换那句话拆成大白话

原句确实塞了太多没解释的词。逐个拆开。

**背景：这句话在说什么场景？** TLS（Transport Layer Security，给网络连接加密的协议）要求服务器持有两个文件：一个证书（公开的身份凭证，通常叫 `cert.pem`）和一个私钥（与它配对的秘密，通常叫 `key.pem`）。两者必须来自同一次签发、配套使用；服务器拿新证书配旧私钥，TLS 握手直接失败。轮换证书时，磁盘上的 `cert.pem` 和 `key.pem` 不可能在同一瞬间同时变新——总有一小段时间一个新、一个旧。应用恰好在这个间隙读文件，就读到"新证书 + 旧私钥"这种不匹配的组合。

**原句里的词逐个解释：** **配对错配** = 读到"新证书配旧私钥"这种不匹配的一对，握手失败。**对不修改代码的应用** = 改不了源码的黑盒程序，它自己决定何时读 `cert.pem`、何时读 `key.pem`，内核无法知道"读这一对"从哪开始到哪结束。**结构上为零** = 从机制设计上保证这种错配永远不会发生，而非"通常不发生偶尔漏一个"。**超时钉定** = 一种实现思路：应用读了旧版本后，把旧版本继续对它可见一段固定超时时间，再切新版。**第二轮内审 / 精确计算推翻** = 第二轮红蓝对抗（项目内部找漏洞的评审）里算出：轮换很频繁时，固定超时仍会漏过约每秒 5 次错配；缩短超时又与"新版本要在 10ms 内可见"冲突，两头够不着，所以"对黑盒应用保证错配永远为零"这条强主张放弃了。

**放弃了什么、保留了什么：** 放弃"对任意黑盒应用错配永远为零"；保留"对有一请求一进程结构的应用，按请求钉住版本、一个请求内零混合"。

**"真实软件（PHP-FPM 一请求一进程）"是什么：** "真实软件"这词含糊，实际指"有一请求一进程结构的应用"。PHP-FPM 运行 PHP 时，每个 HTTP 请求由一个单独 worker 进程处理，处理完就结束——于是"一个请求"天然是一个边界，这个请求读到的所有文件属于同一批。把文件视图版本号按这个请求（即按进程/task）钉住，一个请求全程只看同一版本，读到的证书和私钥必然同版本。有这种结构的：PHP-FPM、Apache prefork、gunicorn sync；没有干净边界的：一个进程里事件循环穿插处理多请求的服务器（nginx worker、Node.js），窄主张不覆盖它们。

### ⮑ 解答（2 追问）：rename 能不能做证书原子轮换、我们是不是在做更好的 rename

**单次全局替换——rename 已经解决，而且是生产在用的成熟方案。** rename(2)（Linux 系统调用，把一个文件名原子地换成另一个）对单个路径名本身就是原子的：把新文件 rename 覆盖旧文件，读的人要么看到完整的旧文件、要么完整的新文件，不会看到写了一半的内容。这正是 Kubernetes 的 kubelet 里 AtomicWriter（负责把 ConfigMap/Secret 写到 Pod 可见目录的组件）用的办法——把 `..data` 这个符号链接原子地 rename 指向新版本目录，一次 rename 把整棵目录树切过去。所以对"原子地给所有人换掉一个东西"这个需求，rename() 加符号链接已经够了，是成熟方案，我们在这个场景上不赢，也不主张赢。

**rename 结构上做不到的两件事——这才是 namei_ext 加值的地方。**

第一，一个读取会话内跨多次读的一致性。rename 的原子性是"每次系统调用"级的，不是"一个读取者的整个请求"级的。一个应用在时刻 t1 读 cert.pem、时刻 t2 读 key.pem，如果 rename 发生在 t1 和 t2 之间，两次读各自看到的都是完整文件，但这一对证书和私钥仍然来自不同版本——不匹配。`..data` 符号链接方案把两个文件放在同一个目录、原子换目录，但只有当应用每次都重新经过 `..data` 解析、且两次读之间链接没被切换时才一致；如果应用缓存了解析结果，或者两次读恰好跨越了切换时刻，仍然会错配——这就是 2 提到的残余窗口。namei_ext 用 epoch（一个递增的版本号，标记"这组路径名当前指向第几代后端文件"）把整个请求钉在同一个版本上：请求开始时记住当前 epoch，此后该请求内的所有路径名解析都指向同一代后端，中途发生的版本切换对它不可见，直到请求结束。这不是 rename 能给的——rename 没有"按请求钉住"的概念。

第二，按 cgroup 或按 task 选择性分版本。rename 是把磁盘上的名字对所有人换掉，做不到"cgroup A 看版本 N、cgroup B 看版本 N+1"。namei_ext 的策略是按 cgroup/task 决定解析结果的，不同的进程组可以同时看到同一路径名的不同版本，而磁盘上只存一份、不需要额外挂载。

**所以 namei_ext 不是"更好的 rename"，是另一种原语。** rename() 原子地换磁盘上的名字，对所有人生效；namei_ext 的 epoch 原子地改"一个名字解析到哪个后端"，按 cgroup 区分、按 task 钉住，而且不碰磁盘上的目录项。对"给所有人原子换一次证书目录"这种最简单的情形，rename() 加符号链接已经原子、更简单（不用打内核补丁），没有理由用 namei_ext。namei_ext 的价值只在 rename 结构上给不了的两块：按请求或会话保持一致（不错配），以及按 cgroup/task 分版本。

### 重点 3：策略多样性六族表——证明这是一个通用的可编程扩展点

这不是一个单独的端到端实验，而是"扩展点论文"的核心举证方式：证明同一个钩子（namei_ext 在内核路径解析流程中插入的可编程回调点）能表达至少 6 种不同的策略，每种几百行 eBPF 代码，每种都替代一个今天需要各自写专门内核代码的机制。对标的范例是 cache_ext（SOSP'25），那篇论文用一个钩子写了 8 种页缓存驱逐策略来证明扩展点的通用性。

两行重点策略：

(a) **casefold**（大小写不敏感的文件名匹配，给 Wine/Proton 跑 Windows 游戏用）——内核社区花了 6.5 年在 4 个文件系统里逐个硬编码这件事：ext4 自内核 5.2 起、f2fs 自 5.4、tmpfs 自 6.13、bcachefs 自 6.15，overlayfs 自 6.18；而 btrfs 和 XFS 至今没有支持。namei_ext 用一条 eBPF 策略一次表达，不需要改每个文件系统的代码。

(b) **CI 缓存的按视图撤回**：在持续集成的编译缓存里，一个条目被怀疑污染了（比如编译产物和源码不一致），需要标记成"在这个视图里当它不存在"而不是从全局缓存里删除。

底线要求：表里至少 6 行真实跑通。只跑 2 行会被审稿人以"覆盖面太窄"为理由拒稿（这是内审总结出的教训）。

## 三、是否不该提 BranchFS、转而提别的

**结论：BranchFS 仍然会出现在论文里，但只作为第三人称引用的一个 FUSE 基线，以及设计空间讨论里的一个对比点。它不再作为论文的应用场景，也不再作为 namei_ext 接入的目标系统。** 承担主要实验证明任务的场景转成了 FaaS（重点 1）和证书/配置轮换（重点 2），这两个都是非本组、今天有具名使用者的系统。这正是之前几轮讨论的落点。

不拿 BranchFS 当应用场景或接入目标的三条原因（均已核实原文 arXiv 2602.08199《Fork, Explore, Commit》）：

1. **它是本组自己的工作**（作者 Cong Wang @ Multikernel 加 Yusheng Zheng = 郑昱笙 @ UC Santa Cruz）。双盲投稿下拿自己组的系统当主打场景，既要全文第三人称回避，又容易被审稿人读成"这只是你们自己上一篇工作的增量改进"。

2. **它刻意留在 FUSE、不进入内核**——原文明确说保持 FUSE 是为了免 root、不写文件系统专属代码。它提议的 `branch()` 系统调用走三个通用 ioctl，不涉及路径解析。而 namei_ext 要打内核补丁、要 root、要加载 BPF 程序——把 namei_ext 接进 BranchFS 等于让 BranchFS 放弃自己明写的核心设计选择，审稿人读了原文就会发现矛盾。

3. **它的 passthrough 模式已经拿到 82% 的原生吞吐**（常规 FUSE 才 19%），"BranchFS 因为 FUSE 开销慢所以需要 namei_ext" 这个动机大半被它自己的优化削掉了。

BranchFS 在论文里的确切角色：密度实验（图 4）里的一个 FUSE 基线档位（建分支 292–317 µs、读吞吐 19%/82% 两档）+ 相关工作里"名字解析层没有原生可编程扩展点"这个设计空间讨论中的一个对比。策略表里原来写的接入 BranchFS 那一行改名为"合成分支策略"，被替代的机制写成通用工具（git worktree / overlayfs），不写 BranchFS。

## 四、agent 场景是被排除了吗——以及之前说的"顺风逆风"到底具体指什么

**agent 场景没有被排除。** 它从"独占一组主实验结果的旗舰应用"降级为两个具体角色：

- 在 Section 2（"为什么是现在"）里作为前瞻性动机写一段话，排在第二位，不领开场。
- 作为密度实验的负载发生器：用 agent 并行分支的轨迹造出真实的文件访问负载，喂给密度实验（图 4）。

它不占独立的实验名额，论文不主张"加速了任何 agent 系统"。这是降级而不是删除。

**现在把之前写的"顺风逆风"用大白话讲清楚——那个说法确实含糊，以下是它具体指的意思：**

一篇论文如果用"某个未来趋势"当动机开场，这个趋势在现实里的代表性例子建在系统栈的哪一层，直接决定它帮不帮得上论文的论证。

先看正面例子：ghOSt 和 Bento 这两篇论文。ghOSt 开场引的趋势是"定制调度策略越来越有价值"，它的现实例子（Google 生产环境中各种调度策略需求）正好发生在调度器层——而 ghOSt 的机制也在调度器层。Bento 引的趋势是"新存储器件带来的压力"，现实例子也在存储层——而 Bento 动手的地方也是存储层。趋势的现实例子和论文机制在同一层，读者顺着趋势往下想，正好想到论文要解决的地方。

再看 namei_ext 和 agent 趋势的情况：agent 大规模并行分支这个趋势本身是真实的——E2B 一年从 4 万涨到 1500 万次/月（375 倍）、Morph Infinibranch 做到 <250 ms 建分支、Modal 也在做类似的事。但是，这些最有力的现实例子全部建在虚拟机 / microVM 层：每个分支是一个独立的轻量虚拟机或虚拟机快照 fork，各自有一个独立内核。而 namei_ext 工作在完全不同的一层：共享同一个内核的文件名解析层。

所以如果拿 agent 分支趋势当论文的开场动机，审稿人的第一反应会是"对，E2B 和 Morph 已经用虚拟机 fork 做了这件事"——这会把审稿人的注意力引向一个和 namei_ext 架构相反的方案（各自独立内核 vs. 共享内核），而不是引向 namei_ext 的机制。这不是"这个趋势可不可信"的问题（趋势本身可信），是"这个趋势在现实中的实现层和 namei_ext 不在同一层，拿它开场会先把审稿人的思路带到别人的方案那边去"。

作为对比，论文今天就有三条直接指向文件名解析层的事实，用它们开场才准确：

1. 【2026-07-10 更正:此条从"做不到"降级为"代价高"——主 agent 亲手在 5.15 内核实测:root 下、或无特权但在多线程化之前先建 user namespace 的情况下,同一进程的两个线程可以各自 unshare(CLONE_NEWNS) 得到不同挂载视图;只有"无特权、已多线程、未提前安排"的进程才吃 EPERM。】准确的说法是:给线程各自视图,挂载路线做得到但每线程要付一份完整挂载表复制(O(挂载数))+ 全局锁 namespace_sem 排队 + 该线程永久失去与兄弟线程共享的当前目录(unshare 会解除 CLONE_FS 共享)+ 必须在多线程化之前安排;namei_ext 的差异是一条表项、不复制挂载表、保住共享当前目录、随时可绑。这是成本与粒度优势,不是能力独占。

2. 挂载在约 10^3 量级今天就出现伸缩性问题（SOCK 的放弃是一个证据、systemd #31137 是另一个、Red Hat 19 倍差距是第三个）。

3. 挂载的建立串行在一把全局锁 `namespace_sem` 上，复制挂载表是 O(挂载数)（v6.12 源码可验证）。

所以 agent 趋势放到 Section 2 当"为什么现在越来越重要"的第二段理由——它的需求侧证据是真的：DeltaBox 每任务 2311 次恢复、SWE-Search 的树搜索带回滚。而 Section 1 开场用上面三条今天就成立的结构性事实。这样即使 agent 趋势将来没有走到"共享内核上万级视图"这一步，论文靠这三条事实照样站得住。

## 五、一句话总纲

论文的基础押在三条今天就成立的能力缺口上（线程间不同视图做不到、万级视图的零挂载表增长做不到、按 cgroup 或 task 的原子版本切换做不到），主要实验落在两个今天有具名使用者的场景（FaaS 密度、证书轮换），再加上一个通用可编程扩展点的机制贡献（一次内核补丁加全 eBPF 策略，做法对标 sched_ext / cache_ext 这一类扩展点论文）。BranchFS 只作为被引用的基线。agent 作为前瞻动机和负载来源出现，不作为主实验结果。
