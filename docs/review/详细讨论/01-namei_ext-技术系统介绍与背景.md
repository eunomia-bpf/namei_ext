## 0. 一句话定位

> **namei_ext 把 `sched_ext` 的设计范式搬到 VFS 路径解析层:内核继续拥有路径行走、dentry/inode、权限检查、mount 遍历与下层文件系统的全部机制和对象生命周期,而一个受 verifier 约束的 eBPF 程序只负责"这个路径组件应该怎么解析"这一窄决策。它既不是一个新文件系统,也不是访问控制框架。**

核心论文 claim(作者原文):

> *namei_ext shows that production path-resolution customization can be expressed as a narrow in-kernel BPF policy over VFS name resolution, without turning BPF into a filesystem implementation.*

## 1. 背景:路径解析定制的需求,与现有隔离/沙箱手段全景

要理解 namei_ext 的贡献,必须先理解它想填补的空白。本节系统梳理 Linux 上"让同一个路径名在不同上下文下指向不同后端对象"这一需求,以及现有各类隔离/视图/沙箱手段的能力边界——这正是审稿人评估"是否真有空白"时会逐一对照的清单。

### 1.1 共性需求:per-workload 的"目录树视图"

大量现代系统并不需要一个自定义文件系统,它们需要的是**对一棵已有目录树的不同视图**:
**问:** 为何说大量现代系统并不需要一个自定义文件系统,而是需要对一棵已有目录树的不同视图?尤其在 LLM/agent 语境下,这个问题该如何精确定义?

**⮑ 解答:**

**先分清两个概念:"文件系统(filesystem)" vs "视图(view)"。**

- 一个**文件系统**要负责把字节真正存到磁盘上的一整套机制:块分配、inode 管理、目录结构、数据读写、崩溃一致性(日志)、缓存、`fsck` 修复…… ext4/XFS/Btrfs 已经把这件事打磨了几十年、极其可靠。
- 而上面列的这些场景,**没有一个想改变"字节怎么存"**。源码、依赖包、构建产物这些字节,已经好端端地躺在 ext4 上。它们想改变的只有一件事:**"`/x/tool` 这个名字,对当前这个 workload,到底应该指向哪个已经存在的对象。"**

所以"做一个新文件系统"是杀鸡用牛刀且自找麻烦:① 工程量和风险巨大(等于重写一个存储引擎);② 会**丢掉与下层共享 page cache 的能力**(同一份只读基底被 N 个沙箱共享时,本可只在内存里缓存一份);③ 会把**本可共享的字节复制 N 份**。

一句类比:这些场景要的不是"盖一栋新楼"(新文件系统),而是"给同一栋楼里的不同住户发不同的门牌指路图"(视图)——楼和房间(字节)是同一套,只是每个人按图找到的"门"不同。

**LLM / agent 语境下,这个问题怎么精确定义?**

coding agent 的典型用法是**大规模、短命、并行**地开沙箱:每个任务一个、甚至每个候选 patch / 每条探索分支一个,而它们几乎都**从同一个 pinned 的 repo 状态 + 同一套巨大的依赖树出发**(`node_modules`、`.venv`、Go module cache 动辄上 GB)。于是诉求变成:

- 所有沙箱**共享只读基底**,不要每个都拷一份 GB 级的树;
- 每个沙箱有**自己的可写工作区**,以及**自己 pin 的版本/代际(generation)**;
- **fork 出新沙箱、rollback 到上一个 checkpoint,代价应是 O(元数据) 而非 O(复制整棵树)**;
- 沙箱里 `open` 能打开的文件,和 `ls` 列出来的名字必须**一致**(否则 agent 的工具会困惑)。

写成一句可被实验检验的定义:

> 给定一个共享内容存储和一棵 pinned base tree,为每个 agent 任务呈现一个**工作区视图**,使得:(a) 工作区文件按代隔离、可回滚;(b) deps/cache 路径解析到"共享后端"或"该代专属后端";(c) fork/rollback 不复制大树;(d) lookup 与 readdir 结果保持一致。

这就是 namei_ext 想用"按工作负载可编程的路径解析"去回答的问题——**它要替换/简化的是"如何廉价地为每个 agent 沙箱生成并切换这种视图",而不是"字节存哪"。**(这个动机是否真成立、是否已被 overlay/btrfs 快照等廉价满足,正是后面对抗性审查的核心,见本节末第 8 条解答。)

- **构建系统(Bazel/Ninja/BuildKit)**:为每个 build action 构造一个隔离的、只暴露声明依赖的文件视图(sandbox)。
- **容器与 serverless runtime**:把若干 layer 组装成一个 root filesystem 视图,并为每个实例提供。
- **包管理器 / 语言环境(npm/pnpm、venv、Nix)**:把共享 store 映射成项目私有的目录布局。
- **兼容层 / 路径迁移**:把 legacy 固定路径重定向到新的存储布局。
- **(论文转向后的主场景)AI coding agent 沙箱**:从某个 pinned repo 状态 fork 出大量工作区、做 checkpoint/rollback、为 deps/cache 提供按代(generation)隔离的视图。

**问:** 上述五类场景(构建系统沙箱、容器/serverless rootfs、包/语言环境、兼容层/路径迁移、AI agent 沙箱)各自的架构是什么?各有什么优缺点?能否用流程图分别展示?

**⮑ 解答:** 这五类场景**本质是同一个模板的不同实例**:

```
共享后端树(只读基底) ──▶ [物化/重映射机制] ──▶ 每-workload 视图 ──▶ 运行 ──▶ (可丢弃/回滚)
```

区别只在中间那个方框用什么机制、以及运行完是"丢弃"还是"保留/回滚"。逐个画:

**(1) 构建系统沙箱(Bazel / BuildKit)**
```
声明的输入集(BUILD 文件)
      │  每个 action 只取自己声明的那部分
      ▼
 [sandbox builder]  ← symlink forest / bind / overlay / FUSE 任选其一“物化”
      ▼
 该 action 的隔离视图(只看得到声明依赖)
      ▼
 编译器在视图里跑 ──▶ 产物 + action cache key
```
优点:hermetic(密闭可复现)、可缓存、可远程执行。
缺点:**每个 action 都要物化一次视图**;action 多而小时 setup 延迟主导;symlink forest 产生海量 inode。

**(2) 容器 / serverless rootfs**
```
镜像层 L0(base)+ L1 + … + Ln(app)
      │  联合挂载
      ▼
 [OverlayFS] lower=L0..Ln(只读) upper=容器可写层
      ▼
 merged rootfs ──▶ 容器进程(首次写触发 copy-up 到 upper)
```
优点:层在多容器间共享、启动较快、生态成熟。
缺点:每实例一次 overlay 挂载;冷启动开销;首写 copy-up。

**(3) 包 / 语言环境(npm/pnpm/venv/Nix)**
```
共享 store(~/.pnpm-store、/nix/store —— 内容寻址、去重)
      │  link / copy / project
      ▼
 项目私有布局(node_modules/ 形态)
      ▼
 应用按它期望的路径 import
```
优点:存储去重(一份包多项目共享)。
缺点:仍要为每个项目物化一套链接/副本;版本漂移;符号链接语义坑(见下文 symlink 解答)。

**(4) 兼容层 / 路径迁移**
```
应用写死的 legacy 路径  /usr/bin/foo
      │  wrapper 改写 / symlink / overlay redirect
      ▼
 新存储布局  /opt/foo-2.1/bin/foo
```
优点:不改应用就能搬动后端布局。
缺点:每条路径都要维护映射;wrapper/symlink 脆弱、易泄露真实路径。

**(5) AI agent 沙箱生命周期(论文主场景)**
```
pinned base repo + 巨大 deps 树(只读)
      │  per-task fork 出视图
      ▼
 [fork 机制] ← copy-tree / git-worktree / overlay / btrfs 快照 / namei_ext
      ▼
 任务工作区 ──▶ agent edit/test 循环 ──▶ checkpoint
      │                                    │
      └──────────── rollback/restore ◀─────┘  (回到上一代,重放后缀命令)
      ▼
 大规模 fanout:同时 10²~10⁴ 个这样的视图
```
优点:支持并行探索、失败回滚、可复现重放。
缺点:**fork/rollback 代价 + deps 重复**是瓶颈;fanout 规模一上来,"每个沙箱物化一套"的存储与 setup 成本爆炸——这正是 namei_ext 主打的痛点。

**横向一句话**:前四类是工业界已在用、机制各有成熟方案的场景;第五类(agent)是论文押注的新场景,赌"在超大规模 ephemeral fanout 下,把视图生成从'物化一棵树'降为'挂一段 BPF 决策'能省下数量级的 setup/存储成本"。这个赌注成不成立,取决于能否找到一个**只有它能廉价做、overlay/快照都做不好的杀手操作**(见末尾第 8 条解答与后续设计方向文档)。

这些场景的共同点是:**需要的是可编程、可组合、按工作负载定制的路径解析,而不是一个新的文件系统实现**。

### 1.2 现有手段分层全景

下面按"机制处于哪一层 / 能力 / 代价 / 粒度"梳理现有手段。namei_ext 的设计论证就建立在指出这张表里没有一格同时满足"可编程 + 内核态数据路径 + 细粒度按 workload"。

#### (A) 进程级命名空间隔离

- **`chroot` / `pivot_root`**:改变进程看到的根目录。粒度粗(整棵子树),静态,且 `chroot` 本身不是安全边界。
- **问:** chroot 命令的作用和使用场景是什么?

  **⮑ 解答:** `chroot`(change root)把某个进程(及其子进程)看到的"根目录 `/`"换成你指定的某个子目录。
  - **命令形态**:`chroot /srv/newroot /bin/bash` —— 之后这个 bash 眼里的 `/` 就是宿主的 `/srv/newroot`,它再也看不到外面的 `/etc`、`/home`(只能看到 `/srv/newroot/etc`……)。
  - **典型用途**:① **系统救援**——从 U 盘 live 系统启动后 `chroot` 进坏掉的硬盘系统去修 grub、改密码、装包;② 在一个发行版里跑另一个发行版的用户态;③ 构建/打包的干净环境(debootstrap、早期 build chroot);④ 历史上的 BSD `jail`、早期容器雏形。
  - **关键警告:`chroot` 不是安全边界**。有 root 权限的进程能"越狱"出去(经典手法:先 `open()` 一个 chroot 外目录拿到 fd,再 `chroot` 进子目录,然后 `fchdir` 到那个 fd、连续 `cd ..` 走回真正的 `/`)。所以现代容器**不**靠 `chroot` 做隔离,而是用 **mount namespace + `pivot_root`**(更彻底地换掉挂载视图并卸载旧根)。
  - **与 namei_ext 的关系**:`chroot` 是"换根"——粒度是**整棵子树、静态、二选一**;namei_ext 改的是"单个名字解析到哪",粒度细得多,且可按 workload 编程。
- 
- **mount namespaces(`CLONE_NEWNS`)**:每个 namespace 一套挂载视图,是容器文件系统隔离的基础。强大但**重**:重配置要操作挂载点、传播属性(shared/private/slave),大规模 fanout(成千上万个短命沙箱)时挂载/卸载与命名空间开销显著;且视图由"挂载了什么"决定,无法表达"按组件名做条件重写"这种细粒度逻辑。
- **idmapped mounts(Christian Brauner)**:让同一棵树在不同挂载点呈现不同的 uid/gid 映射。是"同一对象、不同视图"的内核先例,但只重写**身份(ownership)**,不重写**名字解析到哪个对象**。

#### (B) 内核态文件树组合(固定策略,原生数据路径)

- **bind mount(`MS_BIND`)**:把一棵子树挂到另一处。快、保留原生 VFS 行为,但策略固定、粒度是挂载点级别;每个别名一个挂载点,大规模时挂载表膨胀。
**问:** bind mount 命令的作用和使用场景是什么?

  **⮑ 解答:** **bind mount** 把已存在的一个目录(或文件)"再挂"到另一个位置,于是**同一份文件在两个路径都能访问**(不是拷贝,是同一对象的两个入口)。
  - **命令形态**:`mount --bind /data/src /mnt/view`(或 `mount -o bind …`);`--rbind` 连同里面的子挂载一起递归绑过去。之后 `/mnt/view` 与 `/data/src` 是同一批文件。
  - **典型用途**:① **把宿主目录暴露进容器**——`docker run -v /host:/in/container`、k8s 的 `hostPath`,底层就是 bind mount;② **只读暴露**——bind 后再 `mount -o remount,bind,ro`;③ **拼装 chroot/构建根**——把宿主 `/usr`、`/lib` 等 bind 进新根,省得复制;④ 把深层目录挂到浅显位置方便访问。
  - **优点**:走**原生 VFS**,数据路径零额外开销。**缺点**:粒度是"一个挂载点"——想让 100 个别名各指不同后端,就得 100 条 bind mount,挂载表膨胀、建立/拆除有成本,且无法表达"按组件名条件判断"。
  - **与 namei_ext 的关系**:它是 namei_ext 的"固定策略版近邻"——同样保留内核数据路径,但映射**静态、挂载点级**;namei_ext 想做**组件级、可编程、每 cgroup 动态切换**。
- **OverlayFS**:lower/upper/work 三层联合视图,是容器镜像分层的事实标准。**关键:OverlayFS 其实已经内置了名字重定向能力**——`redirect_dir` 与 `metacopy` xattr 允许目录在 copy-up 后"重定向"到 lower 的另一路径,`overlay.redirect` xattr 记录重定向目标。这使得 OverlayFS 成为 namei_ext "组件重定向"最危险的近邻(详见关联研究文档)。但 OverlayFS 的重定向是**由文件系统在挂载时按 xattr 静态决定的,不可编程、不可按 workload 动态切换**。
**问:** OverlayFS 的命令形态、作用和使用场景是什么?

  **⮑ 解答:** **OverlayFS** 把两层(或多层)目录"叠"成一个合并视图:下层 `lowerdir`(只读,可多层)+ 上层 `upperdir`(可写),用户看到"上层盖在下层之上"的结果。
  - **命令形态**:
    ```
    mount -t overlay overlay \
      -o lowerdir=/img/base,upperdir=/img/rw,workdir=/img/work \
      /merged
    ```
    读:上层有就读上层,否则穿透到下层;写:第一次写某个下层文件会 **copy-up**(整份拷到上层再改),之后改的是上层副本。
  - **典型用途**:**Docker/容器镜像分层的事实标准**(`overlay2` 驱动)。多个容器共享同一批只读 lower 层,各自一个薄薄的可写 upper 层——这就是"镜像层共享 + 每容器可写层"。
  - **和 namei_ext 最危险的撞车点——内置的"名字重定向"**:OverlayFS 其实**已经能重定向路径名**。当你在合并视图里 `rename` 一个目录,overlay 为了不整目录 copy-up,会在上层打一个 `trusted.overlay.redirect` 扩展属性(xattr),记录"这个名字其实对应 lower 的另一条路径";`metacopy` 则允许"只 copy-up 元数据、数据仍指向 lower"。也就是说,**"一个名字解析到另一个后端对象"这件事,内核里早就有一个(受限的)实现。**
  - **关键差别**:overlay 的重定向是**文件系统在挂载时按磁盘上的 xattr 静态决定的**,服务于 copy-up/rename 内部优化,**不可由用户编程、不能按 workload 动态切换**。namei_ext 想把"重定向到哪"变成一段**可编程、按 cgroup 作用、运行时可换**的 BPF 决策。(正因 overlay 已有静态版,审稿人一定会问"你比 overlay redirect_dir 多了什么"——这是撞车检索重点。)
- **symlink forest**:为每个别名建一个符号链接。灵活但物化成本高(海量 inode/链接),且符号链接语义(`O_NOFOLLOW`、`realpath`、`..` 行为)与"真目录"不同,正确性边界微妙。
- **问:** symlink forest 的"正确性边界微妙"具体指什么?哪些场景下它和"真目录/真文件"的行为会不一致?

  **⮑ 解答:** "正确性边界微妙"是指:**用符号链接拼出来的"视图",在很多细节行为上和"真目录/真文件"并不等价**。这些差异平时不显眼,但某个 workload 一旦踩到就会出错或泄露真实布局。具体有:
  1. **`O_NOFOLLOW` 直接失败**:有些程序为安全用 `open(path, O_NOFOLLOW)`,遇到最后一段是符号链接就报 `ELOOP` 拒绝打开——你的别名是 symlink,它就打不开。
  2. **真实路径会"穿帮"**:`realpath()` / `readlink()` / `/proc/self/fd` 会解析出**后端的真实路径**(如 `tool.real`)。任何拿"规范化路径"做事的逻辑——构建系统拿绝对路径当 cache key、安全检查比对路径前缀、日志打印——都会看到后端名,别名的假象破功。
  3. **`..` 反直觉**:对"指向目录的符号链接",跟进去后 `..` 回到的是**目标的父目录**,而不是链接所在的位置,`cd /view/link/..` 会跳到意想不到的地方。
  4. **`stat` vs `lstat` 不一致**:`lstat`/`ls -l` 会显示它是个链接(`l` 类型、带箭头),真目录不会;有些校验会因"类型不是目录/普通文件"而拒绝。
  5. **inode 身份、硬链接计数、`d_type` 都不同**:把别名当真文件做 inode 去重、`st_nlink` 判断、`find -type f` 过滤时,结果会和真文件不同(`find` 默认还**不跟随**符号链接)。
  6. **相对/绝对链接在移动或 chroot 后断裂**:绝对目标链接被搬进容器/chroot 后指向落空;相对链接对层级敏感。
  7. **rename/unlink 语义**:删/改的是"链接本身"还是"目标"?批量操作时极易误删目标或只删了链接。

  一句话:**symlink forest 是"看起来像视图,但在 `O_NOFOLLOW`、路径规范化、`..`、类型/inode、跨 chroot 等十几个角落和真目录不一样"。** namei_ext 主张自己更"干净":它让内核用**正常 VFS 机制**去解析重定向后的**真实组件**,在这些角落更接近"真目录"而非"套一层 symlink 语义"。(这条优势同样要被审查,见后续文档。)
- **projected volumes / Kubernetes secrets**:把若干来源投射成一个目录视图,是 W2"service fixture"场景的现实对照。
- **问:** Kubernetes 的 projected volume / secret 挂载具体是怎么做的?它是如何把若干来源投射成一个目录视图的?

  **⮑ 解答:** Kubernetes 的 **projected volume / secret 挂载**,是"把若干来源投射成一个目录视图"的工业级真实例子,也是 namei_ext W2(service fixture)场景的现实对照。它的做法:
  1. **来源合并**:`projected` 卷可把多个来源——`secret`、`configMap`、`downwardAPI`(Pod 元数据)、`serviceAccountToken`——**投射进同一个目录**,比如 `/etc/app/` 里同时出现 `username`、`password`、`config.yaml`、`token`。
  2. **落地用 tmpfs(内存盘)**:kubelet 在节点上把这些内容写进一个 **tmpfs**(内存文件系统,secret 因此不落盘),再 **bind mount** 进容器的挂载点。
  3. **原子更新靠"隐藏目录 + 符号链接交换"**(最巧妙的部分):kubelet 不直接覆盖 `username` 等文件,而是——
     - 把新一版内容写进一个带时间戳的**隐藏目录** `..2026_06_26_10_00_00.123/`;
     - 让 `..data` 这个符号链接**原子地**从旧隐藏目录切到新隐藏目录(`rename` 一个 symlink 是原子操作);
     - 用户看到的 `username` 其实是 `username -> ..data/username -> ..2026.../username`。
     于是更新 secret/configMap 时,容器要么看到全套旧值、要么全套新值,**不会读到"半旧半新"**。
  4. **结果**:容器里 `ls /etc/app` 看到一个普通目录,但它实际是 "tmpfs + 一层 `..data` 符号链接间接 + 原子 symlink 交换" 拼出来的**物化视图**。

  **和 namei_ext 的关系**:这正是 namei_ext 想"用一段 BPF 决策替代"的那类东西——K8s 用 tmpfs+symlink+原子交换**物化**出 fixture 视图并支持原子更新;namei_ext 设想用"组件重定向 + 版本 epoch"**不物化**地达到同样效果。所以审稿人会拿 projected volume 当强基线:它已能原子切换、已在生产用了海量次,namei_ext 要证明自己在 setup/update 成本或规模上明显更优(目前 W2 nginx 的"setup 快于 FUSE、update 持平"就是冲这条去的)。

这一层的共同短板:**策略是固定的、粗粒度的,动态重配置代价高,无法表达"按工作负载/版本/上下文编程"的解析逻辑**。

**问:** namei_ext 的粒度到底是什么?为何说 bind/overlay/symlink/namespace 等现有方案是粗粒度的?为何需要按工作负载/版本/上下文编程?这是否是伪需求?

**⮑ 解答:** 分四问回答,最后给"是否伪需求"的诚实裁决。

**(1) 这个项目的粒度是什么?**
namei_ext 的决策单位是**"一次路径行走中的单个组件名"**:在 `lookup`/`readdir` 发生那一刻,对"当前这个名字、在当前这个 cgroup(workload)下,解析到哪个后端组件"做一次 BPF 判断。即最细到**"一个名字一个决策"**,作用域**每 cgroup**,时机**每次解析**。
> 但要诚实:如第 4.4 节所述,**当前 Phase 1 实现的粒度还没到这么细**——ctx 里没有父目录身份、`cgroup_id` 也没填,所以实际只能按"组件名字符串 + 事件类型"决策,即"在挂了策略的 cgroup 内,对**任意目录**里叫 `tool` 的名字统一改写"。真正的"每父目录 / 每视图 / 每代"细粒度,要靠尚未实现的 target registry + view_id/epoch。这条落差本身就是审查要盯的点。

**(2) 为何说别人(bind/overlay/symlink/namespace)是"粗粒度"?**
- **bind / namespace**:决策单位是**整个挂载点 / 整棵子树**。想表达"`tool`→`tool.real` 但只对 build action #4718、且只在这一分钟有效",你得为它新建一套挂载/overlay。
- **OverlayFS**:能重定向,但单位是**整层 + 磁盘上的静态 xattr**,挂载后定死,不能按 workload 在运行时编程切换。
- **symlink forest**:看似"每链接"很细,但它是**物化的**——不是"声明一条规则",而是真在磁盘上建了 N 个 inode;改映射 = 重写一批链接(O(N) 物理操作)。
- 共同点:要么作用单位粗(挂载点/层/子树),要么虽细但需**物化**且**静态**。于是"想要 (workload × 版本 × 上下文) 各不相同的映射"时,它们逼你为每个组合**物化一棵新树**,成本 O(workloads × 树大小)。namei_ext 的卖点是把这件事从"物化一棵树"降成"挂一段 O(1) 的决策逻辑"。

**(3) 为何需要"按工作负载/版本/上下文"编程?**
因为这三个维度在真实系统里确实会变:build action / 容器实例 / agent 任务**各要不同映射**(workload 维);canary/rollout/rollback 要求**同一名字在不同时刻指向不同版本**(版本/epoch 维);决策还可能依赖运行时信息如事件类型、调用方(上下文维)。静态机制要覆盖这些就得穷举物化。

**(4) 是否伪需求?——诚实裁决(这正是对抗性审查的核心争点)**
不是纯伪需求,但**也远没到"刚需已坐实"**,要分情况:
- **真实的一面**:① 超大规模 ephemeral fanout(同时上万个短命沙箱)下,"每个都物化一套"的 setup/存储成本确实会主导,O(1) 挂决策有数量级潜力;② service fixture 替换(W2 nginx)是个**具体、已有正向数据**的真实痛点;③ 它保留内核数据路径,理论上比 FUSE 这种"什么都要的灵活方案"更省。
- **被质疑的一面**:① fork/rollback 的"廉价"**很大程度已被既有 CoW 技术满足**——overlay 快照、Btrfs/ZFS reflink、`git worktree` 都能不复制大树地开分支、回滚;namei_ext 得证明在这些之外还有不可替代的赢点。② 许多 agent 工作负载的真正瓶颈是**容器/VM 启动 + 依赖安装 + 跑测试**,这些 namei_ext 根本不碰,于是"省了路径视图物化"可能只是省了端到端里很小一块。③ 当前最强的"可编程路径解析"卖点所需能力(跨目录、版本视图)恰恰**还没实现**,所以现在的证据更像"在 fixture 替换这个窄场景上比朴素 FUSE 快",而非"通用刚需被验证"。

**问:** overlay 快照和 git worktree 各自的具体场景是什么?各自的工作原理是怎样的?用户是如何使用的?namei_ext 和它们的区别与优势到底在哪里?此外,namei_ext 相比 FUSE 的性能优势是否成立?FUSE 的原理和 namei_ext 的原理分别是什么?namei_ext 相比这些既有手段,killer scenario 是什么?

**⮑ 解答:** 分三部分回答。

### 第一部分:overlay 快照 与 git worktree —— 场景 / 原理 / 用法 / 我们的区别

这两个是"为 agent 开工作区"时**最常用、也最便宜**的既有手段,正因为便宜,它们是 namei_ext 必须越过的门槛。

**A) OverlayFS(当快照/fork 用)**
- **场景**:给每个 agent 任务一个**可写工作区**,但所有任务共享同一份只读基底(repo + GB 级依赖树),不复制大树。失败就丢弃工作区、回到基底。
- **工作原理**:`lowerdir` = 共享只读基底,`upperdir` = 每任务一个**空白可写层**,`merged` = 任务看到的工作区。读:上层没有就穿透到 lower(原生 page cache,多任务共享一份缓存);写:第一次改某文件触发 **copy-up**(拷到 upper 再改)。**fork** = 新建一个 upper 指向同一 lower;**rollback** = 丢弃/清空 upper。配 Btrfs/XFS **reflink** 时,连 copy-up 都能是 O(1) 元数据操作(写时才真正分裂数据块)。
- **用户怎么用**:`mount -t overlay overlay -o lowerdir=base,upperdir=task1,workdir=w1 /merged`,每任务一次;或由容器 runtime、agent 工具自动管。
- **我们 vs overlay**:overlay 给的是"整棵子树的联合 + 可写层",粒度是**目录树 / 挂载点**,视图**绑在 mount(进而 mount namespace)上**。它**做不到**"同一 mount namespace 里不同 task、同一进程里不同 thread 看到不同视图"。大规模 fanout 时每个工作区一次 mount,串行在全局锁(见第三部分)。**但 overlay 现在的强项 namei_ext 没有**:它提供**完整可写分支**(create/unlink/rename 都行)+ **近原生读**;namei_ext Phase-1 只读、且读还慢于 native。所以单论"可写 fork/rollback",overlay(尤其 +reflink)目前更强——这正是文档二把 fork/rollback 轴让给它的原因。

**问:** 什么叫"整棵子树的联合"?什么叫 overlay 提供"完整可写分支(create/unlink/rename 都行)+ 近原生读"?overlay 是 XFS 和 Btrfs 独有的吗?ext4、NTFS 是否支持 overlay?请在必要时画图演示。

**⮑ 解答:** 上一段内容跳跃太快,这里重新组织,一步步配图说明。**并先纠正上文一处容易造成误导的地方**:overlay 和"底层是什么文件系统"是两回事(第 3 小节专门拆开)。

#### 1. 什么叫"整棵子树的联合(union)"

- "**子树**" = 一个目录连同它下面所有东西。"**联合**" = overlay 把**两棵真实目录树**叠在一起,让你在一个目录里**同时看到两边内容**,同名的以上层为准。

两棵真实目录树(都放在普通 ext4 上):
```
lower(只读基底,大家共享)        upper(可写层,一开始是空的)
/base/                            /rw/
├── app.py                        (空)
├── utils.py
└── lib/core.py
```
用 overlay 叠成一个合并视图:
```
mount -t overlay overlay -o lowerdir=/base,upperdir=/rw,workdir=/w  /merged

/merged/   ← 进程看到的只有这一个目录
├── app.py        (其实来自 lower)
├── utils.py      (其实来自 lower)
└── lib/core.py   (其实来自 lower)
```
现在在 `/merged` 里改 `app.py`。overlay 不动 lower(只读),先把 `app.py` **拷一份到 upper**(这一步叫 **copy-up**),再在 upper 上改:
```
lower/(没变)              upper/(多了改过的 app.py)
├── app.py  (原版)        └── app.py  (改过)
├── utils.py
└── lib/core.py

/merged/ = "upper 盖在 lower 上":
├── app.py      ← 来自 upper(改过的,盖住 lower 原版)
├── utils.py    ← 来自 lower
└── lib/core.py ← 来自 lower
```
**这就是"整棵子树的联合"**:`/merged` 是 lower 整棵树和 upper 整棵树**合并后的样子**。关键收益:**lower 一个字节没动、被很多个 `/merged` 视图共享;每个视图只为"自己改过的东西"占 upper 的一点空间。**

#### 2. 什么叫"完整可写分支(create/unlink/rename 都行)+ 近原生读"

把 `/merged` 想成"从共享基底拉出来的一个**文件系统级分支**"(像 git 分支,但作用在文件而非提交)。

- **"完整可写" = 三种写操作都能做,且只影响这个分支、不碰共享基底**:

  | 操作 | 命令 | overlay 在背后做什么 | 对 lower / 别的分支的影响 |
  |---|---|---|---|
  | create 新建 | `touch /merged/new.py` | 直接写进 upper | 无,别人看不到 |
  | 修改 | `echo x >> /merged/app.py` | copy-up 到 upper 再改 | 无,lower 原版不变 |
  | unlink 删除 | `rm /merged/utils.py` | lower 只读删不掉 → 在 upper 放一个 **whiteout**(特殊标记)把它挡住 | 这个分支里消失,lower 和别的分支照样有 |
  | rename 改名 | `mv /merged/lib /merged/lib2` | 目录改名用 `redirect_dir` xattr 在背后记录(就是前面提过的那个) | 只在本分支生效 |

  → **对比 namei_ext**:它 Phase-1 **只读**,你不能通过它 create/unlink/rename。"给 agent 一个能随便改、改完整体丢弃的工作区"这件事,overlay 现在能做、namei_ext 还做不到。

- **"近原生读" = 读一个没改过的文件,速度几乎等于直接读底层 ext4**:

  读 `/merged/lib/core.py`(没改过)时,overlay 只是把读**转交**给 lower 上那个真实文件,走同一个 ext4、同一份 page cache → 比"直接读 `/base/lib/core.py`"慢不了几个百分点,这叫 **near-native(近原生)**。
  → 对比:FUSE 读要进用户态绕一圈(慢得多);namei_ext 当前因 RCU 退化,读元数据比原生慢 **1.5–4.4 倍**。**"近原生读"是 overlay 的硬优势,也正是 namei_ext 目前还没追上的。**

#### 3. overlay 是 xfs/btrfs 独有吗?ext4 / ntfs 支持吗?(这里纠正我上一段)

要分清**两层**,它们是不同的东西:
```
   ┌──────────────────────────────────────────┐
   │  OverlayFS(一种“联合”文件系统,挂载时 -t overlay)     │ ← 负责“叠加 / 重定向”逻辑
   ├──────────────────────────────────────────┤
   │  底层真实文件系统:ext4 / xfs / btrfs / f2fs …        │ ← 负责“把字节真正存盘”
   └──────────────────────────────────────────┘
```

- **OverlayFS 本身不是 xfs/btrfs 独有**。它是一个**独立的文件系统类型**,叠在别的文件系统之上;它的 lower/upper 目录放在 **ext4** 上完全没问题——**Docker 的 `overlay2` 默认就常跑在 ext4/xfs 上,ext4 是最常见的底层**。所以:**ext4 完全支持 overlay。**

- 那我上一段为什么提 Btrfs/XFS?因为我把**两件事揉进了一句话**,这里拆开:
  - **(a) "能不能用 overlay"** —— ext4、xfs、btrfs、f2fs 都行,**不需要** xfs/btrfs。
  - **(b) "能不能让 copy-up / fork 变成 O(1)(reflink 加速)"** —— 这才需要底层支持 **reflink**(块级写时复制,系统调用 `FICLONE`)。支持 reflink 的有 **XFS(`reflink=1`)、Btrfs、bcachefs、ZFS(block cloning)**;**ext4 不支持 reflink**,**NTFS 也不支持**。
  - 区别:在 **ext4** 上改一个大文件,copy-up 要**整份拷贝**(大文件就慢、费空间);在 **XFS/Btrfs** 上,copy-up 可以"先共享同样数据块、等真正写时才分裂"(O(1)、省空间)。**所以底层是 XFS/Btrfs 不是 overlay 的前提,而是一个让 fork/copy-up 更便宜的加速器。**

- **NTFS**:实践中 Linux 上**不会**把 overlay 叠在 ntfs 上(ntfs3 驱动缺 overlay 需要的一些语义),ntfs 自己也没 reflink。可认为 **overlay 是 Linux 生态、叠在 Linux 文件系统(ext4/xfs/btrfs)上的东西,与 ext4 搭配最常见**。

**这一节一句话**:overlay ≠ 需要 xfs/btrfs;它在 **ext4 上跑得好好的(Docker 就是)**。只有当你想让 **copy-up/fork 近乎零成本**时,才需要底层是 **XFS/Btrfs** 这类支持 **reflink** 的文件系统。我上一段写"overlay(尤其 +reflink)"把这层关系压没了,特此更正。 

**问:** overlay 是用户态还是内核态的?是 Linux 自带的开源组件还是第三方工具?上述这些功能不能通过 git 实现吗?overlay 和 git 实现各有什么优劣?overlay 相比 git(branch/worktree)的杀手场景在哪里?

**⮑ 解答:** 分四问。

#### 1. overlay 是用户态还是内核态?是 Linux 自带还是要装包?

- **内核态。** OverlayFS 是 **Linux 内核里的一个文件系统模块**(源码在内核树 `fs/overlayfs/`),用 `mount -t overlay …` 挂载。`mount` 命令本身是用户态工具,但真正干"叠加/copy-up/whiteout"活的逻辑全在**内核**里跑。
- **Linux 内核自带、主线、开源(GPL)**,从内核 **3.18(2014)** 进主线;只要内核编了 `CONFIG_OVERLAY_FS`(主流发行版默认都编),**不用额外装任何包**。它**不是**第三方用户态程序。
- 这正是它和 FUSE 的根本差别:FUSE 的文件系统逻辑在**用户态 daemon**(每个 op 要进用户态绕一圈);overlay **全在内核**,所以读写**近原生**。**Docker / Podman / containerd 的默认存储驱动 `overlay2` 就是它。**

#### 2. 你说的这些,能不能用 git 实现?

**不能——它们管的是不同层面的东西,而且 git 有几个硬限制。** git 只能管"**一个 git 仓库内、被 git 跟踪的文件**",它对下面这些**无能为力**,而这些恰恰是 overlay 的核心用途:

- **不在 git 仓库里的东西**:依赖树(`node_modules`、`.venv`、`/nix/store`)、系统库(`/usr`、`/lib`)、构建缓存、容器镜像层、任意宿主目录——这些通常被 `.gitignore` 掉或根本不在 repo 里。overlay 能叠加**任意目录树**,不要求是 git repo。
- **大文件 / 海量小文件 / 二进制**:git 对 GB 级依赖、大二进制很差(每次 checkout 要重建整棵工作树);overlay 不在乎内容是什么。
- **"只读共享一份 + 写时才占空间"的存储语义**:overlay 的 lower 被多个视图**共享同一份字节**,只有 copy-up 才占空间;而 git **每个 worktree 都 checkout 一整份工作文件**(只共享 `.git` 历史对象,**不共享工作树字节**)。
- **系统级 / 容器 rootfs 视图**:你不会把整个 `/`(`/usr`、`/bin`、`/etc`)塞进 git 再 checkout;overlay 正是为这个而生(容器镜像)。

**问:** 为什么有把 /usr、/bin 这些系统目录纳入版本控制的需求?git 为何在历史设计上每个 worktree 都要 checkout 一整份工作文件,而不能像 overlay 一样进行细粒度的 copy-on-write 控制?

**⮑ 解答:** 两个问题。第一个问题其实与上文中一句说得过于绝对的表述有关,先澄清。

**(1) 真有"把 /usr /bin 塞进版本控制"的需求吗?**
- **澄清**:我的意思**不是**"有人想把 /usr /bin 放进 **git**"——恰恰相反,**没人用 git 管 /usr /bin**(GB 级二进制、不是逐行 diff 的源码、还得被运行中的系统挂载着用)。我只是拿它举例说"overlay 要组合的那种**系统级目录树**,git 管不了"。这句说省了,抱歉。
- **但确实存在一个相关真实需求**:对整棵 OS 树做**快照 / 分层共享 / 原子回滚 / 多实例复用**——只是它**不靠 plain git** 实现:**容器镜像**(base 层被成百上千容器共享,每层内容寻址如一次 commit)、**OSTree/rpm-ostree**(字面"给 OS 文件树做 git 式版本管理",原子升级+回滚)、**Nix/NixOS generations**、**Android/ChromeOS A/B 分区**。
- **结论**:"像版本控制一样管 OS 树"的需求是真的,但由"**镜像层 + CoW 快照**"系统实现,而非 plain git——因为 OS 树太大、是二进制、要被实时挂载。我那句该读成"**git 管不了系统树,这活归 overlay/镜像层**"。

**(2) 为何 git 每个 worktree 要 checkout 一整份,而不能像 overlay 那样 CoW 共享?**
- **先纠误解**:git 的"历史"其实**非常省**——`.git/objects` 是 zlib 压缩 + delta 打包的内容寻址对象,极紧凑。真正"占一份"的是 **working tree(解压成可编辑的真实文件)**;而 `git worktree` **已经共享了 `.git` 历史对象**,只为每个 worktree 物化工作文件。
- **为什么工作文件不 CoW 共享?三个设计原因**:
  1. **存的是压缩/打包形态,不是可用文件**:`.git` 里 blob 是压缩(常 delta 打包进 packfile)的,必须**解压展开**成真实文件才能编辑/编译/运行。checkout = "解压物化",不是"指过去"。
  2. **git 要跨平台通用**:跑在 ext4/NTFS/APFS/FAT/网络盘/Win/mac/Linux 上;**CoW reflink(`FICLONE`)是文件系统专属、并非到处都有**(ext4、NTFS 都没有),核心设计不能假设它存在,所以用最朴素的"写文件"。
  3. **index 假设每个工作文件独立**:git 用 index 记录每文件 stat 判断"改没改";若两 worktree 用 **hardlink** 共享,改一个会污染另一个(而 **reflink** 因"写时分裂"无此问题,故理论可行)。
- **能不能让 git + CoW 一起用?能,大家就这么干**:worktree 放在 XFS/Btrfs 上用 `cp --reflink` 克隆工作树(块级共享、写时分裂、O(1));新版 git 本地 clone 也能对对象库用 reflink/hardlink。并行 agent 正是**把 git worktree(管分支/历史)叠在 overlay 或 reflink(管廉价物化)之上**。
- **根因一句话**:**git 优化"内容随时间的历史与去重"(delta 压缩,历史超省);overlay/reflink 优化"同一时刻多实例对活文件的共享"(块级 CoW)**。git 默认不 CoW 工作树是为了跨平台通用 + 它的数据模型是"压缩逻辑对象 + 独立可编辑工作树";但两者可叠加各取所长。 

**问:** git worktree 产生的是不是全新的工作文件?如果是,为什么不能在发生修改时才在新目录下产生新文件,这样即使没有 reflink 也能模仿 overlay 的 CoW 机制?

**⮑ 解答:** 这个直觉**完全正确**——"只在文件被修改时才产生新文件"正是 CoW 思想,也正是 overlay/reflink 干的事。问题只有一个:**git 做不了,因为 git 根本不在"写"的路径上。** 拆开讲。

**1. 先确认:git worktree 产生的确实是"全新的独立文件"。**
`git worktree add`(以及 `git checkout`)把每个被跟踪文件的 blob 从 `.git` 里**解压展开**,在新目录写出**独立的 inode**(各有自己的数据),默认既不是 hardlink 也不是 reflink。所以对一个大工作树,N 个 worktree = N 份完整物化。(而且 deps 如 `node_modules` 根本不在 git 里,worktree 对它们一点忙都帮不上。)

**2. 你的"改了才拷" = CoW,思想对,但它要求"有人盯着每一次写"。**
"copy on write"的字面意思:**写发生的那一刻**,把要改的文件先拷一份再改。要做到这点,必须有人**站在写操作的必经之路上、在 `write()` 那一刻拦截**。谁能站在那?
```
你用 vim / 编译器 / >> / sed 改文件
        │  这是一次 write() 系统调用
        ▼
  ┌──────────────────────────────┐
  │ 文件系统 / overlay / FUSE 层  │  ← 它们在写的必经之路上 → 能“写时拷贝”
  └──────────────────────────────┘
        ▼  磁盘

  git 在哪? —— 根本不在这条路上。
  git 只在你敲 `git status / commit` 时才运行;
  你编辑文件时 git 没在跑,它无法在“写之前”插一脚。
```
**git 不是文件系统,它不在 I/O 路径里。** 它只能在你**事后**跑 `git status` 时,靠比对 stat/内容**发现**"这个文件改过了"——但那时**修改早已发生**,来不及"先拷再改"。所以 git 没有那个"写时"的时机,做不了 CoW。

**问:** 能否通过 ld_preload + eBPF 拦截 write 的参数,发现文件名命中了 CoW 记录的文件时进行劫持,让它先复制再写?

**⮑ 解答:** 这是个很好的"能不能换个拦截点自己做 CoW"的想法。结论先说:**可以做一个"尽力而为"的玩具,但做不成"可靠/健全"的 CoW;而且上面描述的机制其实就是 overlay 的 copy-up,只不过被搬到了错误的拦截点。** 逐层拆。

**先纠一个技术细节(它会让你的设计更准):CoW 该拦的是 `open()`,不是 `write()`。**
`write(fd, buf, n)` 拿的是 **fd,不是文件名**——到 write 时文件早就 `open()` 过了,你手里只有一个指向 inode 的 fd,没有"文件名命中 CoW 记录"这个信息了。真正能"先复制再写"的时机是 **`open(path, O_WRONLY/O_RDWR…)` 那一刻**(你还握着 path 和"要写"的意图):命中 CoW 集就先把它拷到私有副本、再让 open 打开副本。所以下面都按"拦 open-for-write"讨论。

**你说的是两个完全不同的拦截点,分开看:**

**① LD_PRELOAD(用户态,libc 层)——能做,但不健全,漏洞一大把。**
LD_PRELOAD 让你替换动态链接程序的 libc 函数(`open`/`openat`…)。拦到 open-for-write 时做"拷贝+改打私有副本",逻辑上可行。但它**漏掉**的情况足以让它作为 CoW/隔离**不可靠**(漏一次写就把共享基底改坏):
- **静态链接程序**、**Go 程序**(Go 直接发裸 syscall,绕过 libc)、任何 `syscall()` 直发的程序 → 你的 libc 替换根本不被调用;
- **setuid 程序**:出于安全,LD_PRELOAD 会被内核**剥掉**;
- **mmap 写**:程序把文件 mmap 进内存后直接对内存赋值(编译器、数据库、动态加载器常这么干)——**根本没有 write() syscall 可拦**;
- 程序给子进程 unset 掉 LD_PRELOAD、或在你的 shim 之前就拿到了 fd;
- TOCTOU 竞争(检查"是否命中"和"拷贝+改打"之间有窗口)。
→ 它跑在**目标进程里**、靠目标"配合"才生效,**一个不配合的程序就能绕过**。当"尽力而为的协作式 shim"凑合,当"保证"不行。

**② eBPF(内核态)——能 observe / deny,但做不了"拷贝后再写"。**
关键事实:**eBPF 基本是"观察 + 放行/拒绝",不是"暂停这次操作、去干一堆别的活、再恢复"。** 在 write/open 的内核 hook(BPF-LSM / kprobe / tracepoint)里:
- 它能**看**参数、能**拒**(LSM hook 返回 `-EPERM` 把操作变成错误)、能**通知**用户态;
- 但它**不能**"暂停这次 open,自己去**打开另一个文件、读 N 字节、写出一份副本**(这本身是会阻塞、要分配、要做 VFS I/O 的整文件拷贝),再回来把 fd 指向副本"。BPF 程序跑在**受限、基本不可睡眠、有界**的上下文里,**没有"拷贝整个文件"这种原语**,也不能在一个热 syscall hook 里做任意阻塞 I/O;
- 它**更不能**把"这次 open 即将得到的 fd"在中途**改指到另一个 inode**(copy-up 的本质是"让后续访问落到新 inode",这正是 BPF 在 write/open hook 里做不到的)。
→ 所以"BPF 拦 write → 拷贝文件 → 放行"**不是 eBPF 能干的事**。它最多**拒掉**,或**喊一个用户态 daemon 去拷**——但那一喊,你就**重新发明了 FUSE/overlay,还附赠额外延迟和竞争**。

**③ 关键洞察:你描述的"命中就先拷再写"本来就有,它叫 overlay 的 copy-up——而且它做得健全,正因为它在对的拦截点。**
overlay 之所以能**可靠**地"写时才拷":它**就是文件系统**,坐在 inode/fd 上,**所有写都绕不过它**——syscall 写、mmap 写、静态/动态、Go/C、每个进程,全都经过 VFS 落到那个文件的 inode,于是它能**原子地**把上层副本顶上去。**任何 interposition(LD_PRELOAD 或 BPF-at-write)都达不到这种完整性和原子性**,因为它们覆盖不全、也无法原子换 inode。"做 CoW 的正确地方,是那个**拥有 inode、且在所有写者必经之路上**的层"——这是结构性事实,不是工具不够强。

**那你的想法有没有"正确版本"?有,但都不是"BPF 改写 write":**
- 要**可靠 + 可编程**:就是 **fuse-bpf / 一个 stacked 文件系统 / overlay**——在文件系统层做 copy-up(付 FUSE 代价或直接用 overlay)。
- 要**syscall 级拦截**:用**真正的沙箱**——`seccomp-user-notify`(seccomp unotify)或 ptrace/gVisor,把 open 陷入一个**监督进程**,由它"命中 CoW 就拷贝+重打开副本"。这能健全(因为陷的是内核 syscall 入口、进程绕不过),但**重**(每个相关 syscall 都陷一次给监督者),这就是文献里"用户态 supervisor 虚拟化文件效果"那一类(如 Sandlock)。**LD_PRELOAD 给不了这种健全性**(它在用户态、可被绕过)。

**④ 回到 namei_ext(同一个原理,正反两面)**
- 这恰好镜像出 namei_ext 为什么成立、又为什么不做写:**namei_ext 把 hook 放在 `lookup`**——这是**所有名字解析的唯一内核 chokepoint**,而且它只需做**替换一个名字**(便宜、有界、不阻塞、不做 I/O),所以一个 BPF hook 就能干。
- **CoW-on-write 是另一回事**:它要 ① 抓住**所有写者(含 mmap)**,② 做**无界的整文件拷贝 + 换 inode**——这俩都不是 lookup 式 hook 或 BPF 程序能做的。
- 所以同一条原则("**守在所有人必经的 chokepoint 上,且动作要足够轻**")**既解释了为什么 redirect 适合 BPF lookup hook,也解释了为什么 CoW 不适合 BPF write hook**。写时复制的 chokepoint 是文件系统(overlay),且"拷一个文件"这个动作对 BPF 太重——这正是 namei_ext 只读、把可写 CoW 让给 overlay 的根本原因。

**问:** Lookup 是一个 syscall 吗?namei_ext 中是如何替换名字的,是 rename 原子操作吗?此外,DeltaBox 这个最近的对比对象做了什么、和 namei_ext 有何区别?namei_ext 相对它能有 FAST/OSDI 级别的创新优势吗?

**⮑ 解答:** 分两部分。

#### 第一部分:lookup 是 syscall 吗?怎么"替换名字"?是 rename 原子操作吗?

**(1) lookup 不是 syscall,是内核里的一个"内部步骤"。** 你在用户态发的是 `open`/`openat`/`stat`/`statx`/`access`/`execve`/`rename`/`unlink`…… 这些 syscall **进内核后,都要把路径一段一段走下来**(`/a/b/c` 先解析 `a`、再 `b`、再 `c`)。**每解析"一段组件"就是一次 lookup**(内核函数 `walk_component()`/`link_path_walk()`)。所以 lookup 是"**几乎所有带路径的 syscall 共用的内部公共步骤**",namei_ext 的 hook 插在这个内部步骤里——这也是它"一处 hook 覆盖 open/stat/exec/… 全部"的原因。

**(2)"替换名字"= 在这一次解析中,把内核内存里"要查的那个名字"换掉,磁盘一个字节不动。** 走到组件时,内核已把当前组件解析成 `nd->last`(一个 `qstr` = 名字 + 长度 + 哈希)。REDIRECT 时把 `nd->last` **临时换成**一个校验过的同父组件(如 `tool.real`,在同一 parent 下重算哈希),让正常 VFS 去查这个新名字,**查完再还原原 `qstr`**。

**(3) 它不是 `rename`,也不是磁盘上的原子操作——这点很关键:**

| | `rename(2)`(syscall) | namei_ext 的 redirect |
|---|---|---|
| 改什么 | **磁盘上真实的目录项** | **只改这一次 lookup 在内存里"查哪个名字"** |
| 持久性 | 持久,所有人都看到 | 临时,只影响这一次解析 |
| 范围 | 全局,对所有进程 | 每 task/cgroup,只对挂策略的工作负载 |
| 磁盘是否变 | **变** | **不变**(没有条目被创建/移动) |
| 原子性 | POSIX 保证的原子磁盘操作 | 谈不上"原子 rename",它根本不碰磁盘 |

→ 一句话:**redirect 不是"把文件改名",而是"这次找路时临时把你指到另一个已存在的名字"。** 正因它不是磁盘原子换名,**lookup/readdir 一致性才要单独论证**(审查 C5:别名可见性是"附带的",不像 rename 那样天然一致)。

#### 第二部分:最致命的 arxiv 对比对象 = DeltaBox,区别与能否有 FAST/OSDI 级优势

**它是谁**:**DeltaBox**(arXiv [2605.22781](https://arxiv.org/abs/2605.22781),上海交大 IPADS,2026,顶级组,FAST/OSDI 审稿人**一定**知道)。它与 namei_ext 转向后的主场景**完全撞**:agent sandbox lifecycle、从 pinned 状态 fork/fanout、checkpoint rollback/restore、SWE-bench framing。

**它怎么做**:`DeltaFS`(运行时**重配 overlayfs 层栈**:冻结可写 upper、插入新 upper)+ `DeltaCR`(**CRIU 式进程内存 checkpoint/restore**)。**实测**:checkpoint ~1.12ms、restore ~5ms、**1419 forks/秒**、每子沙箱 ~11MB,**支持写隔离 + 内存回滚**。

**区别(一句话:它在 fork/rollback 这条轴上严格做得更多)**:DeltaBox 是**整沙箱的 snapshot/fork/checkpoint 引擎**,粒度是"一个沙箱(挂载+进程树)",给写隔离/真 fork/内存回滚;namei_ext 是**只读、同父、单组件的 BPF redirect**,不 fork、不 checkpoint、无写 union、无内存状态、无 rollback、无跨目录。**在 fork/checkpoint/rollback 轴上 DeltaBox 完胜——namei_ext 绝不能争"agent fork 更快",懂 DeltaBox 的审稿人会当场击穿(fatal 质疑 C17)。**


**问:** DeltaBox 究竟做了什么?请用具体的步骤和例子来解释,避免过多抽象名词。此外,per-task/per-thread 零挂载视图和 namespace_sem 的"悬崖"效应到底是什么?什么是 semaphore(信号量)?以下几个概念分别是什么意思:静态预计算表;运行中的原子切换(什么场景下会有这个需求,比如数据库?);可编程性在这里具体指什么?per-task 视图能否举一个具体应用例子,为什么现有方案做不到?能否从理论上论证需要 per-task 视图和可编程性的需求,能否找到对应的场景?什么是"重配 overlay 层"、"CRIU 进程"?

**⮑ 解答:** 这次全程用大白话 + 具体例子,凡名词第一次出现就解释。

#### 1. DeltaBox 到底做了什么(忘掉那些名词)

**场景**:一个会写代码的 AI(agent)在帮你修 bug,它像下棋引擎一样**试很多步**:"改文件 A → 跑测试",失败了就想**退回到改之前**再试别的;成功了想**复制一份**继续往下探。要做到"退回/复制",它得能**给整个沙箱拍快照、然后秒级还原**。

**老办法的痛点**:拍一次快照要把**整套东西全拷一遍**,慢到**几秒**。一次搜索要拍几千次,几秒×几千 = AI 大部分时间耗在"存档/读档",而不是干活。

**DeltaBox 做的就是给这个沙箱做一个"快速存档/读档/克隆按钮"**,分两块:

- **文件这块(他们叫 DeltaFS)**:把文件想成**投影仪上一摞透明胶片**——最底下那张是原始代码仓库(只读),你新写的东西写在**最上面一张新胶片**上,从上往下看就是合并后的样子。**要存档,就把当前最上面那张胶片"冻住",再铺一张新空白胶片上去**——这只是个"换张纸"的小动作,**不是拷贝**;**要退回,就把最上面那张扔掉**。"重配 overlay 层"这个吓人的词,翻译过来就是这句"**冻住当前这张、再铺一张新的**"。(这摞胶片就是前面讲过的 OverlayFS。)
- **内存这块(他们叫 DeltaCR)**:沙箱里还有个**正在运行的程序**,它内存里有一堆状态(变量、跑到一半的测试进程)。要真正"时光倒流",光存文件不够,还得把**这个程序的内存也存下来、再装回去**。用的技术叫 **CRIU**(可以理解成"**给单个程序做休眠/唤醒**"——像合上笔记本盖子休眠、再打开恢复到原样,只不过是对一个程序做)。DeltaBox 让它快的诀窍:**只存"变了的那部分内存",不是全部**。
- **结果**:存档从"几秒"降到**毫秒级**;每秒能克隆约 **1400 个**沙箱;每个克隆只多花 **~11MB**。

**一句话**:**DeltaBox = 给整个 AI 沙箱(文件 + 运行中程序的内存)做的"快速存档/读档/克隆按钮",让 AI 能反复试错、秒级撤销,而不是每次浪费几秒。** 而 namei_ext **这些都不做**——它只改"查一个文件名时,把你指到哪个文件"。所以在"时光机"这件事上 DeltaBox 完胜,我们不去争。

#### 2. "per-task 视图 + namespace_sem 悬崖"是什么?顺便讲什么是"信号量/锁(sem)"

**先讲"视图"**:就是"一个程序眼里能看到哪些文件/路径"。两个程序可以对同一块硬盘看到**不同的视图**。

**给不同视图的常规办法 = "挂载(mount)"**:挂载就像"在某个位置插一块盘"。但有个硬限制:在 Linux 里,**一个程序(进程)和它内部所有"工人线程"看到的是同一套挂载**。所以如果一个程序开了 50 个线程,这 50 个线程**只能看到同一个文件视图**——你**没法**用挂载给"线程1"和"线程2"不同的视图。这是**结构上做不到**,不是慢不慢的问题。

**per-task / per-thread 视图 = 给每个任务/每个线程它自己的文件视图。** namei_ext 能做,是因为它在"查名字"那一刻按"是谁在查"来决定,**根本不用挂载(零挂载)**——所以它能干挂载干不了的事:**一个程序内部,不同线程看到不同文件**。

**什么是信号量(semaphore,简称 sem)/锁**:就是一把**"轮流用"的锁**。打个比方:**一个厕所只有一把钥匙**——同一时间只能进一个人,别人排队等。锁保证了不出乱子(不会两个人同时改坏共享的东西),但代价是**把并行排成了队**。

**`namespace_sem` 是什么**:它是 Linux 内核里**一把全局的锁**,专门守着"挂载表"。**任何人要加/删一个挂载,都得先抢到这把锁** → 大家轮流来。

**"悬崖"是什么**:如果你想靠"挂载/卸载"去开**几千个**短命沙箱,它们**全挤在这一把厕所钥匙前排队**。几个的时候没事;几千个的时候,**队伍爆炸、整台机器卡死**,性能**断崖式**下跌。这就是经典的"**一把全局锁,大家抢**"问题。

**对我们的意义**:namei_ext 给视图**不用挂载**(只是查名字时做个判断,不抢那把全局锁),所以在"开海量沙箱"时**绕开了这个悬崖**。**这才是它真正的结构性优势——不是"挂载更快",而是"压根不挂载,所以不用排队"。**

#### 3. "静态预计算表 vs 可编程",以及"运行中原子切换"(配数据库例子)

- **静态预计算表**:**事先**把所有规则写死在一张固定的表里。比如:"名字 `tool` → 文件 `tool-v2`;名字 `config` → 文件 `config-prod`"——一张**跑之前就定好、跑的过程中不变**的清单。
- **可编程(eBPF)**:不是固定清单,而是装一小段**程序**,在运行时**现算**答案,而且能**依赖实时情况**、还能**一边跑一边换**。

**我那段话的意思(翻译成大白话)**:如果你**跑之前就完全知道**每个名字该指向哪、而且**跑的过程中永远不变**,那**一张固定表就够了**,用不着"程序"。**程序只有在"映射需要一边跑一边变"的时候才有价值**(这就叫"在线动态")。

**什么时候映射会一边跑一边变?具体场景:**
- **灰度发布 / 金丝雀(canary,服务领域天天用)**:你要把 `lib` 从 v1 换到 v2,但先让 **5%** 的负载用 v2、观察有没有出错,再升到 50%、100%;一旦 v2 出问题,**立刻把所有人切回 v1、不用重启**。"`lib` 这个名字现在指哪个版本"这件事**随时间变、还分人群** → 固定表写不出"现在 5%→50%→100%、出错就回滚",得靠一段能实时改的程序。
- **数据库类比(你问的)**:这正像数据库的 **MVCC(多版本并发控制)**——一个事务看到的是某个**一致的版本快照**,新事务看到更新的版本,**版本切换是原子的**,没人会看到"一半旧一半新"。
- **"运行中原子切换"** 说白了 = **在系统不停机的情况下,把"这个名字解析到哪"对所有人(或某一群人)在某个瞬间干净利落地一次性切过去,而且能再切回来**。就像**蓝绿/金丝雀部署**:把流量从旧版本一瞬间切到新版本,坏了立刻切回。

**这对论文为什么关键**:我们评估用的是**确定性、事先录好的命令回放**——可固定回放的脚本**恰恰是"静态表"最擅长的情形**,所以它**本身证明不了"非要可编程"**。要证明需要 eBPF 可编程,**必须拿出一个"运行中会变"的场景**(灰度/回滚/原子版本切换),否则审稿人会说"一张固定表就够了"。

**问:** 是否可以这样理解:DeltaBox 只能做固定表所以测 SWE-bench,而 namei_ext 因为灵活所以不测 SWE-bench? 

**问:** 现有的 namei_ext 仓库实现跑过 SWE-bench 吗?

**⮑ 解答:** **不对,这里有个误读,我先把它拆开;然后回答第二问(答案是:目前没测)。**

#### 1. 误读纠正:"DeltaBox 只能做固定表" 和 "我们灵活所以不测 SWE-bench" 都不成立

**(a)"DeltaBox 只能做固定表"——这是个张冠李戴。**
"静态表 vs 可编程"这个区分,讲的是**我们(namei_ext)自己策略怎么写**(我们的 redirect 规则是一张写死的查找表,还是一段能实时改的 eBPF 程序)。它**跟 DeltaBox 没关系**。DeltaBox **根本不是做"名字→后端"这种表**的东西——它是个**存档/读档引擎**(冻 overlay 层 + CRIU 进程休眠/唤醒)。所以"DeltaBox 只能做固定表"是把两个不相干的东西混在了一起。

**(b)DeltaBox 测 SWE-bench,跟"固定表"无关。**
DeltaBox 测 SWE-bench(MCTS 树搜索 / RL rollout),是因为**那正是"需要反复 checkpoint/rollback"的真实 agent 场景**(深度搜索要回退)——SWE-bench 是它**rollback 这个卖点的动机/正确性工作负载**,不是因为它"只能做固定表"。

**(c)"我们灵活所以不测 SWE-bench"——恰恰相反,我们应该测,而且我们本来就计划测。**
仓库转向后,工作负载就是**从 SWE-bench Verified / SWE-bench Pro 派生的命令 trace**(见 `docs/tmp` 的命令 trace 调研)。"灵活 → 不测 SWE-bench"这个推论**不成立**。我那段话的真实意思是另一回事:

> **一段"确定性、事先录好"的 SWE-bench 回放,恰好是"静态表"就能搞定的情形;所以"只用确定性回放"这件事本身,证明不了我们"非要可编程"。**

注意:这是对**我们自己评估方式**的批评,**不是说"别测 SWE-bench"**。正确做法是:**SWE-bench 留着(用于真实性 / 动机 / 正确性),再额外加一个"运行中会变"的实验(灰度 / 原子 epoch 切换 / 出错回滚 / fleet churn)**,用它来 justify "可编程"这个词。**= SWE-bench 照测 + 补一个动态实验**,而不是"用灵活换掉 SWE-bench"。

一句话:**"静态表 vs 可编程"是在说我们的策略机制;它得出的结论是"SWE-bench 确定性回放 + 再加一个在线动态实验",不是"DeltaBox 是固定表"、也不是"我们不测 SWE-bench"。**

- SWE-bench **只停在计划**:命令 trace 调研文档里写的是"下一步:搭建命令 trace 采集器、先跑 3 个任务的冒烟";仓库里**没有 `workload/` 目录、没有 SWE-bench/agent harness、没有任何 SWE-bench 结果**。

**问:** 能否找到服务于这一场景的、被工业界或学术界认可的 benchmark?

**⮑ 解答（续 1）:** 诚实说:**没有一个"既正好服务我们场景、又被业界/学界公认"的现成 benchmark——这是真实痛点。** 拆开看:
- **agent 侧的公认 benchmark = SWE-bench / SWE-bench Pro**(业界+学界都认),但它们打分的是"**agent 有没有把任务做对**"(能力指标),**不是"文件系统/沙箱开销"**。公认,却不直接测我们要的东西。
- **文件系统侧的公认 benchmark**(Filebench、fio、各类元数据基准)有,但**不是 agent 场景**。
- **没有任何一个同时是"我们场景(agent 路径视图准备/规模)+ 公认"的 benchmark**,这是已知空白。
- **标准做法(DeltaBox 也是这么干的)**:**拿一个公认的 agent 工作负载(SWE-bench MCTS/RL)当真实驱动,再自己搭一套系统指标 harness**(setup 时间、挂载数、拷贝量、随 N 的扩展性)去测。DeltaBox 就是用 SWE-bench Verified 当工作负载、测它自己定义的 checkpoint/restore/fork 指标。**我们应照搬**:"**公认的工作负载来源(SWE-bench / 公开 trajectory 数据集)+ 我们自己定义并公开的系统指标**"。
- 所以"被认可"来自**工作负载(SWE-bench)**,而非一个现成的"路径视图开销 benchmark"(那个不存在,得我们自己定义,像 DeltaBox 定义它的指标一样)。

**问:** SWE-bench 为何被称为"静态"的?它不包含版本发布之类的内容吗?

**⮑ 解答（续 2）:** 这里"静态"不是说"SWE-bench 没有版本",我说精确点:
- **我说的"静态" = "确定性、事先已知"**,讲的是**我们回放方式**的性质,不是说 SWE-bench 内容空。把一个 SWE-bench 任务录成固定命令 trace 回放时,**文件操作的序列是确定的、跑前就知道** → 一张静态映射表就能服务它。
- **SWE-bench 有没有版本/发布?——有,但是"钉死的版本",不是"在线切换"**。要区分两种"版本":
  - **① pinned 依赖版本(静态、每任务定死、跑前已知)**:每个任务都 pin 了具体 repo + `base_commit` + 依赖版本("在提交 X 上修这个 bug")。**SWE-bench 有这个**——但它正是静态表最擅长的情形。
  - **② 运行中的在线版本切换 / 灰度 / 原子回滚**:任务 #417 永远是提交 A + 依赖集 V,**跑的过程中不会发生"把 v2 灰度到 5%、再回滚"**。**SWE-bench 没有这个。**
  - **justify "可编程"需要的恰恰是 ②(运行中变),而 SWE-bench 只有 ①(钉死)。** 所以"SWE-bench 回放是静态的"指的是缺 ②,不是说它没版本。
- 连 SWE-bench MCTS 里那点"动态"(agent fork 出分支去探索)也是**状态层面的 fork/rollback(DeltaBox 的地盘)**,不是"同一个名字在线解析到不同版本"(我们的地盘)。所以 SWE-bench 自带的动态性**也不替我们 justify 可编程**。
- **结论**:SWE-bench 有版本,但是**静态钉死**的;要撑"可编程",得**在 SWE-bench 之外另造一个"运行中切版本/灰度/回滚"的实验**(见前面 §3 灰度场景)。


#### 4. "可编程性""per-task 视图"在这里到底指什么?举个应用例子,为什么对手做不到?

- **这里的"可编程性"**具体就是:"名字 → 后端文件"的判断规则是**一小段你能写、能随时改的程序**,还能用实时信息。例子规则:"**如果发起请求的工作负载属于金丝雀组,且版本 ≥ 8,就把 `libfoo` 解析到 `libfoo-v2`,否则 `libfoo-v1`**"。
- **"per-task 视图"举个具体应用**:设想**一个**服务进程,内部用 **1000 个线程**同时跑 1000 个 AI 修 bug 任务(为省下开 1000 个独立进程/容器的成本)。线程 #417 在修"仓库 R 的提交 A",线程 #902 在修"提交 B"。你想让 **#417 看到的 `/repo` 是提交 A 的文件、#902 看到的是提交 B**——**在同一个进程里**。
  - **挂载/overlay/bind:做不到**——挂载被整个进程的所有线程共享(大家共用一张挂载表)。要分开,只能把每个任务塞进**各自独立的进程 + 各自的挂载命名空间**,那更费内存和启动成本,几千个还会撞上面说的"那把全局锁"悬崖。
  - **FUSE:能给每人不同视图,但每个文件操作都要绕一趟用户态,慢。**
  - **namei_ext:在"查名字"那一刻按"是哪个任务"来判断,不挂载** → #417 和 #902 在**同一个进程里**看到不同的 `/repo`,而且便宜。**这就是只有它占的格子。**

#### 5. 能在理论上论证"需要 per-task 视图 + 可编程性"吗?有对应场景吗?(诚实版)

- **per-task 视图什么时候是刚需(理论上)**:**当且仅当**你要在**同一个挂载命名空间内**让 N 个不同视图共存(比如一个进程里很多线程,或一个命名空间下很多 cgroup),**而且 N 大到"给每个都配独立命名空间"太贵**(内存 + 那把全局锁的排队悬崖)。结构性事实:挂载是"命名空间级"的、线程共享命名空间,所以"一个命名空间/进程里要很多视图"**用挂载在任何速度下都达不到**——这是个**能力缺口**,不是速度差。
- **可编程性什么时候是刚需(理论上)**:**当且仅当**映射要**在运行中变**(灰度/回滚/原子版本切换/整个集群实时更新),或依赖你事先无法穷举的实时上下文——因为静态表**定义上**就是"装载时定死、不变"。
- **一个两者都需要的场景**:一台机器上跑**一大群 agent 沙箱(以线程/cgroup 形式塞在少数几个进程里)**,其中 (a) 每个要自己 pin 的 repo+依赖视图【需要 per-task】,并且 (b) 你正在灰度一个新依赖版本,必须在**运行中**对这群沙箱的依赖视图做**原子切换或回滚**【需要可编程 + 在线动态】。挂载(做不到 per-thread + 挂载悬崖)、静态表(切不动、回滚不了)、FUSE(每次操作绕一趟)**都覆盖不了这个格子**。
- **但要诚实(我一直在提的 C4)**:这个场景**逻辑上成立、也合理**,但论文**还没指出一个真在这么跑的真实系统**(今天的 agent 基础设施大多给每个沙箱**各自一个容器/虚机/命名空间**,于是"独立命名空间"是白送的,用不着 per-task)。所以**理论需求是真的,缺的是"真有人这么用"的实证**。结论:**你能从理论上论证它(上面那个"线程化沙箱群 + 灰度依赖"的场景是最干净的一个),但论文要落地,还得找到或造出一个可信的真实客户,光有理论不够。**

#### 6. 顺手收尾的两个名词(再确认一次,免得还绕)
- **"重配 overlay 层"** = 前面胶片那个比喻:**把当前最上面那张可写胶片冻住,再铺一张新空白胶片**(用于快照/回滚)。
- **"CRIU 进程"** = **给一个正在运行的程序做"休眠/唤醒"**:把它内存状态导出成文件存起来,以后再装回去让它从原地继续跑。

> 一句话总览:**DeltaBox = 整沙箱的"快速存档/读档/克隆按钮"(文件用胶片分层、内存用休眠/唤醒);namespace_sem = 守挂载表的"一把全局厕所钥匙",几千个挂载抢它就卡成悬崖;per-task 视图 = 同一个进程里不同线程看不同文件(挂载做不到、只有我们能);可编程 = 规则是一段能实时改的小程序(只有"运行中要变"如灰度/回滚才用得上,固定回放用静态表就够)。**

**能否有 FAST/OSDI 级优势?——有条件的"能",但必须换一条轴,且现在还没兑现:**
- **在 DeltaBox 主场(fork/rollback 延迟、写隔离、内存回滚):不能,别试。**
- **在 DeltaBox 结构上**没占**的轴上:可能能。** DeltaBox 给不了:① **可编程、按组件、运行时的路径解析策略**(它组合整层,不做按名字决策);② **per-task/per-thread、绕开 `namespace_sem` 的零挂载视图**(它重配 overlay 层栈是挂载级、mount-ns 作用域;1419 forks/s≈0.7ms/fork 虽快,但仍是**每沙箱一次 mount+CRIU**,不是 **O(1) 无锁 per-lookup 决策**);③ 一个**能被 DeltaBox 复用的视图原语**(互补而非替代)。
- **诚实战略结论(与文档二/三一致)**:把 **DeltaBox 当基线 + related-work 核心,显式让出 fork/checkpoint 轴**,重定位成"可编程、按组件、零挂载的 per-task 路径视图原语,DeltaBox 这类系统可建在其上";**FAST/OSDI 级 delta 只能来自 DeltaBox 没占的轴(组件粒度可编程 + per-task/per-thread 视图)**;**但这条 delta 现在还没兑现**——需要把跨目录/view_id/epoch 可编程性(未实现,fatal C1)和 per-task 杀手应用 + `namespace_sem` 悬崖证据做出来。**在做出来前,namei_ext 既没 DeltaBox 的 fork 赢点、也没自己差异化的赢点**——这正是判定"现状不可投"的原因。

> 一句话:**和 DeltaBox 比 fork/rollback 必输;唯一可能的 FAST/OSDI 级优势是"它做不到的可编程按组件 + per-task 视图",但那是设计方向、尚未实现。** 正确动作:把 DeltaBox 写进 related work、让出快照轴、去把那条差异化轴建出来并测出 `namespace_sem` 悬崖图。

- **需不需要"per-task 视图"?——这是最强的差异点,但前提是真有人需要它(质疑 C4)**:per-task/per-thread 是 mount/overlay **结构上做不到**的唯一格;**但论文目前没指名一个"非要 cgroup/per-task 作用域、而不是各自 mount namespace"的客户**。agent 沙箱、build action、serverless worker 通常各自就有自己的 mount namespace。**所以 per-task 是真差异点,但"谁非要它"还没钉死**——这也是为什么文档三把"找到那个 customer + 测 `namespace_sem` 悬崖"列为投稿前必做。

#### (3) 我们会比 DeltaBox 有"理论"性能优势吗?

- **在 DeltaBox 的轴上(fork/checkpoint/rollback):没有**——它做得更多(写隔离 + 内存回滚),namei_ext 连写都没有。
- **在 per-task 视图供给这条轴上:理论上有,但很窄、且在另一条轴上**:
  - namei_ext 提供一个视图 = **一次 O(1) 的 per-lookup BPF 决策,零挂载、无锁**;
  - DeltaBox 提供一个沙箱 = **重配 overlay 层栈(挂载级操作)+ CRIU 进程 C/R**,是 **mount-namespace 作用域**的,**做不到同进程不同线程不同视图**,且每个沙箱有挂载 + C/R 的成本(它报 1419 forks/s ≈ 0.7ms/fork,已经很快)。
  - **所以理论优势不是"fork 更快"**(人家 fork 的是含内存的整沙箱),**而是"粒度根本不同:不挂载、不起进程,只做一次解析决策"**。在"一个 namespace/进程里要 N 个视图"这个场景,namei_ext 是 O(1) 无锁、mount 类(含 DeltaBox 的层栈重配)会撞 `namespace_sem` + 每沙箱挂载成本。**这是真实的结构性优势,但只在 DeltaBox 没优化的那条轴上,且需要工作负载真的要"同 namespace 多视图"才用得上,目前未经验证。**

**3. 那"hardlink 共享 + 改了再拷"为什么也不行?**
你可能会想:那就先让新目录的文件和基准 **hardlink 共享**(不占空间),等改时再拷。问题:**hardlink 是同一个 inode**,而 git 抓不到"正要改"的时机;很多工具是**原地写**(`>>` 追加、`sed -i`、编译器覆盖 `.o`、mmap 写、数据库),一旦原地写,**两个 worktree 的文件同时被改坏**(同一个 inode),等 git 发现已经晚了。
- 这正是为什么 **`git clone --local` 只敢 hardlink `.git/objects`**(那些对象**内容寻址、不可变、永不原地改**),而**绝不 hardlink 工作树文件**(工作树文件是要被改的)。
- 而 **reflink** 之所以行、hardlink 不行:reflink 是**独立 inode 只共享数据块**,内核在你写的那一刻**自动写时复制**受影响的块——"改了才真正分裂"由**内核**在写路径上替你做了,应用毫不知情。这恰恰是你描述的机制,只是必须由**内核**来做。

**4. 关键:你描述的"改了才拷"其实已经存在,它叫 overlay(哪怕没有 reflink)。**
- **overlay 在 ext4 上(没有 reflink)就已经是你说的那样**:lower 共享,只有第一次写某文件时才 copy-up(把那个文件拷到 upper)。**没有 reflink 也能"懒拷贝"**——只不过是**整文件**拷,而非块级;reflink 只是让这次拷贝更便宜(块级、写时才分裂)。
- 所以**你的方案 = "把 git worktree 放在 overlay 之上"**:lower = 一份共享的基准 checkout,upper = 每个 worktree 自己的可写层。这样**没有 reflink 也能模仿 CoW**,而且**现成、Docker 天天在干**。只是干这事的是 **overlay**(站在写路径上的文件系统),**不是 git**。

**5. 结论(也回到 namei_ext)**
- **要"只在修改时才物化",你不该改 git,而应把 git 的工作树放到 overlay / reflink 文件系统之上**——**git 管版本语义,overlay/reflink 管"写时才拷"。** 这就是前面说的"git 叠在 overlay/reflink 上,各取所长"。
- 顺带点题:**"谁在写路径上"决定了谁能玩这种把戏。** overlay 在写路径上、能 CoW;**namei_ext 在 lookup(读解析)路径上**,所以它天然能做"**把名字解析到哪**"(视图/redirect),但它当前**不在写路径上做 copy-up**——这正是它只读、把"可写 fork"让给 overlay/CoW 的根本原因(写时复制不是一个 lookup hook 的活)。

一句话:**git 能模拟"源码的分支/回滚",但模拟不了"把任意目录树(尤其依赖/系统/缓存)按层共享叠加"。**

#### 3. overlay 和 git 各自优劣

| 维度 | OverlayFS | git(branch / worktree) |
|---|---|---|
| 层次 | **内核文件系统** | **用户态 VCS(版本控制)** |
| 管什么 | **任意目录树**(repo / deps / 系统 / 缓存) | **只管被 git 跟踪的仓库文件** |
| 共享方式 | lower **共享一份字节**,copy-up 才占空间 | 共享 `.git` 历史对象,但**工作树文件每个 worktree 一整份** |
| 大文件 / 依赖 | 无所谓 | 差(大二进制 / GB 依赖很痛) |
| 操作语义 | 文件系统级(create/unlink/rename/whiteout) | **提交级**(commit/branch/merge/diff/history) |
| 历史 / diff / merge / 协作 | **没有**(它不懂"版本历史") | **强项**(这是 git 的本职) |
| 回滚 | 丢弃 upper 层 | reset/checkout,且有**完整历史可回到任意点** |
| 读性能 | 近原生 | 不涉及(它不是运行时的文件系统层) |
| 谁在用 | 容器镜像、CI sandbox、依赖共享 | 源码协作、并行 agent 分支 |

**一句话**:overlay 强在"**任意目录树的廉价分层共享 + 文件系统级读写 + 近原生**";git 强在"**源码的版本历史 + 协作语义(diff/merge/PR)**"。**它们解决的是不同问题,不是同一把锤子。**

#### 4. overlay 相比 git(branch/worktree)的杀手场景

overlay 能、git 根本做不了的:
- **容器镜像分层(overlay 的本命)**:一个 base 镜像层被成百上千个容器**共享一份**,每容器只加一个薄可写层。你不可能把整个 OS rootfs 放进 git 再给每个容器 checkout 一份。
- **依赖 / 缓存共享**:一份 `node_modules` / `.venv` / build cache 被 N 个工作区共享,各自只写自己的增量。git worktree 做不到——每个 worktree 要么各装一份依赖,要么你手动 symlink。
- **大规模 ephemeral fanout**:秒级起几百个"共享同一只读基底"的环境,每个只占自己写的那点空间。

反过来,**git 的杀手场景 overlay 也给不了**(所以二者互补):**版本历史、`git diff`/`git blame`/`git merge`、PR 协作、跨机器同步**——这些是"语义/历史"层面的,overlay 完全不懂。

**问:** 为什么 overlay 不设置版本历史?难道历史上没有人做过类似的、把层叠/CoW 文件系统和版本历史结合起来的开源工具吗?

**⮑ 解答:** 两层意思:overlay 为什么不做历史,以及"层叠/CoW 文件系统 + 版本历史"的工具其实**做过很多**。

**(1) 为什么 overlay 自己不做版本历史(这是刻意的"职责分离")**
- overlay 是一个**挂载时的"视图合成器"**:它的全部职责就是"**此刻**把 lower+upper 叠成一个合并视图",不记录"随时间怎么变"。版本历史需要三样东西 overlay 都没有:① 用内容/id 标识的**不可变快照**;② 快照之间的**父子/血缘图**;③ 命名并在它们之间切换的能力。overlay 写文件时 copy-up 是**原地覆盖**,没有"commit"概念。
- **更重要的是:历史本就该由上一层做。** 在容器镜像里,**每一层都是内容寻址(sha256)、不可变、可堆叠的**——这本身就是"文件系统 diff 的线性/DAG 历史"。所以 "**overlay + 镜像管理器**" 合起来**确实有版本历史**(docker tag、layer digest、`docker history`),只是历史活在**镜像/层管理层**(OCI registry、containerd snapshotter),overlay 只负责挂载被选中的那一摞层。
- **为什么要拆成两个工具**:典型的关注点分离——**热路径(挂载/读)必须近原生且简单**,而历史/去重/GC 是慢的管理面工作。把它们揉进一个内核文件系统既危险又慢。

**(2) "层叠/CoW 文件系统 + 版本历史"历史上有人做吗?——非常多,这是一条丰富的谱系**
- **OSTree / libostree**:字面上就是"**给操作系统二进制做的 git**"——内容寻址的整棵文件树对象库、commit、branch、原子部署 + 回滚。Fedora Silverblue、Flatpak、Endless OS 在用。**这是"git-but-for-filesystem-trees"最像的一个。**
- **Btrfs / ZFS 快照**:文件系统**原生**的 CoW + 快照历史(`btrfs subvolume snapshot` / `zfs snapshot`),可回滚到任意快照——"CoW 文件系统 + 历史"天生合一。
- **NILFS2(Linux 主线)**:**日志结构、连续 checkpoint** 的文件系统,你可以**挂载过去任意一个 checkpoint**——等于文件系统自带时间机器。
- **Nix / NixOS**:内容寻址的 `/nix/store` + generations + 原子回滚。
- **OCI 容器镜像 / containerd snapshotter / Docker 层**:层叠 fs + 版本化 tag + 内容寻址 digest(就是上面说的那套)。
- **Fossil + Venti(Plan 9,Bell Labs ~2002)**:Venti 是内容寻址归档存储,Fossil 是其上的**快照文件系统**——"快照文件系统 + 内容寻址历史"的学术祖先。
- **研究型 versioning filesystems**:ext3cow、Elephant FS、WAFL(NetApp)——专门把"每次写都留历史版本"做进文件系统。
- **bup / borg / restic**:内容寻址去重备份(树的历史),用 git 式对象模型。
- **Dolt**:"**git for data**"(带 git 式版本/分支/merge 的 SQL 数据库),证明"git 语义"可被搬到非源码领域。
- **VFS for Git / Scalar(微软)**:虚拟化超大 git repo,不全量 checkout——和"别物化整棵树"的思路相关。

**(3) 结论 + 和 namei_ext 的关系**
- 所以**不是没人做**,而是大家发现**正确的拆法是"快的视图/CoW 层" + "单独的历史/快照管理层"**:overlay/Btrfs 管"廉价合成与读写",OSTree/镜像/Nix/Dolt 管"版本历史与回滚"。把历史塞进热路径文件系统既复杂又慢,所以被刻意分开。
- **同理,namei_ext 也不该自己做版本历史/快照/写**——这正是文档二/三的建议:namei_ext 只当"**可编程的路径视图原语**",把 fork/checkpoint/rollback 让给 DeltaBox / overlay-reflink / OSTree 这类系统,自己专注"按 workload/epoch 把名字解析到哪个后端"。"要不要给 namei_ext 加版本历史?"的答案和"overlay 为何不加"一样:**那是另一层的活。**

> **三层定位(回到正题)**:**git** 在 VCS 层管"提交历史与协作";**overlay** 在内核文件系统层管"任意目录树的分层共享与读写";**namei_ext** 在更下面的 VFS 解析层管"按 workload 把名字解析到哪个后端"。三者层次不同、可叠加使用(例如:git worktree 管源码分支 + overlay 共享依赖 + namei_ext 给 deps 提供按代视图),而非互相替代。

**B) git worktree**
- **场景**:一个 git repo,想同时在多个分支/任务上并行工作——每个 agent 占一个独立工作目录,互不干扰,各自能 commit/出 PR。Cursor 2.0 的 Agents、Devin 类工具就用它做并行 agent。
- **工作原理**:`git worktree add ../wt-task1 <branch>` 创建一个**独立的 checked-out 工作目录**,但**共享同一个 `.git` 对象库**(不复制 git 历史/对象);每个 worktree 有自己的 HEAD、index、工作区文件。rollback = `git reset --hard` 或直接删掉该 worktree。
- **用户怎么用**:`git worktree add <路径> <分支>`;agent 在自己的 worktree 里 edit/test;并行 N 个任务 = N 个 worktree。
- **我们 vs worktree**:worktree 在**应用 / VCS 层**,给的是完整可写独立 checkout + 分支/commit/PR 语义,是 agent 用户**已经期望、已经在用**的东西。它的代价:虽然共享 `.git` 对象,但**会把工作树源码文件 checkout 一整份**(每个 worktree 一棵真实文件树),而且 **deps(node_modules/.venv)它根本不管**——还是要每个 worktree 各装一遍或手动共享。namei_ext 在它**下面一层**(VFS 解析层),没有分支语义、没有写,但能**不复制地**为 deps/cache 提供共享或按版本的视图。所以两者其实不是替代关系,更像"worktree 管源码分支、namei_ext 管 deps/cache 视图"。

**问:** git worktree 和 git branch 在用法和原理上有何区别?

**⮑ 解答:** 一句话:**branch 是"一个指向某次提交的名字(指针)";worktree 是"一个真实摊在磁盘上的工作目录"。它们不是一回事,也不是替代关系——worktree 用来让多个 branch 同时被摊开。**

**原理区别**:
- **git branch** 极轻:一个 branch 就是 `.git/refs/heads/<名字>` 里的**一个 41 字节小文件**,内容是一个 commit 的 SHA。建一个 branch = 建一个指针,**不产生任何工作目录、不复制任何文件**。
- 一个普通 repo **只有一个工作目录 + 一个 index**。`git switch <branch>` 是把**这同一个目录里的文件就地换成**该 branch 的样子。所以**同一时刻只能有一个 branch"摊开在磁盘上"**;要切到另一个 branch,得先把当前改动 commit/stash 掉。
- **git worktree** 给你**额外的工作目录**:`git worktree add ../wt-dev dev` 新开一个目录、把 `dev` 分支 checkout 进去,而你的主目录还停在 `main`。每个 worktree 有**自己的工作文件 + 自己的 index + 自己的 HEAD**,但**共享同一个 `.git/objects`(历史只存一份)**。git 规定:**同一个 branch 同一时刻只能被一个 worktree 占用**。

```
普通 repo(只有 branch):
  /repo/   ← 唯一一个工作目录,一个 .git
    git switch main → 目录里的文件变成 main 的样子
    git switch dev  → 同一个目录,文件就地换成 dev 的样子(main 的不见了)
  ⇒ 同一时刻只有一个分支摊在磁盘上,只能串行切换

git worktree:
  /repo/      (主工作树, 在 main) ─┐
  /repo-dev/  (worktree, 在 dev)  ─┼─ 共享同一个 .git/objects(历史只存一份)
  /repo-fix/  (worktree, 在 fix)  ─┘
  ⇒ 三个分支同时摊在三个目录,可同时 build/test、互不干扰
```

**用法区别**:

| | git branch | git worktree |
|---|---|---|
| 命令 | `git branch foo` / `git switch foo` | `git worktree add <目录> <分支>` |
| 产生什么 | 一个指针(几十字节) | 一个真实工作目录(整棵源码 checkout) |
| 同时存在几份文件 | **1 份**(就地切换) | **N 份**(N 个目录并存) |
| 能否并行 build/test | 否(同一目录会打架) | 能(各自独立目录) |
| 代价 | ≈0 | 一次完整 checkout 的磁盘 + 时间(但 `.git` 历史共享) |

**关系与对 agent 的意义**:worktree **不是**取代 branch——你照样用 branch,worktree 只是让**多个 branch 同时被物化成可独立操作的目录**。并行 coding agent 正需要这个:每个 agent 要有自己的真实文件去 edit/build/test、互不干扰,而光靠 branch(一个目录就地切)做不到,所以用 worktree。

**和 namei_ext 的层次对照**(回到正题):branch 是**纯元数据指针**(便宜但不给你隔离的文件);worktree 是**物化的文件目录**(隔离但要付一次 checkout);namei_ext 在**更底下的 VFS 解析层**,既不是指针也不物化整棵树,而是"按 workload 把名字解析到不同后端"。三者管的东西不同:**branch 管"提交历史的名字",worktree 管"源码树的并行实例",namei_ext(设想)管"deps/cache 等路径的按代视图"**——可以叠加使用,而非互相替代。

> 一句话:overlay 和 worktree 都已经把"可写、近原生、毫秒级"的工作区 fork 做得很好。**namei_ext 不能在"可写 fork"上跟它们比,只能在它们结构上做不到的地方比**(下文第三部分)。

### 第二部分:FUSE 原理 vs 我们的原理 —— "比 FUSE 快"到底对不对?

**先把话说清楚:项目的人没说错,namei_ext 确实比他们测的那个 FUSE 快(p99 快 2.26x–99x),我并不否认这个数字。我质疑的不是"快不快",而是"该不该把『比 FUSE 快』当论文卖点"。** 这是两件事。

**FUSE 的原理(开销从哪来)**:FUSE 把文件系统逻辑放在**用户态 daemon**。一次 `open("/fuse/x/tool")` 的路径:内核 VFS → FUSE 内核模块把请求打包 → **唤醒用户态 daemon(上下文切换)** → daemon 处理(可能再去访问真实后端文件)→ 把回复拷回内核 → VFS 继续。**每个 lookup / getattr / readdir 都至少一次"用户↔内核往返 + 上下文切换 + 数据拷贝"。** 一个纯"重映射型"FUSE(把 `tool` 转发到 `tool.real`)即便逻辑极简,每个元数据 op 仍要付这趟往返。(`FUSE passthrough` 让**数据读写**直通下层、绕过 daemon,但**路径解析/元数据决策仍要进 daemon**。)

**namei_ext 的原理**:决策在**内核里**用 BPF 跑完,**没有用户态往返、没有上下文切换**;redirect 后内核用正常 VFS 机制查真实组件,数据读写本就走下层原生 page cache。所以对"纯重映射元数据 op",namei_ext 省掉的正是 FUSE 那趟往返——**这就是它比 FUSE 快的根本原因,也是这个数字真实可信的原因。**

**那我"似乎不这么认为"是什么意思?** 我说的是三个更细的点,它们和"比 FUSE 快"不矛盾:
1. **那个 FUSE 是个容易赢的对手**:它是朴素 libfuse low-level、没开 writeback cache、没用 passthrough、没多线程。要让"比 FUSE 快"在审稿人面前站得住,得比 **FUSE-passthrough / ExtFUSE / FUSE-BPF** 这些强 FUSE,而不是最朴素的那个。
2. **高端倍率是极端 regime 的产物**:99x 是在 **tmpfs 内存盘 + 热缓存 + 最 trivial 的纯元数据 op**(路径解析≈100% 的单 op 成本)这种最有利的条件下测的;唯一含真实 per-op 工作的 `exec` 只 2.26x。换个更真实的盘和负载,倍率会大幅缩水。
3. **(最关键)"比 FUSE 快"是错的靶子**:namei_ext 真正的竞争对手**不是 FUSE,而是 overlay/bind**——它们既比 FUSE 快、又**近乎原生**。而 namei_ext 现在**还慢于 native**(lookup 1.53x、readdir 4.37x),连自己定的 near-native 门禁都没过。也就是说它"比慢的(FUSE)赢,比快的(overlay/native)输",落在中间。**而 overlay 既比 FUSE 快、又比 namei_ext 快,还支持写**——所以"比 FUSE 快"根本不能证明 namei_ext 有存在价值。

**问:** 如果改进 namei_ext 的设计,能支持写操作吗?

**⮑ 解答:** 要先把"支持写"拆成三种,因为它们难度天差地别——一种**已经能**,一种**能补但有界**,一种**做了就等于变成 overlay(不建议)**。

**(1) 写一个"已存在的、被重定向到的目标"——现在就能,本来就是原生的。**
若 `tool` 重定向到已存在的 `tool.real`,你 `open("tool", O_WRONLY)` 再写:redirect 把 `tool` 解析到 `tool.real`,之后**正常 VFS 打开 `tool.real` 写,数据直接落到下层文件系统的原生 page cache**。所以"**对已存在后端的普通读写**"今天就工作——namei_ext 只改"解析到哪",不拦数据路径。**它不是"只读到骨子里",只是不做 create/隔离。**

**(2) create/unlink/rename through alias + 把新写引到"每沙箱私有后端"——能补,但有界,且对"已存在基底文件的首次写"会卡在 copy-up 上。**
- 现在缺的具体是:`LOOKUP_CREATE + REDIRECT` 直接返回 `-EOPNOTSUPP`(审查 C8:连 `gcc -o tool`、`> tool` 这种幂等写都被挡)。**这个可以放开**:让 open-for-create/write 时,把名字**重定向到本 cgroup 的一个私有后端目录**(相当于给每个工作负载一个"upper")。这样"**新建的名字 / 新写**"能被导向私有位置,不污染共享基底。
- **但"对一个已存在的基底文件第一次写"** 就必须先 **copy-up**(把基底那份拷进私有目录,再让后续解析指过去)。问题正是前面讲的:**copy-up 需要"在写路径上、做一次整文件拷贝 + 换 inode",而 namei_ext 的 hook 在 lookup(读解析)路径、且 BPF 做不了整文件拷贝**。namei_ext 顶多能在 open 时**触发**一个 copy-up,但"谁真正把文件拷过去、并保证所有写者(含 mmap)都落到私有副本"——这件事它自己干不了。

**(3) 完整可写 union + CoW 隔离(agent 真正想要的"写")——做了就等于重新造一个 overlay,不建议。**
"每个沙箱随便写、共享基底不变"= 需要 **upper 层 + copy-up + whiteout + 在写路径上兜住所有写者**。namei_ext 要支持它,只有两条路:
- **(a) 自己长出 overlay 那套机制**(upper、copy-up、whiteout)——**作用域爆炸,而且恰好重新发明了 overlay / fuse-bpf**,还把自己拖进 overlay/DeltaBox 的主场去硬碰(它们在那条轴上完胜);
- **(b) 叠在 overlay/reflink 之上**——让 overlay 管"写时拷",namei_ext 管"可编程的按组件解析"。

**结论 / 建议(与文档二、三一致)**:
- **别让 namei_ext 去"拥有写"。** 把**写 / fork / CoW 隔离让给 overlay-reflink / DeltaBox / OSTree** 这些**本就站在写路径上、为这件事而生**的系统;
- namei_ext 守住它**唯一能独占**的轴:**lookup 路径上的、可编程的、按组件的 per-task 视图**(redirect),数据读写继续走原生下层;
- 真要碰写,**最多走到 (2) 的"把新写引到私有后端 + 放开 O_CREAT 幂等写"**(这能修 C8、也够支撑"轻量写"场景),**但对已存在文件的 CoW 仍应靠下面叠的 overlay/reflink**,而不是让 lookup hook 去做 copy-up。
- 一句话:**namei_ext 改进后能支持"轻量的、引导到私有后端的写",但"完整可写 + CoW 隔离"应当组合 overlay 来做,而不是自己实现——否则它就不再是"窄的 VFS 路径策略",而变成了"又一个 overlay",丢掉了自己的差异化。**

**结论**:项目的人测的没错,但"比 FUSE 快"是个**真命题、却撑不起一篇顶会**。论文要赢的不是 FUSE,而是要找到一个**连 overlay/快照都做不到的能力**。这就引出第三部分。

### 第三部分:killer scenario 到底是什么

综合深度研究(文档二 Part C / 文档三),namei_ext 唯一"竞品结构上做不到"的格子是:

> **在超大规模、短命、并行的 fanout 下,同一个进程内(甚至同一地址空间的不同线程)持有各不相同的、零物化的路径视图。**

为什么只有它能做(这是个**结构性**而非"快一点"的差别):
- **overlay / bind / mount 的视图绑在 mount namespace 上**,而一个进程的**所有线程共享同一个 `mnt_ns`**(`task->nsproxy->mnt_ns`)。所以"同一进程里两个线程,让 `/deps` 解析到不同代际""同一 mount namespace 里一万个 task 各看不同视图"——这些 mount 类机制**在任何速度下都做不到**(不是慢,是根本表达不了)。
- 而且 mount/umount 的供给**串行在一把全局锁 `namespace_sem`** 上;上万个短命沙箱反复挂/卸会产生**悬崖式退化,拖垮整机**。
- **FUSE** 能做 per-caller 重定向,但每个元数据 op 付一趟用户态往返——慢一个数量级。
- **namei_ext** = per-task 的 BPF 决策 + O(1) 无锁"挂一段策略" + 原生元数据速度。**这是唯一同时给出"per-task 灵活 + 原生速度 + 无锁供给"的点。**

**一个能上主图的具体实验("一个进程,一万个工作区")**:一个 agent 编排进程开上万个线程级探索分支,每个线程从同一 pinned base 出发,但要让 `/repo`、`/deps` 解析到自己那一代的后端。用 overlay/mount:你画不出这个点(线程无法各持视图,且上万次 mount 卡在 `namespace_sem`);用 FUSE:画得出,但慢一个数量级;用 namei_ext:O(1)、无锁、原生速度。**这张"mount 根本画不出、FUSE 慢一截、namei_ext 平"的图,就是 killer scenario 的证据。**

**诚实提醒**:这个 killer scenario 是深度研究**推荐**的方向,不是现状——它需要先把"per-task/per-thread 视图绑定"和"RCU-safe 快路径"实现出来(见文档三的"投稿前差距清单")。当前 Phase-1 还没演示它。换句话说,**namei_ext 真正的价值不在"比 FUSE 快",而在"做 overlay/worktree 表达不了的 per-task 零物化视图"——但这一点目前还是设计方向,尚待实现与测量。**

**结论**:它是一个"**真实但尚未被钉死的需求**"——存在确凿的窄场景(fixture、超大 fanout setup),但"通用、非伪"的论证责任在论文身上:必须找到一个**只有可编程内核路径解析能廉价做、而 overlay/快照/worktree 都做不好的"杀手操作"**,并用端到端数据证明。这个"杀手操作存不存在、是什么",正是接下来"研究级设计方向"文档要回答的核心问题。

#### (C) 用户态文件系统(灵活,但有用户/内核穿越代价)

- **FUSE**:把文件系统逻辑搬到用户态。极其灵活,能表达任意路径重映射,是 namei_ext 唯一**功能等价的可编程基线**。代价:每次操作要在用户/内核间往返,且常常**重复实现 VFS 与下层文件系统已经做好的事**(目录遍历、权限、缓存)。经典量化见 *To FUSE or Not to FUSE*(FAST'17)。
- **FUSE passthrough**:近年内核支持让已打开文件的数据读写直接走下层,绕开用户态——缩小了**数据路径**开销,但路径解析决策仍在用户态。
- **ExtFUSE(ATC'19)**:用 eBPF 在内核侧为 FUSE 装"快捷路径",命中时不进用户态。**这是把 eBPF 放进文件系统请求路径的直接先例**,但它服务的是 FUSE 这个后端模型。
- **FUSE-BPF(Google/Android,Daniel Rosenberg)**:更激进——用 BPF 在内核里处理 FUSE 的路径操作/passthrough,大幅减少用户态往返。**这是 namei_ext 在"BPF 介入内核路径操作"上最危险的撞车候选**,作者自己的 related-work 清单里**遗漏了它**(已交由 deep-research 文献 agent 重点核查)。

#### (D) 访问控制 / 事件观测(只能 allow/deny/observe,不能 redirect)

- **LSM / SELinux / AppArmor**:强制访问控制,决定"能不能",不改变"解析到哪"。
- **Landlock**:非特权进程可自我施加的文件访问沙箱,基于路径的 allow/deny。
- **seccomp**:过滤系统调用,粒度在 syscall 层,看不到路径行走语义。
- **BPF-LSM**:把 eBPF 挂到 LSM hook 上做 MAC/审计。**与 namei_ext 在"内核里的 BPF 策略 hook"上同族**,但 BPF-LSM 的动作是 allow/deny,不是 redirect 路径解析结果。
- **fanotify / fsnotify**:观测甚至拦截(`FAN_DENY`)文件事件。能拒,不能重定向。

这一层的共同短板:**它们回答"是否允许/通知",而不回答"这个 workload 看到的路径树应该长什么样"**。namei_ext 的差异化正是:它改变**路径解析的结果对象**,而非访问权限。

#### (E) 强隔离沙箱运行时(粒度更粗、目标是安全边界)

- **gVisor**:用户态内核拦截 syscall,提供强隔离,代价是兼容性/性能开销。
- **Kata Containers / Firecracker microVM**:每个沙箱一个轻量 VM,隔离强、启动成本相对高。
- **容器 runtime(runc/crun)**:namespaces + cgroups + overlay 的组合。

agent 沙箱产品(E2B、Modal、Daytona、OpenAI Codex sandbox)就建立在这一层之上,并叠加 **snapshot/fork** 能力。namei_ext **明确声明自己不是这一层的替代**(不提供安全隔离),而是这一层之下、可被它们复用的一个 VFS 视图原语。

#### (F) 可编程内核扩展范式(namei_ext 的方法论来源)

- **eBPF**:在 verifier 保证安全终止的前提下,把用户提供的逻辑安全地运行在内核热路径上。
- **sched_ext(`SCHED_EXT`)**:让 BPF 程序定义 CPU 调度策略,内核保留调度机制与对象生命周期。这是 namei_ext 的**直接架构先例与命名来源**:
  ```
  sched_ext: kernel owns scheduler machinery   | BPF chooses scheduling policy
  namei_ext: kernel owns VFS machinery         | BPF chooses path-resolution policy
  ```

### 1.3 设计空间小结(namei_ext 自我宣称占据的格子)

| 维度 | bind/OverlayFS | symlink forest | FUSE | Landlock/BPF-LSM/fanotify | **namei_ext** |
|---|---|---|---|---|---|
| 数据路径 | 内核原生 | 内核原生 | 用户态(passthrough 部分缓解) | 内核原生 | **内核原生(下层 FS 直读)** |
| 路径解析策略 | 固定 | 固定 | 任意可编程 | 不改变解析结果 | **受限动作集上可编程(eBPF)** |
| 粒度 | 挂载点 | 链接 | 文件系统 | 访问决策 | **路径组件 / 每 cgroup** |
| 动态重配置 | 重 | 重(物化) | 中 | N/A | **轻(换/挂 BPF 程序)** |
| VFS 对象所有权 | 内核 | 内核 | 用户态 FS | 内核 | **内核(BPF 不持有 dentry/inode)** |
| 安全隔离 | 否(组合用) | 否 | 否 | **是(其本职)** | **明确不声称** |

namei_ext 的论证:**没有任何一行同时做到"内核原生数据路径 + 可编程解析 + 组件级/每-workload 粒度 + 轻量动态重配置 + 内核保留对象所有权"。这一格就是它声称的空白。**(这一"空白是否成立",尤其是对照 OverlayFS redirect_dir 与 FUSE-BPF,正是 deep-research 撞车检索要裁决的核心问题。)

## 2. namei_ext 是什么:核心设计

### 2.1 设计哲学

内核保留 VFS 的全部机制和对象所有权,BPF 只做一个窄决策。具体地,BPF **不**做以下任何一件事:

- 不实现 `inode_operations` / `file_operations`;
- 不创建或持有 dentry / inode;
- 不做递归路径行走;
- 不处理文件数据 read/write/mmap(数据路径留在下层文件系统,从而保留 page cache 行为,避免 FUSE 式数据路径开销)。

### 2.2 目标抽象

```
parent path + component name + operation context  ->  path-resolution decision
```

Phase 1 的动作集刻意极小:

- **`PASS`**:照常进行正常 VFS 解析。
- **`REDIRECT`**:把当前路径**组件**替换为策略选定的**同父目录**组件(例:lookup `tool` → 改查 `tool.real`),随后内核用正常 VFS 机制完成对替换后组件的查找、权限检查与文件操作。

> 注意"组件级、同父目录"这一关键约束:Phase 1 不做整路径字符串重写、不做跨目录重定向(那需要尚未实现的 "target registry")、不创建任何 VFS 对象。

### 2.3 它明确不是什么

设计文档反复强调的 non-goal:不是 BPF 文件系统、不是 mount namespace 替代、不是权限仲裁框架、不引入 YAML/JSON 策略语言(策略就是 `bpf/policies/*.bpf.c` 里的 eBPF 程序源码本身)。

## 3. 接口与 ABI

### 3.1 程序类型与挂载

```c
SEC("cgroup/namei_ext")
int namei_ext_policy(struct bpf_namei_ext_ctx *ctx);
```

- 新增 `BPF_PROG_TYPE_NAMEI_EXT`,要求 `expected_attach_type == BPF_CGROUP_NAMEI_EXT`。
- **cgroup 作用域**挂载(直接对应 build action / 容器 / serverless worker / agent 沙箱这种"每工作负载"边界)。
- Phase 1 **只允许一个** namei_ext 策略挂在一个 cgroup 决策点;多策略组合是 future work。`BPF_PROG_ATTACH` 不接受 attach flags,`BPF_LINK_CREATE` 被拒(因为通用 cgroup link 路径隐含 multi-attach 语义)。

### 3.2 决策上下文(固定大小)

```c
#define BPF_NAMEI_EXT_NAME_MAX 64

struct bpf_namei_ext_ctx {
    u32 event;                              // LOOKUP=0 / READDIR=1
    u32 flags;                              // lookup flags
    u32 name_len;
    u32 name_hash;
    u64 cgroup_id;
    u8  name[BPF_NAMEI_EXT_NAME_MAX];       // 当前组件名(只读)
    u32 redirect_name_len;                  // 以下为 BPF 可写输出
    u32 reserved;
    u8  redirect_name[BPF_NAMEI_EXT_NAME_MAX];
};
```

输入字段(event/flags/name_len/name_hash/cgroup_id/name)对 BPF 只读;redirect 输出字段可写。verifier 侧增加了对该 ctx 的访问规则,并把返回值约束在 `PASS..REDIRECT`。

### 3.3 动作集与事件语义

```c
enum bpf_namei_ext_action { BPF_NAMEI_EXT_PASS=0, BPF_NAMEI_EXT_REDIRECT=1 };
```

- **LOOKUP + REDIRECT**:内核改在同一父目录下查 `redirect_name`。
- **READDIR + REDIRECT**:把当前下层条目以 `redirect_name` 作为用户可见别名发射出去。

这使得 `open("/x/tool")` 与 `getdents64("/x")` 在 `tool` 是 `tool.real` 别名时保持一致(同一策略同时管 lookup 与目录枚举)。

### 3.4 redirect 输出校验(内核侧强制)

`redirect_name_len ∈ [1,64]`;不含 `/` 或 NUL;不为 `.` 或 `..`;`LOOKUP_CREATE + REDIRECT` 返回 `-EOPNOTSUPP`。

## 4. 内核实现(基于真实补丁源码核对)

> 本节是核对 `fs/namei_ext.c` 与 `include/linux/namei_ext.h` 真实代码后的描述。**有几处真实实现比 `docs/tmp/2026-06-13-kernel-code-survey.md` 的设计描述更简化**(survey 里设想过 `HIDE`/`DENY` 动作和带 `O_PATH` fd 的 cross-directory "redirect registry",但**实际落地的 Phase 1 只有 `PASS`/`REDIRECT` + 同父目录**)。下文以真实代码为准。

### 4.1 Hook 点

- **lookup / open 最终组件**:`fs/namei.c` 在 `walk_component()` 与 `open_last_lookups()` 调用 `namei_ext_lookup()`。若策略要求重定向,调用点临时把 `nd->last` 换成校验过的同父组件,返回前再恢复原 `qstr`。(为什么要两个点:`open()` 的最终组件不走 `walk_component()`,而走 `open_last_lookups()` 自己的逻辑,所以单点 hook 会漏掉普通 `openat` 的最后一段。)
- **readdir**:`fs/readdir.c` 在 cgroup static key 打开时,把 `iterate_dir()` 路由经 `namei_ext_iterate_dir()`。
- 策略执行体在 `fs/namei_ext.c`;cgroup runner 与 verifier 访问规则在 `kernel/bpf/cgroup.c` / `kernel/bpf/verifier.c`。

### 4.2 lookup 路径(真实逻辑)

`namei_ext_lookup()`:构造 `bpf_namei_ext_ctx` → 调 `__cgroup_bpf_run_namei_ext(&ctx)` →
- 返回 < 0:直接返回错误(fail);
- `PASS`:返回 0,照常解析;
- `REDIRECT`:若带 `LOOKUP_CREATE` 返回 `-EOPNOTSUPP`;校验 `redirect_name` 合法性(`namei_ext_redirect_valid`);用 `full_name_hash(parent->dentry, …)` 在**同一 parent** 下算出新组件的哈希,填入 `struct namei_ext_redirect`。这从代码层面坐实了"同父目录、单组件"的约束。

### 4.3 readdir 路径(真实逻辑,含两个值得注意的实现事实)

`namei_ext_iterate_dir()` 用一个包装 `dir_context`(actor 设为 `namei_ext_filldir`)包住调用方原始的 `dir_context`,让底层文件系统把条目发给包装 actor。`namei_ext_filldir()` 对**每一条目录条目**:

1. 用条目名构造 `qstr`、算 hash;
2. 构造 `bpf_namei_ext_ctx`,事件 `READDIR`,**对每条目运行一次 cgroup BPF 程序**;
3. `PASS` → 用原 actor 发射原名;`REDIRECT` → 校验后用原 actor 发射 `redirect_name` 作为可见名;非法/错误 → 记录 error 并停止迭代。

**实现事实 1(性能根因):** readdir 对每个 dirent 都做一次完整的 ctx 构造 + cgroup BPF dispatch。在小目录里,这个固定的每-条目开销相对原生 readdir 非常大——这正是实测 `readdir_alias_view` 出现 ~915ns vs ~209ns(4.37x native)的原因。

**实现事实 2(语义边界):** `namei_ext_filldir` 只对"**下层真实存在并被枚举出来的条目**"做内联改名,**并不会在迭代结束后追加任何合成别名条目**(survey 设想过的 "append synthetic aliases" 并未实现)。因此一个别名 `tool`→`tool.real` 之所以能在 `ls` 里出现,是因为下层有 `tool.real` 且被改名成 `tool` 发射出来。推论:
- 如果别名的后端在**别的目录**,readdir 根本看不到它(与跨目录 redirect 的缺失相互印证);
- 如果下层同时存在 `tool` 与 `tool.real`,READDIR 会把 `tool.real` 改名成 `tool` 发射,同时 `tool` 本身也照常发射 → **可能出现重复的 `tool` 条目**(功能测试里下层只有 `native` 与 `tool.real`、没有 `tool`,所以没暴露这个情况)。

### 4.4 上下文填充的真实细节(对"可编程性"的边界很关键)

`namei_ext_init_ctx()` 先 `memset(ctx, 0)`,然后只设置 `event / flags / name_len / name_hash` 并拷贝 `name`。它**没有填充 `cgroup_id`**(保持为 0),也**没有任何 parent 目录身份字段**。

由此得到一个对论文叙事很重要的事实:**Phase 1 的策略实际上只能基于 `(event, 组件名/哈希)` 做决策**。它无法区分同一个组件名出现在**哪个父目录**;在一个挂了策略的 cgroup 内,`redirect_alias` 对 `tool` 的重写会作用于该 cgroup 内**任意目录**里的 `tool`。OSDI 评估文档自己也把 "parent-specific rule" 列为 `view_redirect.bpf.c` 必须补的能力——即当前 ABI 的可编程性比"可编程路径解析"这个宏大叙事要窄。

### 4.5 静态分支与 RCU 处理

- 未挂策略时,`namei_ext_enabled()`(基于 `cgroup_bpf_enabled(CGROUP_NAMEI_EXT)`)走静态分支快路径,理论上零额外开销(残余开销需实测)。
- **RCU**:Phase 1 在策略生效且处于 `LOOKUP_RCU` 时,从 hook 点返回 `-ECHILD`,借用既有重试机制把这次 lookup 退化到 ref-walk。原因是策略需要稳定的 parent `struct path`、重定向目标是带引用的 `struct path`、且首个 PoC 想保持 verifier/生命周期规则简单。**代价:即使是不重定向的路径也被强制退出 RCU 快路径**——这是 `lookup_native_hot` 与 `pass_only` 都慢于 native 的主因。作者明确把"RCU-safe 的可编程 lookup"列为后续优化。

### 4.6 安全契约(作者声明)

- BPF 可选策略但不拥有 VFS 对象;redirect 输出由内核校验为单一组件;Phase 1 重定向非递归、同父;策略不得绕过下层文件系统本会施加的权限(权限检查仍在正常 VFS/下层路径上);BPF 失败/verifier 拒绝/运行时错误/超时时按挂载模式 fail-closed 或回退正常 VFS。

### 5.3 微基准:vs FUSE 强,vs native 未达门禁

强 FUSE 替代证据(p99,policy 相对项目自建 `fuse_redirect` libfuse low-level 基线的加速比):

| workload | policy/FUSE p99 加速 |
|---|---|
| `lookup_native_hot` | 51.07x |
| `lookup_tool_redirect` | 46.30x |
| `access_tool_redirect` | 50.05x |
| `open_tool_redirect` | 39.90x |
| `exec_tool_redirect` | 2.26x |

| workload | policy/native p99 | 备注 |
|---|---|---|

## 6. 论文叙事的演进:从"通用路径解析"到"agent sandbox 生命周期"

仓库的提交历史与 `docs/tmp/` 记录了一次清晰的叙事转向(2026-06):

### 6.1 转向后的三个顶层 use case

1. **Agent sandbox lifecycle**(主):为 coding agent 沙箱提供工作区 fork/fanout、checkpoint rollback/restore、deps/cache 路径视图。
2. **Service fixture sandbox**:config/secret/cert/endpoint 等 fixture 替换(W2 nginx 是当前最强证据)。
3. **Content-verified cache view**:ccache/BuildKit 类内容寻址缓存视图(W4,目前 mixed/negative)。

### 6.2 工作负载来源:确定性命令 trace,而非活体 LLM

为避免 LLM 不确定性,工作负载输入定为从 **SWE-bench / SWE-bench Pro** 等任务派生的**确定性命令 trace**(SWE-bench Verified 做 pilot,SWE-bench Pro public 731 任务做主集,TerminalWorld 做 control);namei_ext 的证据则来自命令重放时**自采的 file-operation trace**。核心 oracle 不是"agent 是否解出任务",而是确定性的沙箱状态行为(fork/rollback 后可见文件与哈希正确、无跨代混杂、lookup/readdir 一致等)。

### 6.3 明确的非主张(重要,影响审稿预期)

作者**主动**划掉了若干诱人但危险的主张:**不**声称安全隔离、**不**声称 eval/secret 隔离、**不**声称"让 agent 解出更多任务"、**不**声称替代完整沙箱 runtime、Phase 1 **不**支持 writable union(create/unlink/rename through alias)、**尚无**跨目录 redirect(需要 target registry + view_id/epoch 设计,属 future work)。


**问:** 为什么 namei_ext 无法解决上述这些"非主张"中列出的问题?特别是安全性方面:这是因为实现没做完,还是从设计上根本无法解决?审稿人会不会就此发起攻击?能否改进设计来解决这些问题?

**⮑ 解答:** 关键是先把这串"非主张"**分类**——它们不是同一种东西。有的是"实现没做完"(可补),有的是"设计上刻意取舍"(不该做),有的是"必须做、否则没贡献"。逐条判:

| 非主张 | 属于哪类 | 会被审稿人攻击吗 | 能否/该不该改进解决 |
|---|---|---|---|
| **安全/eval-secret 隔离** | **设计层面的取舍(不是没做完)** | **看 framing**:若保留"sandbox"措辞会被攻(质疑 C18);若改框架则不会 | **不该硬做隔离**(会变得不健全 + 撞 BPF-LSM)。正解=改框架 + fault-containment 命题(详见下文,这是本问重点) |
| **跨目录 redirect / target registry / view_id / epoch** | **实现没做完,且这是新颖点所在** | **会(致命 C1)**——只要没做出来,贡献就空 | **必须做**;文档二/三的 D4(RIT)/D5(带非环证书的跨目录)/D7(view-epoch)就是解法,设计上完全可解 |
| **writable union(create/unlink/rename through alias)** | **多为没做完 + 一点真实设计难点** | 轻度(C8:`O_CREAT` 幂等写被挡) | 可做但**不该在这里恋战**——写是 overlay/CoW 的强项,建议保持读多写少,把写让出去 |
| **替代完整 sandbox runtime** | 设计取舍(它是原语不是 runtime) | 不会(诚实 scope-out) | 不该做;定位成"可被 runtime 复用的原语" |
| **让 agent 解出更多任务** | 设计取舍(它是基础设施不是能力增强) | 不会(claim 它反而错) | 不该 claim |

### 重点:安全性到底是"没做完"还是"根本不可解"?

**裁决:它既不是"没做完",也不是"根本不可解"——而是"namei_ext 本质上不是干这个的,且硬要它干会把它弄坏"。** 三点说清楚:

1. **redirect ≠ confinement,是两个正交的轴。** namei_ext 做的是"**改变一个名字解析到哪个对象**",并**刻意保留下层文件系统的权限检查**(inode DAC)。而安全沙箱要的是"**限制不可信进程能碰到什么**"(不能逃逸、不能偷 secret)。"把 `tool` 解析成 `tool.real`"和"禁止进程读 `/etc/shadow`"是完全不同的操作——前者是重定向,后者是拒绝。所以"没提供隔离"不是 bug,是它根本没在做隔离这件事。

2. **当前设计若硬当安全边界,会变得"不健全(unsound)"——这是比"没做"更糟的状态。** 对抗性审查里两条相关质疑(文档二):
   - **C6**:`same-parent` 不变式只校验 redirect 的**名字(key)**,不校验**落地对象**——重定向到的 sibling 可以是 symlink/mountpoint,从而**重新跨目录/跨文件系统逃逸**。作为"视图"无所谓,作为"安全边界"就是个洞。
   - **C7**:redirect 反而**削弱**了一部分路径相关的安全准确性——audit 的 PATH 记录、AppArmor 基于路径名的规则、`/proc/self/exe`、`execve` 名字都会看到**后端 inode 而非请求的名字**。也就是说它对 path-based 安全是**负作用**,恰恰与"提供隔离"相反。
   → 所以"加上隔离"不是补一段代码,而是要先证明"策略无法被骗着重定向到不该去的地方",这比"把视图做好用"难得多。

3. **会不会被审稿人攻击?分两种情况(这点很重要)**:
   - **若论文保留"agent _sandbox_"标题、还引 E2B/gVisor/Firecracker(都卖隔离)做动机,却免责隔离 → 一定被攻**(质疑 C18:"你掏空了 sandbox 这个词,借了隔离产品的紧迫感却兑现不了")。这是**framing 漏洞**,很真实。
   - **若改框架成"per-task 路径视图原语"、把安全显式划出范围 → 不会因'没隔离'被攻**(这是诚实的 scope),审稿人转而问"那赢点是什么"——由 killer app + fault-containment 回答。

### 能否改进设计来解决?——能,但"解决"的正确含义是改框架 + 换正向命题,不是硬塞隔离

- **正解 A(推荐,低风险)**:**不**把 namei_ext 做成安全边界。把它和**已有的 confinement 机制(Landlock / namespaces / seccomp)组合**——它们管"禁止/限制",namei_ext 管"视图",两者正交可叠加。论文措辞从"sandbox"换成"path-view primitive"。这直接拆掉 C18。
- **正解 B(把弱点翻成正向命题,文档三 §5.4)**:门禁本来就不该卖"更安全",改卖 **fault-containment(故障影响半径)**:证明一个出错/恶意策略的**最坏后果被界定在 {fail-closed, sibling-misname}**——因为数据路径原生、static branch 保 no-policy 路径、verifier 界定每次调用终止;对照 FUSE daemon 可以 stall 内核、reclaim 死锁、阻塞 umount。**这是一个"它在可靠性上比 FUSE 强"的诚实正向 claim,而不是"它提供隔离"。**
- **正解 C(若真要碰安全,这是研究级、且是战略分叉,文档三 §5.2 / D3-9)**:把"内核保留全部权限检查"从 over-claim 升级成**精确定理**——① 校验**落地对象**而非只校验 key(关掉 C6);② 证明 **authority-preserving**:重定向后可达 inode 集合 ⊆ 进程在 native 下本可达集合(只保留、不放大权限);③ 恢复 path-name 完整性以修 audit/MAC(C7)。**但注意**:即便如此,它也只是"**不放大**权限",并**不主动 confine(拒绝)**;要真能拒,得加回 `DENY`/`HIDE` 动作(早期 kernel survey 设想过、后被砍),而一旦加 DENY 就**撞上 BPF-LSM/Landlock**,引来"为什么不直接用 BPF-LSM"。所以这是个**战略选择**,不是免费的赢。

### 一句话总结

> 安全隔离**不是"没做完",而是"不属于这个机制的职责"**;硬做会让它不健全、还撞 BPF-LSM。**最优解是改框架(不叫 sandbox、与 Landlock 等组合)+ 把可靠性翻成 fault-containment 正向命题**,而不是补一段隔离代码。真正"必须补、否则没贡献"的非主张是**跨目录/版本视图(C1)**——那个是实现问题、设计上可解,且正是论文的新颖点所在(文档三差距清单已列为投稿前第一优先级)。

## 7. 当前局限与未决问题(客观汇总,均有代码/文档出处)

1. **可编程性窄于叙事**:Phase 1 策略只能基于 `(event, 组件名)` 决策,无 parent 身份、`cgroup_id` 未填充;"per-workload 可编程路径解析"目前更接近"cgroup 内的全局组件改名"。
2. **跨目录与版本视图缺失**:真正支撑 build/package/agent 主场景的能力(target registry、cross-directory backing、view_id/epoch 原子切换/rollback)尚未实现,被列为 future work——而评估门禁(C2/C3)恰恰依赖它们。
3. **near-native 门禁未过**:RCU 退化 + 每-dirent dispatch 两处开销使非重定向路径也变慢;作者只能主张"优于 FUSE"。
4. **readdir 一致性是"附带的"而非"被保证的"**:别名可见性来源于下层后端被改名发射,既无合成条目追加,也可能产生重复条目;"lookup-set == readdir-set" 并非由内核构造性保证。
5. **性能比较的强基线尚缺**:当前主要赢在朴素 `fuse_redirect`;尚未对 FUSE passthrough / ExtFUSE / OverlayFS `redirect_dir` 等更强近邻做正面比较。
6. **宏观工作负载证据混合**:除 W2 外多为 mixed/negative;agent-sandbox 主线尚处"命令 trace pilot 先行"阶段,尚无端到端胜绩。
