# 绝佳场景到底是什么——以及合作者方向、git 瓶颈、怎么找到好问题

> 定稿日期:2026-07-10。本文回答用户的战略问题:绝佳场景是什么(要足够大、不是被现有技术挤压的窄缝)、合作者的两个场景是否合理有无漏洞、git blame/revert 在多 agent 并发下是否内核瓶颈、如何找到好问题。承接三轮红蓝(doc 17/18/19)与合作者 2026-07-09 提交(761a9eb)。**注意:第一节提出的"含写路径的内核分支视图"是候选方向,正在第四轮红蓝对抗中接受审查(结果将存 doc 22),未通过前不算立项。**

**直答：** 绝佳场景是"一台机器上同时运行成百上千个 AI agent 的并行尝试,每个尝试需要立即拿到同一个代码仓库的一份自己独立的工作副本——能独立读、独立写、随时整份丢弃或保留"。今天做这件事的每条现有技术路线都有结构性缺陷,不是调参能修的;而 namei_ext 如果把机制从只管读路径扩展到写路径,就能在内核层用微秒级操作建出这种分支,不复制文件、不挂载、不启动虚拟机,几千个分支共享同一份底座文件的内存缓存。这个场景骑在当下最大的计算趋势——agent 并行探索——上,规模足够大。

---

## 一、几句话说清楚给审稿人听

一台机器上同时跑成百上千个 AI agent 的并行尝试（best-of-N 即从 N 个独立尝试里选最好的一个；树搜索即把一个任务的解题过程展开成一棵决策树逐分支试；RL rollout 即强化学习训练时让 agent 反复执行任务收集轨迹），每个尝试需要立刻拿到同一个代码仓库的独立工作副本——能各自读、各自写、随时丢弃或保留。今天做这件事有四条路，每条都有结构性缺陷：

1. **复制文件**（cp 或 git worktree，即 git 用来在同一个仓库下同时检出多个分支的功能）：对 linux 这种约 8 万个文件的仓库，一次物化就要数秒，N 份就是 N 倍磁盘和内存。
2. **每个副本做一次挂载**（overlayfs，即 Linux 内核的联合文件系统，上层写、下层只读叠加呈现）：每次挂载操作在一把全局锁上排队，挂载表几千条后整机变慢。
3. **用户态文件系统**（FUSE，即 Filesystem in Userspace，让文件系统逻辑跑在用户态进程里而不是内核里）：读写每次操作要在内核和用户态之间往返，吞吐只有原生的 19%；打开 passthrough（一种让数据直接走内核路径的加速模式）后提升到 82%，但查文件名的元数据操作仍走用户态。
4. **每个尝试开一台微型虚拟机**（E2B 约 150ms 建一个沙箱、Morph <250ms 建一个分支）：各分支各自一个 guest 内核（虚拟机内部运行的操作系统内核），互相不共享文件缓存（page cache，即内核用来缓存磁盘文件内容的内存区域），不共享进程池。E2B 一年调用量涨了 375 倍。

我们的方案：把"这个进程看到哪份文件、它的写落到哪个目录"做成内核查文件名时的一条可编程决策——建一个分支等于往内核的表里写几条记录，微秒级完成，不挂载、不复制文件，读写全程原生速度，几千个分支共享同一份底座文件的内存缓存。

这个场景不是窄缝：DeltaBox 实测单个任务就有 2311 次恢复和 1689 次快照；OpenHands 官方支持 32 路并行；E2B 调用量一年涨 375 倍。上述四条现有路线的缺陷都是结构性的——复制的开销随文件数线性增长，挂载走全局锁加共享表，FUSE 的元数据操作必须跨用户态，虚拟机方案不共享内核因此也不共享缓存和进程池——没有哪条能靠调参消除。

## 二、诚实前提

**第一条：写路径是要新建的扩展，不是现有之物。** 现状的 namei_ext 机制只有 PASS（不干预，让路径解析按正常流程走）和 REDIRECT（把路径解析指向另一个已有文件）两种读方向的决策，外加一条受限的写意图规划（只在已重定向的父目录下、明确不做对共享文件的写时复制）。此前的大纲把完整的写语义列为推迟项。所以"内核原生分支视图"是要新建的扩展方向——具体新增：每个视图一个私有写目录，create/rename/unlink 这些写操作在被标记的子树下按视图落到私有目录，读时私有目录里的条目遮蔽底座（和 overlayfs 的上层目录遮蔽下层目录是同一个思路，但按 cgroup/task 选择而不需要每个视图做一次挂载）。工程和语义难点真实存在（rename 跨目录、硬链接、目录列举时合并两层条目），需要按既有规矩先过第一周的低成本实验和下一轮红蓝对抗再立项。

**第二条：端到端任务时长仍被 LLM 推理主导。** Amdahl 定律: AI agent 任务的大头时间花在等 LLM 生成回复上。所以这个场景卖的不是"单个任务端到端变快 X%"，而是"同一台机器能塞下多少并发分支、建分支多快、几千个分支之间共享了多少内存缓存"。

## 三、合作者的两个场景合理吗、有没有漏洞、之前的攻防讨论覆盖过吗

**合理之处：当作负载来源和正确性判定，选得对。** 合作者选的 AgentFS（一个用 SQLite 做后端、通过 FUSE 挂载的 agent 文件系统，外部项目，代码可复现）提供了分支、提交、遮蔽删除、缓存失效等一整套可执行的正确性判定。SWE-Factory-Gym 和 MEnvData 提供环境与缓存的判定。在之前三轮审查中也一直在用它们当负载来源。

**漏洞有两条：**

1. 合作者的框架是"正确性优先，证明这个窄钩子够用"，但整个框架没有性能闸门，也没有提出能力主张。审稿人一定会问"为什么不用 overlayfs 或 FUSE 做同一件事"，在这个框架里能给的回答只有"我们的钩子更窄"。而合作者自己在 7 月 9 日的第五轮自检文档里原话就写了："只说更窄太弱，必须说清窄买到了什么。"这个缺口他点出来了，但两个负载来源本身补不上——它们能证明"我们也能做对"，但证明不了"有一件事只有我们能做"。

2. 拿 AgentFS 这类系统当负载来衡量没有问题，但它们本身都是刻意选择用户态实现的（FUSE/SQLite，免 root、不改内核）。所以"给它们提速"或"替代它们"这种说法讲不通——只能把它们当负载和判定来源用。

**之前攻防覆盖过吗：覆盖了。** 第一轮红蓝就深入审查了 agent 工作区场景（裁决：可以建，但带第一周止损条件）和环境/缓存场景（裁决：辅助角色）。第二轮、第三轮把 agent 场景从主打降为动机与负载来源。结论始终是：这两个场景当负载来源合格，当论文身份不够。这正是与合作者框架的分歧所在——不是没有审查过，而是审查结论不同。

## 四、git blame / git revert 在多 agent 并发下是内核瓶颈吗

**不是，几乎不可能成为内核瓶颈。**

git blame 是在用户态沿提交历史逐版本做文本比对，时间花在解压历史对象和 diff 计算上，是 CPU 密集型操作。git revert 是生成反向补丁再写几个对象。两者的开销随仓库历史长度增长，不随同机并发 agent 数增长；它们不打路径解析，也不撞挂载表。100 个 agent 各跑各的 blame，只是 100 份用户态 CPU 负载，内核层没有共享瓶颈。

**在多 agent 并发下真正打到文件系统层的 git 操作是另外三个：**

1. **git status**：对每个被跟踪的文件做一次 lstat（读取文件元数据的系统调用），linux 仓库约 8 万个文件，一次 status 就是约 8 万次元数据系统调用。N 个 agent 各自的工作树互相冲刷内核的目录缓存。
2. **git checkout / git worktree add**：真实创建几万个文件，秒级，N 份就是 N 倍磁盘和缓存。
3. **N 个并发分支各需要一棵独立工作树这件事本身**。

**但要诚实：其中 stat 风暴已有成熟的用户态解决方案。** git 的 fsmonitor（文件系统监视器，git 2.37 起内置守护进程，此前用 Facebook 的 watchman——一个文件变更监听服务）可以监听文件变化，使 status 不再逐文件调用 lstat。微软和 Meta 对超大仓库造的 VFS-for-Git 和 EdenFS 也都在用户态层解决了这个问题。所以"git 某条命令在内核里成为瓶颈"这个提法本身站不住。

**合作者的直觉指对了地方，但具体操作指错了。** 真正没人解决、且正好落在我们这一层的，是"N 个并发分支不物化 N 份工作树、还能各自写"——这就是第一节说的那个场景。git 是它最好的负载发生器之一（每个分支跑 status、checkout、构建、测试），但 git 本身不是瓶颈。

## 五、怎么找到好问题（三轮攻防验证过的方法）

1. **从有名有姓的痛处出发**：谁今天在为这件事付钱、写事故报告。不要从机制能做什么出发。
2. **核对痛处所在的层和机制所在的层是否一致**：agent 分支的现实产品在虚拟机层，拿它开场就会把审稿人的注意力带去虚拟机方案——这是第三轮攻防的核心教训。
3. **检查"一行配置就能解"的近邻**：nginx 的 `$realpath_root` 一行配置就归零了我们一个实验方向；Bazel 8.3.0 默认开启 `reuse_sandbox_directories`（沙箱目录复用），废掉了一个原本有潜力的场景。任何候选问题先问一句：现有系统改一行配置能不能解决。
4. **趋势只能当"为什么是现在"的放大器，必须配上当下就成立的证据**：纯靠趋势撑起来的论述，连 LegoOS、unikernel 这些趋势型先例都不敢用。
5. **第一周做低成本实验来裁决**，通过阈值预先写死，失守就按预定动作降级，不恋战。

还有一条本轮新的认识：好问题不只能"找"，也能由扩展机制来创造。第一节的场景正是把机制往写路径伸一步之后才出现的；机制不动，就只剩"读视图供给"这个相对窄的问题。
