## 背景：namei\_ext 是什么、当前框架是什么

**namei\_ext 是什么。** Linux 内核里有一层叫 VFS（Virtual File System，虚拟文件系统），它是所有文件系统共用的抽象入口层——程序调用 open、read、write 时，请求先到 VFS，再由 VFS 转给 ext4 之类的具体文件系统。namei\_ext 在 VFS 的"名字解析"阶段插入了一个可编程的 eBPF 决策函数。eBPF（extended Berkeley Packet Filter）是 Linux 内核提供的一种"安全小程序"机制：你写一小段程序，挂到内核某个执行点上，内核先用一个叫 verifier（验证器）的检查器确认它不会死循环、不会崩溃，然后才让它高速运行——好处是接近原生速度，且不需要改内核源码。namei\_ext 同时管两件事：lookup（路径查找，就是内核把 `/a/b/c` 这样的路径名逐级解析到具体文件的过程）和 readdir（列目录，`ls` 命令背后用的就是它）。策略在内核取到文件对象之前就返回一个"路径解析动作"，它不实现文件系统方法，也不拥有 VFS 对象。

**当前框架（合作者提出的）。** 把"状态转移路径视图"定义为：当某个状态发生变化（工作区切换、分支切换、版本代际切换、请求状态切换），这个变化改变的只是"一个路径名解析到哪个已经存在的下层对象"或"这个路径名是否可见"。文件数据本身、写操作、权限、持久化、一致性，全部留给下层文件系统处理。

**先说值得肯定的地方。** 这个框架克制、诚实。idea-story 明确列出了非目标——不主张现有机制（查找表、FUSE）做不到同样的事，不主张替代 agent 文件系统、FUSE 或元数据服务，不做写冲突解决、不做数据路径中介、不做自定义持久化。下面的全部担忧**不是**说"框架吹过了"，恰恰相反：关切在于，**克制之后还剩多少能论证的证据**，以及几处硬伤能否在投稿前修好。

## 三条战略级担忧（本文档最核心的部分）

这三条决定的是"这个方向能不能成为一篇论文"，和具体实现层的问题不同——实现层的问题修好就行，战略层的问题决定的是修好了也值不值得。

### 战略担忧一：这个方向可能没有"杀手级应用"——这是最深的一条

**这是全篇最重要的一条。上一版几乎没写，但它才是所有担忧里最根本的。**

#### namei\_ext 真正与众不同的能力是什么

namei\_ext 声称的独特价值在于：它能在内核里**灵活编程、做丰富的文件操作动作**——不只是"允许"或"拒绝"（那是安全模块就能做的事），理论上还能做：

- TRANSFORM：在内核里就地转换文件内容（比如读到 A 文件时替换掉里面的某些行）
- INJECT：注入数据（比如读某个文件时插入额外内容）
- REDIRECT：重定向（把路径名 A 指向文件 B）
- CoW-divert：写时复制分流——"写时复制"是一种成熟的老技术，意思是写入时不改原件、而是把写入导向一份私有副本
- DELAY：延迟操作

这些丰富动作**相对于现有系统**，本该是 namei\_ext 独特的卖点。

#### 我们做了什么来寻找杀手级应用

我们跑了一个 17 个 AI agent 组成的对抗验证流程（workflow 编号 `wxfb8gc9r`，消耗 574k tokens），专门用来找"杀手级应用"。自己提出了 4 个候选方向，安排正方辩护、反方攻击、第三方再检验攻击是否成立、裁判跨候选打分：

- **K1：治理/信息流平面**——把可编程 VFS 当做 AI agent 的信息流控制执行层（控制哪个 agent 能读到哪些文件、信息往哪里流）
- **K2：可逆加可审计的世界边界**——给每个文件操作打上"这是 agent 第几步推理产生的"标签，做操作粒度的选择性撤销（相当于在系统调用级别做到类似 git blame 加 git revert 的效果）
- **K3：文件系统即上下文/成本管理层**——用可编程的 read 返回文件的精简版或骨架版，帮 agent 省下大模型推理消耗的 token
- **K4：零物化语义视图**——上千个 agent 共享只读底层，每个任务一份逻辑视图，零拷贝、不经 overlay/FUSE

**裁判结论：`the_killer = none`。4 个候选没有一个够格。**

#### "全部落选"揭示的深层问题：可编程 VFS 的动作空间两端都受限

这个"全部落选"的结果不是碰巧，背后有一个结构性原因。

eBPF verifier（验证器，也就是内核在加载 eBPF 程序之前的那个检查器）设有硬性限制：指令数有硬性上限（约 100 万条），循环必须能在编译时证明有终止（不能是无限循环）。后果是：扫描文件内容、跑正则表达式、改写一大段数据这类需要大量计算的逻辑，根本塞不进 eBPF，只能退到用户态去做——而退到用户态就慢了，因为每次操作都要在内核和用户态之间来回切换。

这使得 namei\_ext 可编程能力所能做的动作，两端都受到限制：

**简单的这一头：** 简单到能在 eBPF 里安全运行的动作——比如允许、拒绝、把路径重定向到一个已经存在的文件对象——**已经被现有系统覆盖了，不新**。具体来说：

- composefs 是内核里的只读镜像文件系统，多个容器可以共享同一份去重的底层数据、按需加载，不经 FUSE
- overlayfs 是 Linux 内核自带的联合文件系统，把只读层和可写层叠在一起，Docker 默认用它
- bpf-lsm 是用 eBPF 写的、挂在内核安全模块钩子上的策略程序，但只能"允许/拒绝"，不能改写内容
- CamFlow（SoCC'17 / CCS'18）是一个 Linux 全系统溯源系统，用内核安全模块钩子记录进程、文件、socket 之间的数据流，画出"谁读了谁、谁写了谁"的来龙去脉，只记录不做撤销

这些系统各自覆盖了"简单动作"的不同部分，且已经是成熟的、有社区维护的方案。

**丰富的这一头：** 足够丰富、足够新颖的动作——比如在内核里就地转换文件内容、注入数据、做内容脱敏——它们的计算复杂度**超过了 verifier 的上限**（光是一个通配符匹配就可能超标），只能退到用户态做。而退到用户态做文件系统，就变成了 FUSE（Filesystem in Userspace，在用户态实现文件系统的框架，每次文件操作都要在内核和用户态之间来回一趟，因此比内核原生慢）——**而 FUSE 恰恰是这个项目立项时要消灭的东西**。

**所以"既能在内核里高速运行、又足够丰富新颖"的动作集合，很可能是空的。**

#### 每个候选最终退化成的都是两种已有的固定机制之一

4 个候选方向走到最后，各自站住的那一小点，都退化到了两种早已存在的固定机制之一（它们都不需要"可编程"的丰富动作）：

- 要么退化成"允许/拒绝加一张标签表"——这就是 **P4Control（CCS'24，arXiv 2405.14970）** 用 eBPF 做到的，接近原生速度
- 要么退化成"持续记录日志加写时复制做版本化"——这就是 **CamFlow** 的内核溯源机制再加上一个版本化文件系统

#### 这一条对当前框架的直接含义

当前的"状态转移路径视图"框架，恰恰**退到了上面说的"简单动作"这一头**——PASS 和 REDIRECT 做的事只是"选择或隐藏一个已经存在的下层对象"。而这一头正是 composefs、overlayfs 已经覆盖的那一头。

所以：论文**不能**把立论停在"我们能正确表达这些路径视图"（因为那是已被覆盖的简单动作），而本该成为杀手卖点的"丰富动作新颖性"，恰恰是上面的探索（doc 08）查下来发现是空的那部分。

#### 两处必须诚实交代的部分反驳

上面的结论不是铁板一块。doc 08 自己修正了一半结论，这里必须如实交代：

**第一处：bpftime 打破了这两端限制中的一端。** bpftime 是 eunomia-bpf（和 namei\_ext 同源、你们自己组的项目）出的**用户态 eBPF 运行时**：把 eBPF 程序编译后在目标进程的用户态里运行，用它自己的用户态验证器，不受内核那条"约 100 万指令 / 必须有界循环"的上限约束；而且它是在进程内部拦截系统调用，比 FUSE 那种每次来回的方式快得多。所以前面说的"丰富内容变换只能退回到慢 FUSE"这条判定**不再完全成立**。

**但代价是换了一个问题**：bpftime 在用户态、按进程注入，它给不了"内核里任何新产生的进程都绕不过"的强制保证——没被注入的进程就逃过了拦截。所以对**安全类或强制类**的用途，仍然必须用内核钩子，而内核钩子只能跑简单动作——这一头的原结论仍然成立。到底哪一头是真正的问题，取决于用途要的是"丰富计算"还是"不可绕过的强制"。

**第二处：K2 的"投机执行"更锋利版本，是整轮探索里唯一一处 namei\_ext 的丰富动作可能真的用得上。** 想法是让 agent 不必在每个风险决策点停下来等人类确认，而是先投机地往前做（跑 build、跑数据库迁移、改一大片代码），内核侧把这一步的副作用隔离起来、按分支版本化记下来，做对了就提交、做错了就精确撤掉——和 CPU 的分支预测/投机执行是一一对应的。在这个用途里，namei\_ext 的 REDIRECT 动作第一次有了真正的事情可做：给每个投机分支一套"不物化的轻量私有视图"（如果用 overlayfs 给几千个短命分支各挂一套太重，而 namei\_ext 的按 cgroup 划分的、运行时把路径改写到分支私有写时复制对象的做法，恰好能轻量地做这件事；cgroup 是内核中用来分组管理进程的功能）。**仅就这一个用途**，部分推翻了"丰富动作一个都没用上"的结论。

**但这一处也有三个限制条件**：它需要实验证明 namei\_ext 的每分支视图真的比 overlay/composefs 的每分支挂载更省资源；它是另一组钩子（fork/exec/open/write），不是当前框架的 lookup 钩子；而且新意不能是机制本身——投机执行加不外泄加回滚这个想法已经被做过（Speculator，SOSP'05），事务化的 agent 工具调用也已经被做过（Atomix，arXiv 2602.14849），只有"内核强制加按分支轻量视图加可编程策略扣住不可逆操作"这三者的交集才可能是新的。

#### 真正的开放问题

doc 08 的核心遗留问题至今没有解决：能不能单独推导出一个用途，让丰富的数据面动作（TRANSFORM / INJECT / DELAY / CoW-divert）被真正用上，同时满足以下四条：

1. 在 VFS 钩子能拿到的上下文 `{event, name, hash, cgroup}`（事件类型、文件名、文件哈希、进程所属 cgroup）上，语义上是可执行的
2. AI agent 确实急需
3. 不被 composefs / overlayfs / bpf-lsm / CamFlow 已覆盖
4. 能通过 eBPF verifier 的检查

4 个候选都没能同时满足这四条。**这个缺口本身，才是真正的发现。**

### 战略担忧二：场景为什么窄——尤其在 agent 方向

框架做了前面说的那些克制之后，唯一还站得住的结构性缺口是：**给每个任务、每个线程一份私有的文件视图，且不需要为每个视图单独创建挂载点，在高密度场景下运行**。但这条缺口被以下几件事框得很窄。

#### (a) 密度是"容量"主张，不是"瓶颈"主张

我们**不说**"现有方案做不到一万个 agent 所以是瓶颈"。我们说的是：把 N 份独立的文件视图塞进一台机器，各种方案的资源成本曲线在不同 N 值处开始变得不可运行。

在 64GB 内存预算、linux 规模仓库下的崩溃点（**注意：全部是外推值，不是实测值，见下面 (c)**）：

- **cp 复制副本**：约 N=10 就崩（每复制一份就是一份完整的磁盘数据和内存占用）
- **git worktree**：约 N=11 就崩
- **虚拟机快照分叉**：约 N=11–18 就崩。虚拟机因为 MAP\_PRIVATE（内存映射的私有模式）初始时共享物理页面，但之后会单调地私有化——每台虚拟机的内存占用会涨到约 3.5GB，N=64 时需要约 224GB 内存，直接进入 swap（把内存数据倒腾到磁盘的缓慢操作），这已经是"不可运行"而不只是"慢"
- **overlayfs 和我们**：在读密集负载下约 N=760 才到达临界点

所以 N 在 **64 到 256** 这个区间时，恰好落在"复制路线已经扛不住、共享路线还能运行"的窗口里。

**但这里有一条关键限定**：不管 N 多大，全系统的瓶颈始终是 LLM 推理和进程侧的 fork/CRIU（CRIU 是 Checkpoint/Restore In Userspace，一个在用户态做进程检查点与恢复的工具）——**这恰恰是 namei\_ext 的名字解析层根本碰不到的**。这也正是论文不能作任何"任务变快"主张的原因。

#### (b) 对最强的共享基线 overlayfs，我们打不出吞吐差——这是机制决定的

overlayfs 和 namei\_ext 共享同一份底座页缓存（page cache 是内核把磁盘文件数据缓存在内存里用来加快读取的机制）。实测验证了这一点：512MB 文件首次读取 2.0 秒（受限于磁盘速度），经第二个挂载再读只要 0.08 秒（内存速度），而系统的 Cached 计数全程只增加了 104KB。这说明两个 overlayfs 挂载确实是从同一份内存缓存中读数据的。

所以在"数据缓存"这个维度上，我们和 overlayfs 是同一档次，**打不出聚合吞吐差距**。

我们能赢的只有"复制 N 份"的老办法（cp、git worktree、虚拟机快照）：这些方案创建的是同内容但不同 inode 的副本（inode 是文件在文件系统里的内部编号，可以理解为文件的身份证号），它们在页缓存里就是 N 份不去重的数据。实测 cp 一份 512MB 副本，清缓存后读回 1.95 秒（走磁盘而不是内存），Cached 净增约 1GB（相当于两份 512MB 分别占了一份缓存）。

而且这个赢面的大小是**存储带宽的函数**：

- 顶配本地 NVMe 盘（有效带宽 2–3GB/s）的场景下，只有 **1.2–1.7 倍**的差距——快盘把未命中缓存的读请求吸收掉了，优势被压缩
- 云盘档（AWS EBS gp3 基线 125MiB/s，已核实官方文档）的场景下，差 **8–16 倍**

overlayfs 唯一真实的内存弱点是它每个挂载各存一份元数据。实测每文件每挂载约 0.92KB 的元数据开销，N=1024 个挂载乘以 linux 树 8.5 万文件，约 80GB 的名义 slab（slab 是内核用来缓存小对象的内存分配器区域）占用——但 slab 是可回收的，这只算二阶效应，**只许在实验中测量呈现、不许当卖点**。

#### (c) 10⁴ 密度和所有崩溃点数字都是外推值，不是实测值

没有任何具名的生产部署跑过同机万级互异视图。新纪律（doc 24 §3 定的）：**未经实测的崩溃点数字从此不作论据**。

目前实测的只是机制常数（cp 副本每份多占约 1GB 内存、需 1.95 秒磁盘读回；双挂载 overlay 只多占 104KB、只需 0.08 秒），崩溃点是拿这些常数除以内存预算推算出来的。论文全文要在图上用浅色底纹把外推区标出来（表示这些是推算值、不是实测值）、把 10⁴ 标注为"作者提出的运行点"，不能混同于实测数据。分离点一律由密度实验（预注册行 4）的 N 扫描实测来定位。

#### (d) "每线程视图"这根支柱曾被绝对化表述，已改口

之前写过一个绝对化的说法——"线程之间做不到不同的挂载视图"。主 agent 在 5.15 内核上亲手实测**证伪**了这个说法：root 权限下同进程的线程 B 用 unshare(CLONE\_NEWNS)（CLONE\_NEWNS 是创建新挂载命名空间的标志；挂载命名空间决定进程能看到哪些挂载点）加挂 tmpfs 成功拿到了私有视图；无特权用户只要在多线程化之前先建好 user namespace 也能成功；只有"无特权 + 已多线程 + 没有提前建 user namespace"这个特定组合才吃 EPERM（权限拒绝错误）。

所以绝对化版本作废——留着等于在论文里埋一个审稿人用 20 行代码就能证伪的事实错误。

改口后的准确主张是**成本、粒度、语义**三条，全部可测量、量级很大：

**供给延迟随宿主挂载表条数线性增长**（copy\_mnt\_ns 是 O(M) 复杂度，M 是挂载表条数）：M=57 / 256 / 1024 时，单次 unshare 的 p50（中位数延迟）分别为 **47.8 / 184.7 / 794.3 微秒**。

**内存开销是线程数乘以挂载表条数**：1000 个线程乘以 1024 条挂载表，内核多消耗 **332.8MB 且不可回收**，和算术预测值 327.7MB 只差 2%。

**全局写锁 namespace\_sem 串行化**：namespace\_sem 是内核管理挂载命名空间时使用的全局信号量，同一时刻只有一个操作能持有它。千线程齐射时，一次无关的宿主挂载操作被卡到 **1.27 秒**（基线最大 5.9ms，劣化 **215 倍**）。

**语义绑定**：内核在源码层把"要私有挂载视图"和"放弃进程共享的当前目录/根目录"绑在一起——CLONE\_NEWNS 强制带上 CLONE\_FS（CLONE\_FS 控制线程间是否共享当前工作目录等文件系统状态）。任何对挂载命名空间本身的优化都救不了这个绑定关系，只有把名字视图和挂载解耦才能解决——而解耦正是 namei\_ext 做的事。这条与挂载表大小无关、与权限无关、与是否提前安排无关，是这根支柱里**最硬**的部分。

**诚实边界**：挂载表很小的宿主（M=35）上，成本论证偏弱（只有 47.8 微秒、11KB/线程）。此时需要靠语义绑定加上"部署期就决定了、运行期补救不了"两条来支撑。作为能力论文的**多根支柱之一**，敢押；作为**唯一的主要支柱**，不敢——它得和密度、供给吞吐两根支柱并排站才行。

#### (e) 不要偏移到"为 DeltaBox 提供服务"——那条路 DROP（不敢押论文）

三条独立的致命理由：

**① 闭源。** DeltaBox 的代码和实验环境至今未公开。README 原文："The artifacts ... are not currently public. We plan to release the code after publication." GitHub 搜索 0 个仓库。一个闭源系统不能作为论文的服务对象——审稿人无法验证、我们无法复现。

**② 阿姆达尔定律。** 文件侧占端到端时间 ≤0.2%（详见战略担忧三），把它优化到零也只快 0.2%。

**③ 层和需求双重对不上。** DeltaBox 的扇出吞吐从 N=1 的 1419.2 forks/s 塌到 N=64 的 165.9 forks/s，作者把下降归因于进程侧的内核页表拷贝（fork 的 p50 从 0.57ms 涨到 5.5ms），且作者原话 "The fork primitive is thus not the fan-out bottleneck at our target scales."（连他们自己都不认为这是要修的瓶颈）。这个成本在进程/内存侧，namei\_ext 的名字解析层根本碰不到。挂载侧则被他们自己的设计提前拿掉了热路径——DeltaFS 的切换方式是 "reconfigures the layer stack without unmounting"（重新配置层叠结构但不卸载挂载），不存在挂载可扩展性缺口可以"服务"。

DeltaBox 的正确用法只有三个，都不是研究问题：(i) 在相关工作里当相邻系统写（先写相似之处再写位置上的差异，不写"比它快"）；(ii) 用更正后的 v2 数字作动机引用；(iii) 当作"这个生态愿意改内核、愿意用分层文件视图"的一个具名例子——这条其实是替 namei\_ext 自己的故事说话，比"服务它"值钱。

### 战略担忧三：阿姆达尔定律——文件视图层不是这个场景的瓶颈

阿姆达尔定律是计算加速上限的定律：如果你优化的部分只占总时间的 1%，即使这部分变成零，整体也只快 1%。项目自己定了一个硬闸门：性能优化方面的主张必须证明被优化的是**全系统瓶颈**、且相对**最强基线**有显著的**全系统**收益；算不出或不到 10% 的，优化主张判死，只许改写成能力、容量或正确性方面的主张。

把负载分成三档据实讲：

#### 档 (a)：LLM 驱动的编码 agent（SWE-bench 那种，每步等大模型推理，秒级）

DeltaBox 论文 v2 原文："State management consumes 23–48% of total time on E2B (diff) vs. 1–2% on DeltaBox."

在做得好的系统 DeltaBox 里，全部沙箱状态管理只占端到端时间的 1–2%。而这 1–2% 里 namei\_ext 能碰到的文件系统那一小块，只是 0.07–0.25ms 的 overlay ioctl（overlayfs 的一次内核控制调用），约占端到端 **0.1–0.2%**。

把它清零，全系统收益上限 **0.2%**，比 10% 的判定线**低两个数量级**。

**结论：这一档的文件视图供给不是全系统瓶颈。任何"agent 任务变快 X%"的主张都死于阿姆达尔定律。论文里一个"任务变快"的字都不许写。在这一档上，用户此前的悲观判断完全正确。**

#### 档 (b)：内存/缓存受限的高密度区间——唯一能站住的正面论证

这一档证明的不是"我们比最强基线快"，而是"复制路线在这个区间扛不住了、共享视图是能不能在这个区间运行的最低门槛"。这就是上面战略担忧二 (a)(b) 讲的容量论证。

诚实结论：对"复制 N 份"的老办法（cp、git worktree、虚拟机快照），我们能站住（数值大且诚实）；对最强的共享基线 overlayfs，打平（因为共享同一份页缓存机制）。

#### 档 (c)：建立密集档（分支寿命短、换手率高）

每个分支的建立成本有数量级差距（cp 需要数秒 vs 我们需要微秒级），但只有分支寿命很短时建立成本在总时间里的占比才超过 10%。次要档。

#### 体裁定位建议

把论文正式定位为**能力与容量类型**的论文。先例：Bento（FAST'21）、lwC（OSDI'16）——这类能力论文不受阿姆达尔闸门约束，但必须在论文中明确写出不以提速为卖点。把密度、原子切换、每任务/每线程粒度等能力方面的主张写进主张清单，不要只停在正确性上。

#### 必须同步给合作者的数字更正

此前动机稿里用的 "47–77%" 在 v2 原文里找不到，应改为 "E2B (diff) 23–48% vs DeltaBox 1–2%"。此前引用的 Table 4 数字 14.57ms / 5.14ms / 1.12ms 是 v1 口径，v2 已修订为 overlay ioctl 0.07 / 0.19 / 0.25ms、agent 感知阻塞 0 / 1.86 / 9.29ms。

### 三条战略担忧串起来

经过第四轮终审、第五轮重扫之后不变的判词：**这个项目的前途在机制论文的执行质量，不在等一个"某个场景会把文件视图供给顶进全系统 10%"的机会出现。**

九个探索过的应用方向，死在**同样三条理由**下：

1. 阿姆达尔定律不过闸——被优化的部分占总时间太小
2. 一行配置就有的现成替代——overlayfs 挂载一下、bind mount 挂一下、改一行 nginx 配置就能做到同样的事
3. 痛点所在的层和机制所在的层对不上——比如瓶颈在进程 fork/内存分配层，而 namei\_ext 在名字解析层

这说明当下没有哪个负载能把文件视图供给推进全系统的 10%。这不是"再换个场景就行"的问题，而是要么把论文写成能力/容量体裁并把执行质量做到顶，要么去解那个真正的开放问题（战略担忧一末尾说的那四条同时满足的用途）。

## 框架本身的担忧

### F1 每个具体状态转移都要经得起"改一行配置/用现成机制就行"的质疑【必须在投稿前解决】

idea-story 列出了审稿人会质疑的方向——"用 FUSE 就行""用静态表就行"——回应策略是"展示真实的状态转移加自然基线"。更深的风险在于：对每一个被选中实现的具体状态转移，是否已经存在一行配置或一个现成机制能做到同样的事？

**相似的现成方案清单**（历轮讨论中，以下每一个都曾让某组实验失去价值）：

- overlayfs——每个视图各做一个挂载即可
- bind mount（把一个已有目录挂载到另一个位置，使同一份文件通过两个路径可达）
- K8s projected volume 加 AtomicWriter（用 rename 加符号链接做全局原子替换）
- nginx `$realpath_root`（把请求绑定在已解析出的真实路径上）
- git fsmonitor（消除对大量文件逐个调用 stat 检查状态的开销）
- Bazel 默认的沙箱复用
- Python `.pth`（约 529µs 就能给出等效的环境视图）
- casefold（已进 6.18 内核的大小写不敏感查找功能）

**我们找到的最强结构性论据**：粒度加生命周期管理——per-task/per-thread 视图、且不需要为每个视图单独创建挂载点。具体来说就是战略担忧二 (d) 里写的 CLONE\_NEWNS 强制 CLONE\_FS 那条：想要私有挂载视图就必须放弃线程间共享当前工作目录，这是不改内核绕不开的限制。

**建议**：让至少一个状态转移的基线对照表呈现**结构性的不可行**（不只是"更慢"），并在论文里清楚展示这个结构性失败。

### F2 新颖性是"位置新"不是"机制新"【需注意】

贡献在于：VFS 名字解析是内核主要子系统中最后一个还没有可编程 BPF 扩展点的。FUSE-BPF 的上游合入已停滞 27 个月，且被限制在 FUSE 挂载内部。"选在这个位置"的论据站得住，但有审稿人审美疲劳的风险（"又一个 \_ext"）。

**建议**：论文必须把贡献落在 VFS 名字解析这个位置**特有的技术难点**上——在 RCU-walk 无锁快路径中不退出就完成策略判断（RCU-walk 是内核解析路径名的无锁快路径，下面 I1 会详细解释）、跨视图的 dcache 一致性（dcache 是内核用来缓存目录项的内存结构，避免每次路径查找都访问磁盘；跨视图一致性指不同视图共享同一份缓存时如何各自看到正确内容）、readdir 的逆映射（当目录列表把名字 A 重定向到文件 B 时，列目录时需要从 B 反向找回 A）、挂载表与 namespace\_sem 的可伸缩性——而不是"我们在又一个地方加了个钩子"。当前 idea-story 的贡献列表（抽象、原型、刻画）没有把这些技术难点放到最前面。

### F3 "状态转移路径视图"是真正的独立子问题，还是对机制能力的事后包装【需注意】

主张清单里 C1（模型主张）的状态标注为"next"（尚未验证）。风险：这个抽象读起来像事后归类——把机制碰巧能做的东西打了个包。

**建议**：引用那些**独立地遇到了这个子问题、并为之投入大量工程量**的具名系统作为证据：DeltaFS 需要定制内核加 565 行 ioctl；TClone 改内核加 CRIU；YoloFS 写了 2.5kLoC 内核模块；Bazel 经历了五代沙箱方案；EdenFS/VFS-for-Git/CitC 各自从头造了一遍。写成"这个领域的项目反复为同一个子问题投入大量工程量"，但不要主张服务或替代它们。

### F4 DeltaFS/DeltaBox 只当支持动机的例子，不当服务对象【需注意】

idea-story 把 DeltaFS/IndexFS/TableFS 放在 appendix/related-work 位置——做法正确。两个提醒：

(1) 不要偏移到"为 DeltaBox 提供服务"——那条路 DROP 的三条理由见战略担忧二 (e)。

(2) 如果在动机部分引 DeltaBox 数字，要用更正后的 v2 版：E2B(diff) 23–48% vs DeltaBox 1–2%。旧的 "47–77%" 是 v1 版，不要再用。

## 应用场景层面

### S1 首个工作负载——AgentFS 派生的 agent 工作区【重要】

**(a) 不能主张端到端提速。** 阿姆达尔定律（见战略担忧三）。正确性判定标准通过没问题，但文件视图供给远小于 agent 单步时间的 1%。

**(b) 适配性风险。** AgentFS 本身有自己的语义，需要先确认它的 branch/checkpoint 语义确实经过名字解析（lookup/readdir），而不是走一个独立的对象模型。历史教训：BranchFS 刻意留在用户态 FUSE（Filesystem in Userspace，在用户态实现文件系统的框架，每次文件操作都要在内核和用户态之间来回一趟），它的 `branch()` 根本不碰路径解析，导致 namei\_ext 无法接入它。AgentFS 要避免同样的错配。

**建议**：第一周就核实 AgentFS 的分支/检查点操作是否落在 lookup/readdir 路径上。

**(c) 密度外推数据必须明确标注为外推，不是实测。** 见战略担忧二 (c)。

### S2 第二个工作负载——环境/缓存（SWE-Factory-Gym / MEnvData-SWE）【重要】

**(1) 可复现性风险。** 论文的硬性要求是 Make 一键、在 KVM（Linux 内核自带的虚拟化方案）虚拟机内可复现。Docker 构建波动、远程镜像访问、LLM/API 依赖都是对此的威胁（idea-story 自己也标了）。

**(2) 名字层归属问题——更关键。** 缓存的状态转移（陈旧/损坏/更新窗口）到底是不是靠名字级别的选择解决的？如果真正的修法是"重建/重拉镜像"，审稿人会看出这个状态转移的自然归属在别的层（缓存失效策略），这组实验就站不住。

**建议**：正确性判定标准要挑一个"名字级选择才是自然解法"的状态转移场景。

### S3 论据需加强的第三个负载——服务重载/密钥轮换【需注意】

nginx reload 场景里，`$realpath_root` 一行配置就能把请求绑定在已解析出的真实路径上；K8s AtomicWriter 用 rename 加符号链接做全局原子替换。所以重载/轮换这组实验必须展示这些全局机制给不了的**每任务/每代际独立的粒度**，否则又回到"用改一行配置就能做到同样事的现成方案就行"。

### S4 几个已知偏弱的场景（建议不再投入重点资源）【需注意】

**Bazel**：8.3.0 默认开启沙箱复用（reuse\_sandbox\_directories=true），官方中位沙箱开销 <5%，阿姆达尔闸门不过。只能当开销实测数据源或"生态反复重造"的佐证（五代沙箱演化：symlink → hardlink → sandbox-tmpdir → sandboxfs（FUSE 方案，已废弃）→ reuse\_sandbox\_directories，且 sandboxfs 在 Bazel 7.0.0 被移除归档），不能当重点实验场景或端到端主张来源。裁判判定 BACKUP。

**FaaS/serverless**：大众多租户隔离在 VM（Firecracker）/运行时（V8）层；只有同信任域平台（SOCK/OpenLambda——平台方拥有全部函数代码、函数之间不存在互相隔离需求的场景）受益。E2B 是反向证人——其博客明确说 overlayfs 对"共享只读底层加一层可写"完全够用、不需要改内核（"Scaling Firecracker: Using OverlayFS to Save Disk Space"，e2b.dev 2025-02-12）。E2B 的增长数字（4 万到 1500 万沙箱/月）只作规模趋势证据，工业现实性标注"趋势有据"、不许写成"今天具名"。裁判判定 ADOPT\_MODIFIED（BUILD 辅助场景，边界收紧）。

**casefold**：已进 6.18 内核，我们的差异化缩小到只剩"策略多样性"一个角色。摘要/导言不能出现其性能数字。

## 具体实现层面（最硬的一组）

### I1 RCU-safe PASS 问题——这一个环节解决不了，整篇论文就无法投出【必须在投稿前解决 · 第一优先】

**关切。** 只要挂上任何策略（哪怕什么都不做的 pass\_only）都会退出 RCU-walk。RCU-walk 是内核解析路径名时走的一条无锁快路径：绝大多数文件查找都走它，因为不需要获取锁、速度很快；一旦退出就要走加锁的慢路径。实测 pass\_only 开销达到 **1.71–1.77 倍**，而自设可接受上限是 **≤1.1 倍**，差距悬殊。两次修复尝试均为负结果。系统方向的审稿人第一个问题必然是"策略什么都不做时开销多少"，今天的诚实答案无法发表。

**idea-story 里没提它。** 当前 idea-story 写的投稿前必须通过的检查点是"缺一个 KVM 工作负载"——这低估了真正的障碍。应把 RCU-safe PASS 写成第一号投稿前必过检查点。

**可行修复方向（尚未尝试，必须投稿前落地并复测）。** 用 dentry 标志位控制是否调用 BPF。dentry 是内核中代表路径名某个组件的数据结构（路径 `/a/b/c` 中 a、b、c 各对应一个 dentry）。在 dentry 上设一个标记位（先例：`DCACHE_NEED_AUTOMOUNT`，内核现有的一个类似标记），未标记的路径组件永远不调 BPF，只有返回 REDIRECT 结果时才退出快路径。

项目纪律：RCU-safe PASS 是全项目第一优先修复，8/20 定生死。cache\_ext 报告的空操作开销 ≤1.7% 是论文会被对标的准入线。

### I2 表达力上限 vs 工作负载中留在模型内的行为够不够支撑论文【必须在投稿前解决】

**关切。** BPF 事件只有 LOOKUP 和 READDIR 两种；动作只有 PASS 和 REDIRECT 两种，限于同父目录、单路径组件、非递归；没有写路径（`LOOKUP_CREATE` 下 redirect 返回 `-EOPNOTSUPP`，即"操作不支持"）。框架诚实地把范围收窄到"选择或隐藏已存在的下层对象"，并把超出模型的操作标记为"不算成功"——这个自我约束很好。

**但问题在下游。** 诚实分类之后，两个选定工作负载里留在模型内的行为是否足够支撑一篇论文？agent 工作区分支的核心价值通常在**写入一侧**——可独立读写的副本、copy-up（第一次写入时把底层文件数据复制一份到私有层，之后的写入只改私有层的副本）、commit/abort——这些**全在模型外**。如果模型内只剩"提交前隐藏文件"加"选哪一代的底层文件"，审稿人可能觉得单薄。

**建议。** 第一周就把每个工作负载的"模型内 vs 模型外"行为占比量出来，作为一个廉价的、决定继续还是放弃的判断——比先把系统建完再发现留在模型内的行为太少要划算得多。

### I3 REDIRECT 的硬约束【需注意】

REDIRECT 限于同父目录、非递归、无跨目录注册表、名字缓冲 64 字节上限、不能创建可写别名。跨目录 rename 和注册表是"待建"功能；在建成前论文不能主张跨父目录的能力。如果某个自然的状态转移是"分支切换时把文件在目录间移动"，它在当前实现下出局。

### I4 readdir 每目录项调用一次 BPF 的开销【需注意】

readdir 对每个目录项都调一次 BPF。实测 readdir\_alias\_view 相对原生开销 **4.374 倍**、相对 FUSE **7.35 倍**。密度与性能的论述必须把这笔开销算进去；大目录会进一步放大它。

### I5 需要特权，不是免 root 运行【需注意】

加载策略走标准 cgroup-BPF 路径（cgroup-BPF 是内核把 BPF 程序挂到 cgroup 上的机制；cgroup 是内核用来分组管理进程的功能），需要 `CAP_BPF`/`CAP_SYS_ADMIN` 级别权限；无特权 BPF 自内核 5.16 起默认禁用。所以实际受众是持有内核加载特权的平台方，不是免 root 的终端用户。这是相对 BranchFS 的一个真实弱点——免 root 是 BranchFS 的主要优势之一。idea-story 的"目标受众/会场标尺"一节没写明这个范围限定。

### I6 证据成熟度是当前最大的时间瓶颈【重要】

仓库里 w1–w4 四个工作负载没有一个到"validated"（经过真实工作负载验证）状态，全部停在 functional\_only（只通过了功能正确性检查，没经过真实工作负载验证）。W3（checkpoint\_restore）实际上是 RDB replay，不是真正的 CRIU。所以当前证据是功能正确性层面的，不是"两个工作负载族通过"所需的、经真实工作负载验证的证据。主张清单里 C3 诚实标了"next"——这是决定论文成败的最大时间瓶颈，排期要留足。

### I7 测量卫生与性能实验是必做项【需注意】

当前全部数字都是在 5.15 内核的容器上跑的；任何数字进论文前必须在 6.12 及以上版本的裸机上复测，不得跨内核版本或跨虚拟化层画图。作为参照，cache\_ext（在类似位置做 BPF 扩展的项目）报告的空操作开销 ≤1.7%，这是论文会被对标的准入线。

性能必做五项（裁判确认后生效）：

1. 未挂零开销（对标 cache\_ext ≤1.7%）
2. pass\_only ≤1.1 倍（RCU-safe PASS，第一优先，8/20 定生死）
3. 密度四联曲线（N=1 到 10⁴ × 七基线，四指标：每视图内存 ≤100KB、挂载表零增长、建立 p99 平坦、聚合 lookup/s，外推点用浅色底纹在图上标出）
4. 策略多样性（R1–R6 六个策略族 ≥5 行）
5. 一个宏观负载无害性（≥20 reps）

## 收尾

**最关键的判断，按处理优先级：**

1. **战略担忧一 · 没有杀手级应用**——这是最深的一条。当前框架退到了"简单动作"那一头（选择/隐藏已存在对象），而那一头正是 composefs/overlayfs 已覆盖的；本该是杀手卖点的"丰富动作新颖性"，恰恰被 eBPF verifier 从两端同时限制住、查下来是空的。论文必须靠能力/容量体裁站住，不能靠"能表达路径视图"站住。
2. **I1 RCU-safe PASS**——第一优先修。今天 pass\_only 开销 1.71–1.77 倍无法发表，修复方向（dentry 标志位控制是否调用 BPF）尚未尝试，8/20 定生死。
3. **I2 表达力上限 vs 留在模型内的行为**——第一周量出来。诚实分类之后，每个工作负载里留在模型内的行为占比是多少？如果太薄，论文在完成所有工程之后仍站不住。
4. **F1 每个状态转移都要经得起"配一行现成的就行"的质疑**——至少一个状态转移要呈现结构性的不可行，不只是"更慢"。

**整体判断。** 当前框架克制、诚实、方向正确——内核可扩展点类型的论文站得住（先例：Bento FAST'21、lwC OSDI'16，都不在上游主线，lwC 在 FreeBSD，全部发表于顶级会议）。风险不在吹过头，而在两点：其一，战略层面——这个方向可能没有杀手级应用，克制之后模型内还剩多少能论证的证据；其二，执行层面——上述实现层的硬伤（尤其 RCU-safe PASS）能否在投稿截止前修好。前途在机制论文的执行质量，不在等一个突破性场景。
