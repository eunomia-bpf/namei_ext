## 二、我们先认错

在逐条核实之前，先把我们自己的误判诚实写清楚。

**误判一：旧文档的死刑判决是框架依赖的，我们从来没有在她的框架里审过。** doc 08、09、10 整条审查链都在问同一个问题——"namei_ext 能给 agent 什么不可替代的价值？"这是典型的"应用论文"框架（"X 论文"的意思是：贡献 = 你解决了一个具体应用的具体问题，比全部现有方案都好）。合作者换到的是"扩展点论文"框架（"扩展点论文"的意思是：贡献 = 你在操作系统的某个关键位置开了一个可编程的接口，让别人可以用这个接口灵活地实现各种策略，而不用修改内核源码）。这个框架的 related work 是 XRP（OSDI 2022 最佳论文，在 NVMe 存储栈里加 eBPF 可编程点）、cache_ext（SOSP 2025，哥伦比亚大学，在页缓存驱逐逻辑里加 eBPF 可编程点，改 6.6.8 内核）、extFUSE（ATC 2019，在 FUSE 内核驱动里嵌 eBPF 快速路径）、sched_ext（已进 Linux 6.12 主线，在 CPU 调度器里加 BPF 可编程点）。我们在 doc 02 到 10 里从来没有用这个框架系统性地审过 namei_ext。这不是小疏漏——是整个审查的评判标准选窄了。

**误判二：doc 08 的"便宜动作不新"在扩展性框架下是范畴错误。** doc 08 的核心判决之一是：namei_ext 在 eBPF 里能做的动作（PASS、REDIRECT）太简单、已被 Landlock 和 FUSE-BPF 覆盖，所以没有新意。但在扩展性框架下，这个论证反过来了——sched_ext 的每个调度决策也是一个极简单的动作（把一个任务分配到一个 CPU），"便宜"恰恰是扩展点的必要条件。sched_ext 的公式是：单个动作足够便宜 + 策略组合空间足够多样 = 扩展点有价值。按 doc 08 用的"动作便宜所以不新"这个标准去审，sched_ext、cache_ext、XRP 也全部会被毙掉。我们把"动作便宜"判成死因，属于拿应用论文的审查标准去否定一个扩展点论文的核心特征。

**误判三：doc 10 那轮唯一失败未产出的生成视角恰好是她的角度，我们当时说错了话。** doc 10 的 15-agent 工作流有一个视角（"内核内分支视图"）连续 5 次结构化输出重试都失败、没有产出。当时我们写的是"最弱、不影响"。但回看那条思路正是合作者现在提出的角度——用 namei_ext 直接在内核里提供分支视图，取代 FUSE 实现的 BranchFS。当时的"最弱、不影响"是轻描淡写，要收回。那条线失败不是因为方向弱，而是 agent 没有生成出来；我们把生成失败当成了方向弱的证据，这个归因是错的。

## 三、核实合作者的事实

反驳要认真对待，但反驳里的事实主张也要逐条核实。结论：全部基本属实。

**Crab（arXiv 2604.28138）：真实存在。** 用例包含"RL rollout branching, safe rollback"。合作者提到 Crab 有投机执行相关的 use case，记忆基本准确。

**cache_ext = SOSP 2025 真论文。** 哥伦比亚大学，用 eBPF 定制页缓存驱逐策略（页缓存是内核把频繁访问的文件内容留在内存里的机制；驱逐策略决定内存不够时先踢掉哪些缓存页），改 Linux 6.6.8 内核。这直接证明"扩展点论文"这个体裁在顶会是被接受的。

**我们补充核实的强证据（合作者没提但支持她的框架）：**

- **Google sandboxfs**（给 Bazel 封闭构建提供文件视图的 FUSE 实现）因性能已被废弃——Go 版不行，Rust 重写仍然不行。这说明 FUSE 做文件视图投影在工业实践中确实碰了性能天花板。
- **微软 VFS-for-Git/ProjFS、Meta EdenFS、Google CitC**——三大厂各自造过一遍虚拟文件树投影（虚拟文件树投影的意思是：让进程看到一棵文件树，但树里的大部分文件并不真的存在于本地磁盘，只在访问时才去远端取或按规则生成），全部基于 FUSE 或专有驱动。Linux 内核层面没有原生的、可编程的文件树投影机制。
- **gVisor**（Google 的容器沙箱）为了解决同一个缺口，迭代了四代机制：9P gofer → LISAFS → directfs → rootfs overlay。每一代都是在试不同的方式让沙箱里的进程看到宿主文件系统的子集，代代都受限于 FUSE 或自定义协议的开销。
- **FUSE-BPF** 活在 Android 内核树里（在 lookup 和 readdir 操作上加 BPF 前后过滤），但上游只合并了 passthrough 一半（Linux 6.9），BPF 那一半停在树外（LWN 915717、937433 确认）。

## 五、终审对三个核心问题的直接回答

### 问题一：namei_ext 有没有专属的、别人做不了的场景？

**有，但只有一个，而且是有条件成立的——条件全是还没测出来的数字。**

先把两个不同层面的问题分开：

**(1) "有没有只有 namei_ext 才能支撑的应用？"** doc 08/09/10 的答案（没有）依然正确，跨框架成立，不翻案。每一个我们检验过的具体应用（agent 沙箱、信息流控制、审计、token 精简），都能找到不依赖 namei_ext 的替代实现。

**(2) "有没有只有 namei_ext 才能占住的场景？"** 有。这个场景是：**同一台机器上一万个互不相同的文件视图**——Bazel 每个构建动作需要一个精确的输入文件视图、每个容器需要一个镜像视图、每个 agent 分支需要一个工作区视图——每个视图绑定一个 cgroup（cgroup 是 Linux 内核中把进程分组管理的机制，容器就是建立在 cgroup 之上的），不需要任何额外的挂载操作。

这个场景的专属性来自穷尽检查现有方案后发现的功能空缺：

- **FUSE-BPF** 只在 FUSE 挂载内部生效，每个视图仍然需要一个独立的挂载点和一个用户态守护进程，而且 BPF 那一半还没有进入 Linux 主线。
- **overlayfs / composefs** 是固定功能的文件系统，没有可编程接口，并且每个视图的单位是一个挂载。挂载表本身已经是内核社区公认的痛点——32 位挂载 ID 已经用满、listmount 和 statmount 系统调用因此被引入、挂载操作在全局锁 namespace_sem 上串行。一万个视图意味着一万个挂载，这正好撞上这个痛点。
- **Landlock**（Linux 的无特权沙箱机制，进程自己声明"我只能访问这些路径"）只能做允许或拒绝，不能做路径重定向。
- **ClearCase MVFS** 每个视图需要一个服务进程。

穷尽检索（arXiv、OSDI 2026 / FAST 2026 录用列表、内核邮件列表）未发现有人占住"原生文件系统名字解析可编程点 / 免挂载的 per-cgroup 文件视图"这个位置。

**问:** YoloFS 这篇论文讲了什么,和 namei_ext 有什么关系?Bazel 是什么,有什么用?在 namei_ext 中,相比 overlay 究竟有什么优势,如何说服审稿人不用 overlay?此外,还有两个问题尚待展开:"agent 未来几年什么能力会大规模增长?执行复杂 100 倍、智力 100 倍之后,什么会成为问题?"以及"git 是瓶颈吗?有可能未来变成瓶颈吗?即使现在端到端开销不大。"

#### 1. YoloFS 讲了什么，跟我们有什么关系

YoloFS 出自 arXiv 2604.13536，全称《Don't Let AI Agents YOLO Your Files: Shifting Information and Control to Filesystems for Agent Safety and Autonomy》。作者是 Shawn (Wanxiang) Zhong、Junxuan Liao、Andrea C. Arpaci-Dusseau、Remzi H. Arpaci-Dusseau（威斯康辛大学麦迪逊分校）、Jing Liu（微软研究院）、Mai Zheng（爱荷华州立大学）。这不是我们组，是存储系统领域公认的顶级组（Arpaci-Dusseau 夫妇）。

**它要解决的问题：** AI 编码 agent 直接在用户真实文件系统上操作，经常出事——损坏数据、删掉文件、泄露敏感信息。作者做了第一个系统性研究，分析了 2024–2026 年 13 个 agent 框架的 290 份公开事故报告。

**它的方案由三部分组成：**

- **Staging（暂存）：** 所有 agent 的文件改动先进一个暂存区，提交前不落到真实文件上。用户（或 agent 自身）可以审阅、撤销，相当于一道安全网。
- **Snapshots（快照）：** 把"给当前状态拍一张快照"的能力暴露给 agent，让它自己发现做错了之后能回滚到之前的状态。
- **Progressive permission（渐进授权）：** 在访问时才逐步授权，用最少的打扰取代一次性全量授权。

实验结果：11 个含有隐藏副作用的任务中，8 个 agent 能自我发现并纠正错误，且所有副作用都停在暂存区可供审阅；112 个日常任务中成功率持平，但需要用户手动干预的次数更少（0.4 次，对比 Claude Code 的 0.9 次、Copilot 的 1.3 次）。

**实现层（对我们最关键的部分）：** YoloFS 是一个 Linux 可堆叠文件系统内核模块（stackable filesystem——一种叠在真实文件系统之上、在 VFS 层拦截每一次文件操作的内核模块），代码量 2.5 kLoC C（内核模块）+ 6.2 kLoC Rust（用户态命令行工具）。它不是 FUSE（用户态文件系统），不是 OverlayFS，不是 eBPF。暂存机制的实质是：一个扁平文件仓（改动的文件按 inode 号存进去）加一棵内存中的覆盖树（挂在 VFS dentry 上，记录路径到内容的映射）加一个只追加的目录日志。作用单位是"每个挂载"，不是 per-cgroup。需要安装内核模块，只支持 Linux。

**它和我们的关系有两面：**

**对我们有利的一面（验证）。** YoloFS 是一个独立的顶级组给出的证据，证明"给 agent 做文件控制，正确的层是 VFS/内核层，而不是 FUSE 或 OverlayFS"。而且它自己的实验里给了我们两个极有价值的数据点：OverlayFS 建不了超过 50 个快照——原因是挂载选项字符串长度超过了内核上限；BranchFS（一个 FUSE 方案）慢 20 倍以上——给一个 20 秒的构建任务硬加了 2 分多钟。这两条数据直接印证了一个关键判断：基于挂载或 FUSE 的方案在规模上会遇到结构性的失败。

**对我们构成压力的一面（竞争）。** YoloFS 现在是我们最近的邻居——都在 VFS 层做 agent 文件安全。在 related work 里必须和它正面比较（与 FUSE-BPF 并列为头条参照物）。差异化恰恰在于：YoloFS 把 staging/snapshot/permission 这一套具体策略写死在了一个 2.5 kLoC 的 C 内核模块里。这恰好印证了 namei_ext 的立项动机——每次有人要给 agent 加一个新的文件策略，就又得写一个内核模块、推动上游合并。我们主张的是给出一个安全的、可编程的扩展点（namei_ext），让 YoloFS 那样的"暂存"成为众多能被表达出来的策略之一（还有 Bazel 输入视图、composefs 布局、大小写兼容层等等），而不必每加一个策略就写一个 C 模块。此外，YoloFS 的作用粒度是 per-mount、面向安全控制，不处理"同一台机器上万个各不相同的视图"这个密度场景。

#### 2. Bazel 是什么，有什么用

Bazel 是 Google 开源的大型构建系统——也就是一个把源码编译成可执行文件或库的工具，功能类似 make，但专门为超大代码库设计。

Bazel 的招牌特性是 hermetic build（封闭构建）：每一个构建动作（比如"编译这个 .c 文件"）都在一个沙箱里运行，这个沙箱里只放得下这个动作显式声明过的输入文件，别的一概看不见。这样做的好处是构建结果可复现、可缓存——相同的输入必然得到相同的输出，所以中间产物可以跨机器共享，不必重复编译。

问题在于：为了让每个动作只看到自己声明的输入，Bazel 必须为每个动作现场搭一棵"只含这些输入"的文件树。传统做法是搭一片符号链接森林（把需要的文件一个个链接进沙箱目录），或者用 sandboxfs（一个基于 FUSE 的虚拟文件系统，按需只呈现声明过的文件）——但 sandboxfs 因为性能问题已被 Google 废弃。

这正是本文用例表第 1 行的来由：每个构建动作需要一个免挂载的输入视图。一次大型构建可能有成千上万个构建动作同时进行，每个都需要各自不同的文件视图——这是 namei_ext 密度场景的现实需求之一。

#### 3. 相比 OverlayFS 我们究竟有什么优势

先说清 OverlayFS 是什么：它是 Linux 内核自带的叠加文件系统，在一个只读底层上叠一个可写层，容器镜像就用它来组合多层为一个统一的文件树。它是内核内实现，非常成熟，单个视图的性能很好。

**诚实的前提：单个视图的场景下，我们不主张比 OverlayFS 快——它又快又成熟，在那个场景里我们赢不了，也不应该声称能赢。**

说服审稿人的理由只有两条，而且都必须靠数据支撑：

**第一，密度。** OverlayFS 的单位是"一个挂载"——每创建一个 overlay 视图就要做一次 mount 操作。到了大规模就会撞上内核的硬性限制。这不是我们的推测：YoloFS（arXiv 2604.13536）自己的实验就测出 OverlayFS 建不了超过 50 个快照，原因是挂载选项字符串超过了内核长度上限。此外，挂载表本身是内核公认的痛点——32 位挂载 ID 会被用满；listmount/statmount 这两个新系统调用就是为了缓解挂载表管理问题而引入的；挂载操作在全局锁 namespace_sem 上串行化（意思是所有挂载/卸载操作必须排队逐个完成，不能并行）。我们的场景是同机上万个各不相同的视图，每个视图只是几条 eBPF map 条目、零挂载操作，OverlayFS 的结构做不到这一点。

**第二，可编程性。** OverlayFS 是固定功能的——它能做的事（上下层叠加、写时复制）是写死的，表达不了"按一份规则清单把某些文件名重定向到别的对象"这类灵活策略。每加一个新用途（Bazel 输入视图、composefs 布局、大小写兼容），OverlayFS 路线就得再打一个一次性的内核补丁（OverlayFS 历史上的 redirect_dir、metacopy 等特性就是这样一个个长出来的）。我们给的是一个可编程扩展点，不需要为每个新策略修改内核代码。

**同样要诚实说的：** 这两条优势成立有一个前提——四个还没测出来的数字必须过关（尤其是"挂了策略的路径查找延迟要与 OverlayFS 同量级"这一条，见本文第五节列出的四个条件）。数字没打出来之前，"不用 OverlayFS"只是一个待验证的主张，不是已确立的结论。

#### 4. 合作者的第 4 问和第 6 问

**第 4 问：agent 未来几年什么特性大规模增长？执行复杂 100 倍、智力 100 倍，什么会成为问题？**

未来几年会大规模增长的 agent 特性包括：并行探索（树搜索、best-of-N 采样、子 agent 扇出）、更多 shell-out 子进程、更长时间的自治运行、对更大代码库的操作。当执行复杂度和智力同时放大 100 倍，首先在文件系统/操作系统层会断裂的地方有四处：

- **视图和沙箱的密度。** 今天 1 个 agent 对应 1 个视图，一切运转正常。100 倍的并行分支意味着同一台机器上同时存在成千上万个各不相同的视图。基于挂载的方案（OverlayFS 或 FUSE 每个视图都要一次 mount）在这里会遇到结构性的失败——YoloFS 测出的 OverlayFS 50 个快照上限和挂载表的种种限制就是先兆。这个密度场景本身就是"100x 之后什么会断"的答案，也正是 namei_ext 唯一的专属场景。

- **fork 和页表拷贝的开销。** 大量分支靠 fork（派生子进程）来创建，宿主进程的页表拷贝和进程内存 checkpoint 是主要开销。之前分析 CXLfork/DeltaBox 那条线的结论是：进程状态的保存/恢复比文件状态贵 6–12 倍。

- **页缓存互相驱逐。** 成百上千个 agent 共享一台机器时，彼此会把对方的缓存页从内存中挤出去（页缓存是 Linux 把磁盘文件内容缓存在内存中以加速读取的机制），导致所有 agent 都变慢。这正是 cache_ext 要解决的问题。

- **dcache/inode 缓存污染和路径解析变热。** 每个 agent 都在一棵大代码库里做文件操作，目录缓存（dcache，内核缓存"目录名到 inode 的映射"以加速路径查找）和 inode 缓存（缓存文件元数据）会被大量不同的路径冲刷，路径解析本身成为性能热点。

一句话概括："100x"在文件系统层首先表现为"又多又各不相同的视图"，而这正是我们押注的场景。

**第 6 问：git 是瓶颈吗？有可能未来变成瓶颈吗？即使现在端到端开销不大。**

合作者的直觉大概率是对的，但 git 成为瓶颈的机制不是"单条 git 命令慢"，而是"分支数量乘以代码库大小"。

今天，git 的 checkout/status/clone 在 agent 端到端耗时里不显眼，因为大头是 LLM 推理——每步推理就要好几秒，git 操作相比之下不起眼。

未来会成为瓶颈的路径是这样的：agent 如果跑 100 倍的投机分支（speculative branching，同时尝试多种方案看哪个最好），每个分支按传统做法都需要一棵独立的工作树（git worktree 或 checkout）。N 份大代码库的完整检出意味着磁盘空间和检出时间同时爆炸。这不是假设——微软正是因为大代码库的 checkout/status 太慢，才造了 VFS-for-Git 和 ProjFS；Meta 造了 EdenFS——两者的思路都是把工作树变成"虚拟"的：文件不在 checkout 时全部复制出来，而是在被访问时才按需从后端物化（lazy materialization）。agent 把并发分支放大 100 倍，需要的正是"EdenFS 式的虚拟检出，但要撑上千个临时分支"——这就是本文用例表第 2 行，也是 namei_ext 密度场景的一个有说服力的用途。

诚实的前提同样要说：这条推论和整条主线共用同一个需要先测出来的数字——namei_ext 在大量并发视图下的实际路径查找性能。在那个数字确认之前，"git 在 100x 场景下成为瓶颈、namei_ext 能解"也只是一个合理的预判，不是已验证的结论。


**但这个专属场景要成立，以下四个条件缺一即死：**

**(a) 第一周实测门。** 挂策略的路径查找性能要赢 FUSE 的各种配置（含 6.9 passthrough、6.14 io_uring）2 倍以上，并且与 overlayfs 在同一量级。如果输了，审稿人一句"用 overlayfs 就行"就足以否决。

**(b) 每视图内核增量内存近零 + 挂载表零增长。** 这个承诺必须在真实实现中保住，而不只是设计文档里的断言。

**(c) 动工前给出"被标记子树在 RCU-walk 里可见"的具体设计。** 这是全项目最大的技术未知。RCU-walk 是内核路径查找的无锁快速通道——绝大部分路径查找都走这条快路径，不需要获取任何锁，所以非常快。挂策略意味着路径查找遇到被标记的目录时需要做额外判断，这会不会把查找踢出快路径、掉进需要加锁的慢路径？dentry 标志里有 DCACHE_NEED_AUTOMOUNT 这类"遇到标记才走慢路径"的先例，说明方向可行，但目前只有断言没有设计。走不通的话，条件 (a) 就永远打不出来。

**(d) 开销普查论文（见下文候选 2）的结果必须显示：** FUSE 在最新版本上提速之后，元数据和视图建立开销仍然占沙箱总开销的大头（>50%），并且从 1 个视图扩展到 10,000 个视图时，现有方案的开销是超线性增长的（即视图数翻倍、开销翻不止一倍）。否则整条主线的动机不成立。

**时间窗口。** 2026 年 2-4 月上游正在给 famfs 和 iomap 讨论 FUSE-BPF 式钩子（LWN 1068686），"BPF 进文件系统"的设计空间正在被上游逐步填充。如果不抢在上游的设计定型之前拿出方案和数据，窗口会关闭。

**一句话总结这个场景：** 专属场景在结构上存在（穷尽检查没有人占住这个位置）、在证据上还不存在（四个条件全是未测数字）。它是一个"密度乘以性能"的场景——不是任何单一应用，而是"一万个视图在同一台机器上共存"这件事本身。

### 问题二：合作者的框架站不站得住？

**体裁选择站得住。经过对抗验证，把 namei_ext 当"扩展点论文"来写是成立的。**

扩展点论文的贡献公式是：可编程接口的位置 + 安全契约（保证挂上去的策略不会破坏系统）+ 未挂策略时零开销 + 策略多样性的证据（证明这个接口确实能支撑多种不同用途）。cache_ext（SOSP 2025）证明了这个体裁连"策略全是论文作者自己写的"都能被顶会接受——不需要等待外部开发者在你的可编程接口上写出真实产品。

更重要的是：**我们旧文档三条死刑判决里的两条半，正是因为拿"应用论文"的标准审"扩展点论文"造成的误判。是她的框架把项目从误判里救了出来，这要如实承认。**

**但她有两个具体说法需要修正：**

**(1) related work 锚点不完整。** 她提到的 XRP / cache_ext / extFUSE 是血统先例（同一种类型的论文），但不是最近的直接竞争者。namei_ext 最近的撞车对象是 **FUSE-BPF**——它和 namei_ext 做的事最像（在文件系统操作上加 BPF 钩子），必须放在 related work 的头条正面对比。还需要加上 composefs、overlayfs redirect_dir、Landlock、ClearCase MVFS。extFUSE 只是同一脉络的早期工作，不构成威胁。

**(2) "比改内核 safe 即可"不够。** 安全性（不需要修改内核源码、eBPF 验证器保证策略不会破坏系统）是准入条件，不是贡献本身。每一篇被接受的扩展点论文都同时交付了三样东西：未挂策略时零开销、热路径上打赢现有的固定功能方案、真实的策略多样性证据。namei_ext 还背着一个 XRP / cache_ext / sched_ext 都没有的额外负担——挂策略会让路径查找退出 RCU-walk（无锁快速通道），而现有原型对此的处理方式是整体放弃快路径。这正是 namei_ext 目前过不了自己性能门禁的根本原因。按"safe 即可"去写，审稿人问"为什么不用 overlayfs"就没有答案了。

**结论：她的框架方向对，实质性地纠正了我们的误判。"safe 即可"这个具体说法错了——按那个标准写，会死在性能审查。**

### 问题三：旧文档哪些判决要修正、哪些依然成立？

逐条过：

**(1) doc 02："核心能力已被 FUSE-BPF 确立、新颖点只在打包。"= 半个误判。** 事实部分对（经 LWN 核实确认），但把"打包"当贬义词是体裁错误——扩展点论文的贡献恰恰就是钩子放在哪、安全契约怎么定、开销能不能归零、能撑多少种不同策略。doc 02 自己的 Part C（D1 密度 / D3 慢路径 RCU 安全 / D7 代数号）正好是新主线的建造清单。但 doc 02 的执行债判决（什么都没有实现、零宏观证据）依然成立，从"判定"转为"建造义务"——欠的东西一条没少，只是从"项目已死"变成了"项目要做这些才能活"。

**(2) doc 07："窄。"= 框架依赖的误判。** 当时是在 agent 延迟和 Amdahl 定律（Amdahl 定律说的是：如果你优化的部分只占总时间的一小块，那总加速比会被没优化的部分封顶）的框架下判的。按同一标准审 sched_ext 也会被毙——单次调度决策只花几十纳秒，在整个工作负载的端到端时间里占比极小。在扩展性框架下，"窄不窄"不由单个操作的延迟占比决定，而由策略多样性来度量。

**(3) doc 08 的两半。** "便宜动作不新"= 范畴错误（见上文误判二，同一标准会毙掉 sched_ext / cache_ext / XRP 全部）。doc 08 的另一半——"丰富动作超出 eBPF verifier 上限、只能退回 FUSE"——跨框架依然成立，但它杀的是"丰富动作可编程 VFS"这个方向。新主线不主张任何丰富动作（不做内容转换、不做注入、不做延迟），只做简单的 PASS 和 REDIRECT，所以这半条判决与新主线不冲突。

**(4) doc 08/09/10："没有只有 namei_ext 才能支撑的应用。"跨框架成立，不翻案。** 见上文问题一的 (1)：每一个具体应用都有不依赖 namei_ext 的替代方案。新主线的专属性不来自"某个应用只能用 namei_ext"，而来自"一万个视图在同一台机器上共存、不增加挂载"这个密度场景在现有方案中没有人占住。

## 六、三个候选方向排序

### 候选 1（首选，BUILD 条件式）：namei_ext 主线——VFS 名字解析扩展点论文

把"零物化视图"（即不真的在磁盘或内存里复制出一棵完整的文件树，而是在路径查找时动态计算出进程应该看到什么）合并为旗舰能力。

**三道硬闸门（任一不过即停）：**

- 闸门一：第一周实测。挂策略的路径查找赢 FUSE 2 倍以上、与 overlayfs 同一量级。达不到立刻停。
- 闸门二：动工前完成 RCU-walk 可见标记的纸面设计。
- 闸门三：用例表至少真实跑通 5-6 行。doc 07 的教训是：只跑了 2 行就被审稿人按"场景太窄"毙掉。

**用例表（8 行）：** Bazel 封闭构建输入视图（sandboxfs 已废弃）/ 大仓虚拟检出（EdenFS 类）/ 每分支 CoW 视图（直接给本组 FEC/BranchFS 的 branch() 提供一个内核机制，BranchFS 目前是 FUSE 实现、吞吐只有原生的 19%）/ composefs 式容器镜像 / 沙箱最小可见性 / gVisor 类宿主侧视图 / casefold 兼容 shim（让大小写不敏感的文件系统行为可编程）/ 万级视图密度曲线。

**必赢实验：** 同一台机器上 10,000 个 cgroup 各自持有不同的文件视图。对三个基线（overlayfs / bind-mount 森林 / FUSE 三档配置）打五个数字：

1. 视图建立时间 O(1) 且小于等于约 300 微秒；
2. 每视图内核内存增量近零 + 挂载表零增长；
3. 挂策略的路径查找赢 FUSE 2 倍以上且与 overlayfs 同量级；
4. 未挂策略的路径查找与原生不可区分（零开销）；
5. 如实给出失去 RCU-walk 的代价曲线——这是必须诚实展示的弱点，不能回避。

**实现策略：** 重新设计钩子集，而不是修补现有原型。现有原型的已知缺陷（缺少父目录身份、cgroup_id 未真正填充、readdir 逐条目做 BPF 调度）作为新设计的输入约束，不作为继续修补的基础。

### 候选 2（BUILD 无条件，先动手）：沙箱的文件系统开销——FUSE 变快之后还剩什么

一篇普查加测量的论文，同时兼任候选 1 的 go/no-go 闸门。

**它在做什么。** 在 Linux 6.14 以上内核、FUSE passthrough + io_uring 全开的条件下，测 10 种以上视图提供者的成本——Bazel/sandboxfs、bubblewrap/nsjail、gVisor 四代、EdenFS 类、overlayfs + composefs、本组 BranchFS、FUSE 三档定标。成本分解为四项：数据路径（读写文件内容）/ 元数据与路径查找 / 视图建立（创建一个新视图要多久、花多少内存）/ 挂载表与 dcache（dentry 缓存，内核缓存路径查找结果的地方）。从 1 个视图扫到 10,000 个视图，画规模曲线。

**谱系。** 这篇论文的学术先例是 *To FUSE or Not to FUSE*（FAST 2017）——当年系统性测量了 FUSE 的开销分解。现在 FUSE 变快了（passthrough 绕过用户态、io_uring 批处理请求），需要重新做这个测量。

**撞车经核实不致命。** CoFS（FAST 2026）只测了只读容器镜像的 lookup 优化（提速 86%）且是针对一个特定场景预设而非全面测量"税"。

**两个方向的答案都有用。** 如果测出来 FUSE 提速后剩余的税仍然很大——候选 1 的动机成立，值得继续建。如果测出来剩余的税很小——这本身是一篇负结果论文（证明"FUSE 变快之后，做文件视图投影不再需要内核可编程接口"），照样可以发，同时候选 1 及时止损。

**它同时是用户与合作者分歧的数据裁决：** 四周之内出数字，代替双方继续靠判断争论。

### 候选 3（BUILD，默认并回主线当第二贡献）：零物化文件树视图抽象

清单语言（用声明式规则描述一个视图应该包含哪些文件、排除哪些文件）+ 编译到 BPF + 代数版本号原子切换（用一个全局的版本号把整棵视图树从一个版本瞬间切到另一个版本，不会出现一部分文件是旧版本、一部分是新版本的混合状态）。

独立成篇需要四条曲线全部达标：视图建立延迟与内核内存比挂载方案好一个数量级、视图内热路径查找保住原生性能的约 90%、O(1) 原子切换且无新旧混合。如果独立发不足以成篇，合并回候选 1 作为第二贡献。

ClearCase MVFS 和 composefs 经核实各自缺少这里的头条能力（ClearCase 每视图要一个服务进程、composefs 不可编程且固定只读），组合方案无人占住。

### 执行顺序

开销普查（候选 2）和第一周微基准（候选 1 的闸门一）并行开跑。RCU-walk 设计（候选 1 的闸门二）纸面先行。两道闸门过了之后，全部工程量压到主线。任一闸门不过，主线止损，普查转负结果照发。

## 七、五盆冷水

按严重程度排列。

**第一：三个 BUILD 全是条件式，条件全是还不存在的数字。** 升级的是框架，不是执行。现有原型仍然是 4 字节常量策略、零宏观证据，doc 02 的执行债一条没还。"BUILD"的意思是"值得开工并设好闸门"，不是"能发"。

**第二：最大技术未知只有断言没有设计。** 被标记子树在 RCU-walk 里的低开销可见性——dentry 标志的先例只说明方向可行，不保证代价可以接受。如果走不通，"与 overlayfs 同量级"这个性能目标就永远打不出来。

**第三："赢 FUSE 2 倍"没测过。** FUSE 这两年真的变快了（passthrough 在 6.9、io_uring 在 6.14），真有可能输。第一周先测数字，别先写故事。

**第四：裁决边界。** 本文没有跑任何新实验。五个必赢数字全部是预测。任何实测反例优先于本裁决——如果数字和裁决矛盾，以数字为准。

**第五：怎么跟合作者说。**

如果闸门判 no-go——框架本身没有输。它纠正了我们两条半误判，这记在她的账上。输的是这个子系统在她选定的框架里的入场数字。"比改内核 safe"在任何已发表的扩展点论文里都不曾单独够用过：sched_ext、cache_ext、XRP 全部靠零开销 + 打赢现有固定功能方案 + 策略多样性三样数字进门。namei_ext 还额外背着 RCU-walk 代价。普查照发，框架之争由数据了结，谁都不用认输。

对称地：如果数字打出来了，旧文档那几条死刑就是我们自己把评判框架用错了。这半个错要认——这场分歧双方各对一半。闸门实验让剩下那一半由机器说了算。
